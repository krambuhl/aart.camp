## What happened

Closed Phase 4, the cap PR for the testing-harness adoption project.
Authored `.github/workflows/ci.yml` as a single workflow with a
four-task matrix — `lint`, `build`, `vitest`, `e2e` — running in
parallel with `fail-fast: false` so per-task wall-time is legible
in the GH UI. Pinned Node 24, npm cache via `setup-node@v4`,
Playwright with `--with-deps chromium` on the e2e job per the
golden-image learning. Concurrency block keyed on `github.ref`
auto-cancels superseded runs.

All four matrix jobs went green on first push: lint 29s, vitest 36s,
build 68s, e2e 1m 51s. Total wall-time = 1m 51s (e2e dominates),
total CPU across runners ~4m 4s. The "matrix to understand
performance" decision paid off immediately — e2e is the obvious
optimization target if CI runtime ever hurts; everything else is
rounding error.

Two scope decisions surfaced mid-flight. First, the 22-commit
fast-forward from main when cutting the Phase 4 branch brought
down three substrate test files (`.claude/scripts/{griot,guild,trout}/*.test.ts`)
written against `node:test`. Vitest doesn't recognize `node:test`
test functions as suites — same root cause Phase 1 D3 already
fixed for the original autosave test. Raised it as a scope fork
(options A/B/C); user picked option C (precursor commit on this
branch, two commits in one PR). Second, the OAuth token didn't
have `workflow` scope for pushing `.github/workflows/*`; user
refreshed it externally and the push went through.
