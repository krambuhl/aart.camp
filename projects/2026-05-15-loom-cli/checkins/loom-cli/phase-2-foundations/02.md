# Checkin 02 — loom-cli/phase-2-foundations

**Created**: 2026-05-15 09:42
**Phase**: 2 — Foundations + read API
**Unit**: Storage lib (project + manifest + events + config)

## Contract

- **Goal**: Author the internal storage lib for the project-state
  half of the substrate (manifest, events, config) plus the
  cross-cutting slug-resolution module. Each is a pure function
  that takes a path (or, for slug resolution, a slug + projects
  root) and returns a typed value or throws a structured error.
  Each module ships with a sibling test file exercising it
  against the Phase 1 fixtures. Verbs land in unit 03 and consume
  these libs.

- **Acceptance criteria**:
  1. `.claude/cli/lib/manifest.ts` exports `readManifest(path:
     string): Manifest`. Reads the file, parses JSON, returns
     typed `Manifest`. Throws a structured-error object (matching
     LOOM-CONVENTIONS.md error shape) when the file is missing
     (`manifest-not-found`), unreadable (`manifest-unreadable`),
     or not valid JSON (`manifest-invalid-json`).
  2. `.claude/cli/lib/events.ts` exports `readEvents(path: string,
     opts?: {since?, event?, limit?}): Event[]`. Reads
     events.jsonl line-by-line, parses each line as JSON, filters
     by optional `since` (ISO timestamp), `event` (name), and
     `limit` (max count). Throws `events-not-found`,
     `events-unreadable`, or `events-invalid-line` (with the line
     number) on failure.
  3. `.claude/cli/lib/config.ts` exports `readConfig(path:
     string): Config`. Same shape as manifest reader; structured
     errors `config-not-found`, `config-unreadable`,
     `config-invalid-json`.
  4. `.claude/cli/lib/project.ts` exports:
     - `resolveProject(slug: string, projectsRoot: string):
       string` — returns the absolute path to the project
       directory. Accepts date-less form (`loom-cli`), full form
       (`2026-05-15-loom-cli`), relative path, and absolute path.
       Scans active first, falls back to `archive/`. Throws
       structured errors `slug-ambiguous` (with candidates) or
       `project-not-found`.
     - `listProjects(projectsRoot: string, opts?: {archived?:
       boolean}): Array<{slug, path}>` — enumerates active (or
       archived) projects.
  5. Each module has a sibling `.test.ts` exercising it against
     Phase 1 fixtures (`cli/fixtures/manifest-basic.json`,
     `cli/fixtures/events-all-types.jsonl`,
     `cli/fixtures/config-basic.json`). `project.test.ts` uses
     `mkdtempSync` to set up a fake `projects/` tree with active
     and archived entries to exercise resolution forms and the
     ambiguous case.
  6. All errors thrown by these libs use a `LoomError` class (or
     equivalent typed pattern) that carries
     `{code, message, candidates?}` matching LOOM-CONVENTIONS.md's
     stderr shape. Callers can catch and forward to stderr +
     exit code. The class itself is exported from
     `.claude/cli/lib/types.ts` or a new `.claude/cli/lib/errors.ts`
     (Phase 2 decides) — design call: prefer co-locating with
     existing types.
  7. TDD discipline visible: a commit with test files referencing
     not-yet-existing modules (vitest passes runtime; tsc would
     fail), then a commit with implementations.
  8. `npm run test` passes; biome lint clean.
  9. No CLI verbs yet (those land in unit 03). Lib functions are
     usable from a verb handler but the dispatch surface is
     unchanged from unit 01.

## Execution

Authored five impl modules and four sibling test files across two
TDD-ordered commits:

**TDD step 1** (commit `5d0cb11`): four `.test.ts` files referencing
not-yet-existing modules. Unlike Phase 1's type-only-import variant,
these tests are value imports (real `import { readManifest } from
'./manifest.ts'`), so vitest fails to import the test files at all —
a proper red state with 4/4 test files failing collection.

**TDD step 2** (commit `b2d6439`): five impl modules. Tests turn green.

Modules:
- `errors.ts` — `LoomError` class with `{code, candidates}` fields.
  Message is prefixed with `<code>: <reason>` so it matches the
  substrate's stderr convention AND a regex like `/manifest-not-found/`
  catches it; `error.code` is the canonical structured field.
  `toPayload()` emits the JSON shape for stderr forwarding.
- `manifest.ts` — `readManifest(path)`. Errors `manifest-not-found`,
  `manifest-unreadable`, `manifest-invalid-json`.
- `events.ts` — `readEvents(path, {since?, event?, limit?})`. Splits
  the raw text on `\n`, filters empty lines, JSON.parse per line
  with 1-indexed line number reporting on parse failure.
- `config.ts` — `readConfig(path)`. Mirrors manifest shape.
- `project.ts` — `resolveProject` + `listProjects`. Handles four
  input forms (absolute path / relative path / full slug /
  date-less suffix), with `SLUG_RE` (`/^\d{4}-\d{2}-\d{2}-...$/`)
  gating full-slug validation and `DATELESS_RE` gating suffix.
  Active-first archive-fallback per LOOM-CONVENTIONS.md.

Mid-flight correction: initial test assertions used `.toThrow(/regex/)`
which matches on `error.message`. The first version of `LoomError`
exposed `code` as a property only, leaving `error.message` as the
plain human reason — those tests failed (5/5 failures). Resolution
was to prefix the message with `<code>: `, matching the substrate's
broader `<script>-error: <reason>` stderr convention. `error.code`
remains the canonical structured field for code-driven callers.
