## Contract

- **Goal**: Author the loom CLI scaffold so `bin/loom --help` and
  `node .claude/cli/loom.ts --help` both produce a structured help
  listing of the 8 namespaces, and unknown verbs produce a
  structured error per LOOM-CONVENTIONS.md. No domain verbs land
  yet — the scaffold's job is to make the entry point, dispatch
  skeleton, help system, and executable shim work end-to-end.

- **Acceptance criteria**:
  1. `.claude/cli/loom.ts` exists as the CLI entry point. When
     invoked via `node .claude/cli/loom.ts` (no args) or with
     `--help`, prints a structured help message listing all 8
     namespaces from LOOM-CONVENTIONS.md (project, phase, events,
     checkin, session, pr, retro, doctor) with one-line purposes.
     Exits 0.
  2. Unknown verb (e.g. `loom xyzzy`) prints a structured error to
     stderr matching the LOOM-CONVENTIONS.md error shape
     (`{"error": "unknown-verb", "message": "...", "candidates": [...]}`)
     and exits 1. Candidates list contains the 8 namespace names.
  3. `bin/loom` exists as a shell shim. It is executable
     (`chmod +x` survives in git), forwards `"$@"` to
     `node .claude/cli/loom.ts`, and produces identical output to
     the direct node invocation. The shim uses `exec node …` so it
     replaces itself in the process tree.
  4. Argument parsing uses `parseArgs` from `node:util` per
     substrate convention. Recognized top-level flags: `--help`,
     `--pretty`. (Verb-level flags are unit-tested when verbs land
     in later units.)
  5. Pure helpers (`formatHelp`, `parseInvocation`, `dispatch`) are
     extracted from the entry's main flow and unit-tested in
     `.claude/cli/loom.test.ts`. Tests cover: help output contains
     all 8 namespaces; unknown-verb returns structured error;
     no-args prints help and exits 0.
  6. `npm run test` passes (existing 334 + new tests).
  7. Erasable TypeScript only; stdlib only; ESM via the existing
     `.claude/cli/package.json`. No new npm dependencies.

- **Rules applied**:

## Execution

Authored the loom CLI scaffold across three files and committed in
one go (no TDD-split needed — scaffold + tests are intrinsically
intertwined; the pure-helper extraction makes tests possible at all):

- `.claude/cli/loom.ts` (115 LOC): entry + four exported pure helpers
  (`parseInvocation`, `formatHelp`, `formatUnknownVerbError`,
  `dispatch`), `NAMESPACES` registry, `main()` + `isEntryPoint()`
  guard.
- `.claude/cli/loom.test.ts` (110 LOC): 10 pure-helper tests + 4
  subprocess smoke tests of the node entry + 1 subprocess test of
  the bin/loom shim. 15 tests total.
- `bin/loom` (12 LOC bash): POSIX shim that resolves the entry via
  `BASH_SOURCE`, then `exec node "$ENTRY" "$@"`. `chmod +x` survives
  in git (mode 100755).

Design notes:
- Namespaces register their one-line purpose in a `Record<string,
  string>` const. Recognized verbs that aren't implemented yet
  return a structured `{error: 'not-implemented', namespace}`
  placeholder. This keeps the dispatch wired across the whole
  surface while later units fill it in.
- `parseInvocation` returns a discriminated union (`help` |
  `unknown` | `verb`). Dispatch is a pure mapping from invocation
  to `DispatchResult`.
- `main()` invokes `parseArgs` for forward compat with `--pretty`
  (verbs will use it) but does its own `--help` detection via the
  invocation parser. Slight redundancy noted by evaluator as not a
  flag; documented in the file.
- `isEntryPoint()` uses `realpathSync` for symlink-safe comparison
  of `import.meta.url` to `process.argv[1]`.
