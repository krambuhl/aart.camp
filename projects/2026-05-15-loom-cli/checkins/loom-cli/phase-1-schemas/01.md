## Contract

- **Goal**: Author `projects/LOOM-CONVENTIONS.md` — the design contract
  every subsequent loom unit (Phase 1's schemas + fixtures + tests,
  Phase 2's storage lib + read verbs, Phases 3-4's write verbs and
  skills) reads from. It is the single source of truth for the
  JSON-first substrate, the CLI conventions, and the verb surface.

- **Acceptance criteria**:
  1. Documents the directory layout for a loom-managed project:
     `manifest.json`, `events.jsonl`, `config.json`, `PLAN.md`,
     `checkins/<branch>/NN.json`, `sessions/YYYY-MM-DD-<letter>.json`,
     `retros/<filename>.json` (flat with type in JSON), and the
     archive-form variant.
  2. Documents the CLI conventions: JSON-by-default on read verbs,
     `--pretty` flag for human view, structured errors
     (`{"error": "<code>", "message": "...", "candidates"?: [...]}`)
     to stderr with non-zero exit, slug resolution accepting
     date-less / full / path forms.
  3. Enumerates all 8 namespaces and ~22 verbs from PLAN.md's CLI
     surface, with argument signatures (one line each — no behavior
     prose beyond purpose).
  4. Captures the five resolved design decisions from PLAN.md's
     Decisions section (schemas form, events storage, retro storage,
     internal lib layout, checkin execution shape) as the loom
     conventions doc's own settled-form rules.
  5. Documents the event vocabulary loom uses (drawn from
     `projects/CONVENTIONS.md`'s vocabulary, adapted for JSON
     events.jsonl rather than markdown event tables).
  6. Documents the verb-naming taxonomy (noun-verb), exit-code
     semantics (0 success, 1 error, structured stderr), and the
     conventions for `--<noun>-file=<path>` argument style.
  7. Document is internally consistent: verb namespaces line up
     with directory layout sections, event vocabulary aligns with
     state-change verbs, slug-resolution rules apply uniformly.

- **Rules applied**:
  - Match `projects/CONVENTIONS.md` voice — terse, structured,
    reference-shaped. Short paragraphs; tables where they help.
  - Markdown only. No TypeScript snippets, no JSON schema bodies,
    no code samples that lock in Phase 2 implementation choices.
  - No emojis.

- **Disqualifiers**:
  - Adds verbs not in PLAN.md's CLI surface (scope creep).
  - Specifies concrete TypeScript type shapes (those land in
    unit 02/03, not the conventions doc).
  - Describes or modifies the existing `trout-*` substrate.
  - Authors planning-shaped verbs (no `plan suggest-slug` etc. —
    planning is angular to loom per PLAN.md).
  - Includes the `pr motivation-check` verb (deferred per PLAN.md).

- **Inputs**:
  - `projects/2026-05-15-loom-cli/PLAN.md` — the loom project's plan
  - `projects/CONVENTIONS.md` — voice and structure reference

## Execution

Authored `projects/LOOM-CONVENTIONS.md` (~285 lines) as the design
contract for the loom substrate. Structure follows
`projects/CONVENTIONS.md`'s reference-shaped voice: short intro,
principles, substrate boundary, directory layout, file purposes,
CLI conventions, full verb surface (8 namespaces × 22 verbs tabled),
event vocabulary, settled decisions, versioning surface, and the
coexistence policy with the existing trout substrate.

Key concretizations beyond PLAN.md:
- Event detail shapes (one row per event in the vocabulary table)
  scaffolded for unit 02's types.ts to implement against. Detail
  shape is sketched as JSON-objects, not committed to exact TS
  interfaces.
- Exit codes: 0 success, 1 generic error, 2 argument parse error.
  Codes 3+ reserved.
- Slug resolution: four forms (date-less, full, relative path,
  absolute), with archive scanned only on active-scan miss. Cwd-
  default applies to verbs that can infer slug.
- Retro filenames: `phase-<N>-tier-<M>.json` for session-type,
  `project.json` for project-type. Type field on the JSON is
  canonical; filename is descriptive.
- Branch names with `/` produce real subdirectories under
  `checkins/`, matching existing trout convention.
- `schema_version: 1` on every loom-written file, with migration
  tooling deferred.
