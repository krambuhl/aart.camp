# Checkin 02 — loom-cli/phase-1-schemas

**Created**: 2026-05-15 01:28
**Phase**: 1 — Schemas + fixtures
**Unit**: Project-state schemas (manifest, events, config)

## Contract

- **Goal**: Author TypeScript types for the three project-state files
  (`manifest.json`, `events.jsonl`, `config.json`), representative
  JSON fixtures covering every shape and edge case, and round-trip
  validation tests that prove each fixture conforms to its type.
  TDD-shaped: fixtures + failing tests authored first, then types
  added until tests pass (visible in git history as separate
  commits).

- **Acceptance criteria**:
  1. `.claude/cli/lib/types.ts` exports types for the three
     project-state shapes:
     - `Manifest` (top-level project state), `ManifestPhase`,
       `ManifestStatus` (`active | archived`), `PhaseStatus`
       (`not-started | in-progress | blocked | completed`).
     - `Event` as a discriminated union over the closed
       13-event vocabulary from LOOM-CONVENTIONS.md; per-event
       detail shapes typed.
     - `Config` (PR settings, verification commands, worker
       bindings).
     - Every type carries the `schema_version` invariant where
       LOOM-CONVENTIONS.md says it does.
  2. Types use only erasable TypeScript (no enums, no namespaces,
     no parameter properties, no const enums) per substrate
     convention (L-001 caveat).
  3. Representative fixtures under `.claude/cli/fixtures/`:
     - `manifest-basic.json` — a full manifest with multiple phases
       at varied statuses (covers all four phase statuses) and the
       state-of-the-art event-row equivalence.
     - `events-all-types.jsonl` — one line per event in the closed
       vocabulary (13 lines minimum).
     - `config-basic.json` — verification commands, PR base,
       reviewers, labels, worker bindings — all fields populated.
  4. `.claude/cli/lib/types.test.ts` runs round-trip validation
     for each fixture: load JSON → parse → in-test typeguard
     asserts shape → `JSON.stringify` → compare against original
     (modulo whitespace). Every fixture round-trips clean.
  5. `npm run test` passes (existing 296 tests + the new ones).
  6. Git history shows the TDD order: a commit with fixtures and
     failing tests, then a commit adding the types that make them
     pass. (Reviewer can spot-check `git log --oneline` on this
     branch.)
  7. `.claude/cli/package.json` exists with `{"type": "module"}`
     so the cli/ tree imports as ESM, mirroring
     `.claude/scripts/package.json`.

- **Rules applied**:
  - TDD: fixtures + tests first, types after. Visible as separate
    commits.
  - Stdlib only — no new npm runtime dependencies (existing vitest
    devDep is fine).
  - Erasable TypeScript per L-001 / substrate convention.
  - Tests use vitest (`import { test } from 'vitest'`) per
    `projects/CONVENTIONS.md`.
  - No emojis.

- **Disqualifiers**:
  - Includes types for `Checkin`, `Session`, or `Retro` (those
    belong to unit 03).
  - Implements CLI code, command dispatch, or verb logic (Phase 2).
  - Adds runtime JSON Schema validation libraries (decided against
    in PLAN.md).
  - Uses non-erasable TS features (enum, namespace, parameter
    properties, const enum).
  - Mutates existing files outside `.claude/cli/` and the project's
    checkin/manifest entries.

- **Inputs**:
  - `projects/LOOM-CONVENTIONS.md` — design contract; types must
    line up with directory layout and event vocabulary
  - `projects/2026-05-15-loom-cli/PLAN.md` — phase context
  - `projects/CONVENTIONS.md` — substrate-script conventions
    (stdlib only, vitest, erasable TS, sibling `.test.ts`)

## Execution

Authored TypeScript types and JSON fixtures for the three project-state
schemas (manifest, events, config), plus round-trip validation tests.
Landed as two commits to make the TDD order visible in git history:

1. **TDD step 1** (fixtures + tests): Three fixture files + `types.test.ts`
   with type-only imports referencing types that didn't exist yet. At
   runtime the tests pass (TS type-only imports are erased), but a
   tsc check would fail on the unresolved imports.
2. **TDD step 2** (types): `cli/lib/types.ts` defining `Manifest`,
   `Event` (discriminated union over the closed 13-event vocabulary),
   `Config`, and their supporting types. With types in place, both
   vitest and tsc pass clean.

Types are erasable (no enums, no namespaces, no parameter properties,
no const enums) per L-001 / substrate convention. The `Event` type is
a discriminated union — each event-name variant carries its own
detail shape, so a consumer narrows on `event.event` and gets typed
access to `event.detail`.

Fixtures designed for completeness:
- `manifest-basic.json` covers all four `PhaseStatus` values across
  its four phases, plus optional `blocked_reason` and `pr` fields.
- `events-all-types.jsonl` has 14 lines covering every entry in the
  closed vocabulary, with two `retro-written` events (one each for
  `type: 'session'` and `type: 'project'`).
- `config-basic.json` populates every required field.
