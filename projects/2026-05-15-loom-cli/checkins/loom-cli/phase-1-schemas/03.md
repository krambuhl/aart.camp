# Checkin 03 — loom-cli/phase-1-schemas

**Created**: 2026-05-15 01:48
**Phase**: 1 — Schemas + fixtures
**Unit**: Unit-of-work schemas (checkin, session, retro)

## Contract

- **Goal**: Author TypeScript types for the three unit-of-work files
  (`checkin`, `session`, `retro`), representative JSON fixtures
  covering every shape and discriminator variant, and round-trip
  validation tests that prove each fixture conforms to its type.
  Same TDD-shaped pattern as unit 02 — fixtures + failing tests in
  one commit, types in the follow-up commit. Closes Phase 1.

- **Acceptance criteria**:
  1. `.claude/cli/lib/types.ts` extends with types for the three
     unit-of-work shapes:
     - `Checkin` (top-level checkin record) plus `CheckinContract`,
       `CheckinExecution`, `CheckinVerdict`, `CheckinVerdictResult`
       (`'approved' | 'flagged'`).
     - `Session` (handoff record).
     - `Retro` as a discriminated union over `SessionRetro` |
       `ProjectRetro`; both share a `RetroFinding` base shape with
       `category` (`'kept-well' | 'improvement' | 'process-change'
       | 'follow-up'`) and `description`/`evidence` fields.
     - Every type carries `schema_version: SchemaVersion` per the
       LOOM-CONVENTIONS.md invariant.
  2. Types remain erasable (no new enums, namespaces, parameter
     properties, or const enums).
  3. Representative fixtures under `.claude/cli/fixtures/`:
     - `checkin-basic.json` — approved verdict, structured execution
       (`actions[]`, `files_touched[]`, `corrections[]`), all fields
       populated.
     - `checkin-flagged.json` — flagged verdict with non-empty
       reasons; covers the other discriminator value of
       `CheckinVerdictResult`.
     - `session-basic.json` — handoff with `phases_touched`,
       `checkins_written`, `pr_activity`, `what_happened`,
       `open_threads`, `notes`.
     - `retro-session.json` — `type: 'session'` with `phase` and
       `tier` fields.
     - `retro-project.json` — `type: 'project'` (no phase/tier).
  4. `.claude/cli/lib/types.test.ts` extends with round-trip
     validation for each new fixture: load JSON → in-test typeguard
     → JSON.stringify compare. Plus discriminator-coverage
     assertions: both `CheckinVerdictResult` values exercised across
     the checkin fixtures, both `Retro` types exercised.
  5. `npm run test` passes (existing 301 + new tests).
  6. Git history shows TDD order: fixtures+tests commit first, then
     types commit. Visible via `git log --oneline`.
  7. No prose narrative stored anywhere — execution actions, retro
     findings, session events are all structured arrays/objects per
     PLAN.md's "robots first, narrative composed at output time"
     decision.

- **Rules applied**:
  - TDD: fixtures + tests first, types after. Visible as separate
    commits.
  - Stdlib only; no new runtime dependencies.
  - Erasable TypeScript per L-001 / substrate convention.
  - Vitest for tests.
  - No emojis.

- **Disqualifiers**:
  - Implements CLI verb logic (Phase 2's job).
  - Adds parse/validator helper functions to types.ts beyond pure
    type definitions (per-domain modules like `manifest.ts`,
    `checkin.ts`, etc. land in Phase 2).
  - Adds runtime JSON Schema validation libraries.
  - Uses non-erasable TS features.
  - Stores prose narrative as a string blob in any field (the design
    decision: structure or not at all). Long single-string fields
    like `changes_since_previous` are OK as one-line summaries.

  - `projects/2026-05-15-loom-cli/PLAN.md` — phase context, decisions
  - `.claude/cli/lib/types.ts` — existing project-state types from
    unit 02 to extend
  - `.claude/cli/lib/types.test.ts` — existing test file to extend

## Execution

Extended TypeScript types and JSON fixtures for the three unit-of-work
schemas (checkin, session, retro), with round-trip validation tests.
Same TDD shape as unit 02 — fixtures + tests in one commit, types in
the follow-up:

1. **TDD step 1** (commit `18e7ae9`): Five new fixture files + 7 new
   tests in types.test.ts referencing unresolved types. Vitest passes
   at runtime (TS type erasure); tsc would fail.
2. **TDD step 2** (commit `47af43a`): Types in cli/lib/types.ts adding
   Checkin, Session, Retro (discriminated union over SessionRetro and
   ProjectRetro), plus their supporting types.

Design notes:
- `Retro` is a discriminated union on `type` (`'session' | 'project'`).
  Session retros carry `phase` + `tier`; project retros omit them. The
  discriminator + structural difference let TS narrow correctly.
- `RetroFinding` is shared across both retro types and is categorized
  (kept-well / improvement / process-change / follow-up). Robots can
  filter findings by category without reading prose.
- `CheckinExecution` is fully structured (`actions[]`,
  `files_touched[]`, `corrections[]`) per the design decision. The
  `changes_since_previous` field stays as a single-line summary string
  — that's a one-line orientation hint, not narrative prose.
- All types continue to be erasable.
