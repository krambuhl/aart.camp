## Notes for PR

- The attempt loop is procedural natural-language inside a
  Markdown skill, not pseudocode. Claude reads "iterate
  `attempt` from 1 to `config.rewrite.max_attempts`" and
  executes it as a loop. The structure is explicit (B.1
  through B.7 are sub-steps that clearly run inside the
  iteration), which is the readability goal ŌĆö a reader
  scanning the skill should see the loop without having to
  reconstruct it from prose.
- All JSONL writes go through `operator-checks.ts
  log-intervention`. This was a deliberate refactor: D1's
  skill had four "append a JSONL line" instructions in
  imperative natural language, which Claude executed via
  ad-hoc Bash. Centralizing through the helper makes the
  writer consistent (parent-dir creation, JSON.stringify,
  trailing newline are all handled by the script) and removes
  the implicit-formatting burden from the orchestrator.
- The synthetic verdicts (`RUBRIC_TAMPERED`,
  `REWRITER_FAILED`, `TEST_SUBJECT_FAILED`, `STUCK_LEARNING`,
  `NO_CONSENSUS`) are not real panel verdicts ŌĆö they exist
  only as orchestrator state to route to the right outcome
  branch. They never appear in `mediate-panel.ts` input or
  output.
- `STUCK_LEARNING` is the only outcome that spawns
  `griot-operator`. The other escalation paths
  (`RUBRIC_TAMPERED`, `REWRITER_FAILED`, `TEST_SUBJECT_FAILED`,
  `NO_CONSENSUS`) all surface to the user via the run
  summary and `operator-log.jsonl` without LLM diagnosis ŌĆö
  the cause is already known by the time the loop exits.
- The `attempts_history` accumulator is per-note and exists
  only in working memory for the duration of that note's
  pipeline. It is consumed by `griot-rewriter` (latest entry)
  and `griot-operator` (full list) and discarded when the
  note's outcome is decided.
- bench-history's `run_kind: "compact"` is the canonical
  marker now. Any pre-D2 entries in
  `learnings/bench-history.jsonl` will have
  `run_kind: "compact-skeleton"` from the D1 skeleton phase;
  reading them historically will distinguish "ran without
  rewrite loop" from "ran with full pipeline."
- Next deliverable (Phase 2 D3): regression suite + nightly
  PR body. Re-run the panel against existing rollup entries
  to detect regressions, log any to
  `learnings/regressions.jsonl` (also via
  `log-intervention`), then compose the markdown PR body the
  user takes to GitHub.
