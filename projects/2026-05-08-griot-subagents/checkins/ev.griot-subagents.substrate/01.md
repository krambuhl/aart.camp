# Checkin 01 — ev.griot-subagents.substrate

**Created**: 2026-05-08
**Phase**: 1 — Substrate
**Unit**: Define six griot-* subagent type files (1 base + 5 roles)

## Contract

- **Goal**: Establish the role taxonomy for the new subagent-driven
  griot pipeline. Six new files under `.claude/agents/`: a shared
  `griot-base.md` carrying stance, output framing, and the "what you
  never do" list, plus five role files each referencing the base and
  layering a role-specific rubric on top. Mirrors the
  `evaluator-base` / `evaluator-*` pattern internally to griot, but
  shares no code or files with the `guild-*` or `evaluator-*` systems.
  The skill that consumes these (Phase 2) does not exist yet, so this
  deliverable is purely additive — no behavior change.

- **Acceptance criteria**:
  0. `.claude/agents/griot-base.md` exists. Frontmatter:
     `name: griot-base`, `role: griot-base`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     contains the shared stance every griot subagent inherits:
     stateless / no memory, terse / no hedging, mechanical / no
     speculation beyond inputs, read-only by default, structured
     output (each role's body documents its specific output shape;
     this file establishes the meta-rule that the orchestrator parses
     a designated structured block at the end of the response). Plus
     a "what you never do" list (no clarifying questions, no
     filesystem traversal beyond injected paths, no cross-subagent
     state). Domain-agnostic; no role-specific rubric. Not directly
     callable as a useful subagent — has no role rubric of its own.
  1. `.claude/agents/griot-judge.md` exists. Frontmatter:
     `name: griot-judge`, `role: griot-judge`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     opens by instructing the agent to read `griot-base.md` and apply
     its constraints. Then layers the judge-specific rubric:
     evaluate rubric assertions mechanically, return one of
     IMPROVED / UNCHANGED / REGRESSED / DID_NOT_REPRODUCE, the
     verdict format the orchestrator parses, the rules of mechanical
     derivation (binary, strict comparison logic). Mirrors BASE_SYSTEM
     and verdict schema from `scripts/learnings-compact/judges.ts`.
  2. `.claude/agents/griot-rubric-author.md` exists. Frontmatter:
     `name: griot-rubric-author`, `role: griot-rubric-author`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     opens by referencing `griot-base.md`. Then describes the
     fresh-context stance (sees only correction and wrong-output,
     never the candidate learning), the 2-3 binary assertion
     requirement with examples of good/bad assertions, the
     immutability of rubrics once written. Mirrors RUBRIC_SYSTEM from
     `scripts/learnings-compact/agents.ts`.
  3. `.claude/agents/griot-rewriter.md` exists. Frontmatter:
     `name: griot-rewriter`, `role: griot-rewriter`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     opens by referencing `griot-base.md`. Then describes the rewrite
     stance (revise the learning to satisfy the rubric, cannot modify
     the rubric, 1-2 paragraphs, name forbidden and preferred things,
     don't explain architecture). Mirrors REWRITER_SYSTEM from
     `scripts/learnings-compact/agents.ts`.
  4. `.claude/agents/griot-debate-summarizer.md` exists. Frontmatter:
     `name: griot-debate-summarizer`, `role: griot-debate-summarizer`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     opens by referencing `griot-base.md`. Then describes the
     debate-round summarization stance: take the prior round's
     per-judge verdicts and reasoning, produce a concise summary the
     next round's judges will see. No content evaluation, no
     consensus picking — that's the orchestrator's job. No verbatim
     equivalent exists in the current SDK code (the existing mediator
     passes prior reasoning as raw text); this file establishes the
     role explicitly.
  5. `.claude/agents/griot-operator.md` exists. Frontmatter:
     `name: griot-operator`, `role: griot-operator`,
     `description: <one-line>`, `tools: Read`, `model: inherit`. Body
     opens by referencing `griot-base.md`. Then describes the
     diagnosis stance (categorise why N rewrite attempts failed, do
     not propose fixes, pick exactly one of:
     same_assertion_fails_every_attempt /
     different_assertions_fail_each_attempt /
     control_and_treatment_always_identical / other, plus 1-2
     sentences of specifics). Mirrors OPERATOR_SYSTEM from
     `scripts/learnings-compact/agents.ts`.

## Execution

`griot-base.md` is the shared base. Frontmatter: `name: griot-base`,
`role: griot-base`, `tools: Read`, `model: inherit`. Body covers the
stateless / mechanical / terse stance, the input contract (orchestrator
injects everything; missing sections cause stop-and-report not
improvise), the structured-output meta-rule (every role's response
ends with a fenced markdown block the orchestrator parses; specific
fence label per role), the read-only constraint, the "what you never
do" list (no clarifying questions, no speculation, no filesystem
traversal beyond injected paths, no cross-subagent state, no
training-cutoff knowledge, no rubric relaxation), and the explicit
isolation note from `evaluator-*` and `guild-*` families.

Each of the five role files opens by instructing the agent to read
`.claude/agents/griot-base.md` and apply its constraints. Role-
specific bodies mirror the SDK system prompts:

- `griot-judge.md` mirrors `BASE_SYSTEM` from `judges.ts`. Verdict
  derivation rules (IMPROVED / UNCHANGED / REGRESSED /
  DID_NOT_REPRODUCE) preserved verbatim. Output is a fenced
  `verdict` block with control_evals, treatment_evals, verdict,
  reasoning. Debate-round support: optional "other judges' reasoning"
  section in the input prompt; "do not defer, vote your read"
  preserved.
- `griot-rubric-author.md` mirrors `RUBRIC_SYSTEM` from `agents.ts`.
  Fresh-context stance preserved (sees only correction + wrong;
  candidate learning not visible). 2-3 binary assertion requirement
  preserved with the same good/bad examples. Immutability stated.
  Output is a fenced `rubric` block with a JSON array of strings.
- `griot-rewriter.md` mirrors `REWRITER_SYSTEM` from `agents.ts`. Hard
  rules preserved (cannot change rubric; 1-2 paragraphs; concrete and
  actionable; name forbidden and preferred things). Added explicit
  rule: no meta-commentary in the learning itself (the learning gets
  injected verbatim into a system prompt — narrating the rewrite
  process inside it would leak). Output is a fenced `learning` block
  with the revised text.
- `griot-debate-summarizer.md` is a fresh prompt (no SDK equivalent;
  the existing mediator passes raw judge reasoning forward). Pure
  summarization, no opinion, no consensus picking. Lists what to
  include (vote distribution, common ground, key disagreements,
  optional tier patterns) and what to omit (own opinion, verbatim
  reasoning, new evidence, speculation). Output is a fenced
  `summary` block under 120 words.
- `griot-operator.md` mirrors `OPERATOR_SYSTEM` from `agents.ts`. Four
  diagnosis categories preserved verbatim. "Do not propose fixes"
  preserved. Note shape made more concrete with one good/bad example.
  Output is a fenced `diagnosis` block with category and notes.

## Notes for PR

- This deliverable establishes the `griot-*` family pattern. Each
  role file references `griot-base.md` for shared stance, and layers
  a role-specific rubric on top. Mirrors `evaluator-base` /
  `evaluator-*` internally to griot but shares no files or code with
  that family.
- Behavioral parity with the SDK system prompts is the bar.
  `griot-judge`, `griot-rubric-author`, `griot-rewriter`, and
  `griot-operator` mirror their SDK counterparts directly.
  `griot-debate-summarizer` is a fresh prompt — the SDK has no
  explicit summarization system prompt; it passes raw reasoning
  forward.
- Output contract: every role ends its response with a fenced
  markdown block (`verdict`, `rubric`, `learning`, `summary`,
  `diagnosis`) that the orchestrator parses. Fence labels are part of
  the contract and must be exact.
- All role files use `tools: Read` (Read kept as escape hatch for
  injected file paths). All use `model: inherit` so Phase 2 can pass
  per-spawn model overrides via the Agent tool's `model` parameter.
- Phase 2 (the orchestrator skill) does not exist yet. These files
  are not consumed by anything in the current codebase. Purely
  additive — no behavior change, no SDK path affected.
- The pre-existing build failure on `main` is a Storybook plugin
  install gap, not a regression from this work. Worth surfacing as a
  separate fix-up but out of scope for this deliverable.
- D2/D3 design carries a downstream question: with the panel reduced
  from 7 to 4 judges (2 Opus / 1 Sonnet / 1 Haiku), the consensus
  thresholds need redesign. Likely shape: 4/4 round 1 → 3/4 round 2
  → 3/4 round 3 with a tiebreak rule "both Opus agree → their
  verdict wins." To be settled in D2 (config) and D3 (mediate-panel
  helper).
