# Checkin 01 — ev.draft-cli.scope-shift-wiring

**Created**: 2026-05-15
**Phase**: 1 — Ship CLI + two skills + loop wiring
**Unit**: D7 — Wire scope-shift detection into `ev-loop-confidence`
and `ev-loop-interactive`. Restrictive default (two-signal-concurrence)
to offer `/draft-revise`; one-signal cases surface in the unit's
`Notes for the PR` but do not interrupt.

## Contract

- **Goal**: Add a "scope-shift detection" step to both ev-loop
  skills that, after a unit settles (approved by the evaluator
  panel), checks the unit's findings + user comments + whiteboard
  output for signals that PLAN.md is stale. Two signals must concur
  to offer `/draft-revise`; single signals surface in notes but do
  not interrupt the loop. The step lands as a new sub-step in each
  loop's unit-loop section, between "Iterate or commit" and
  "Autosave."

- **Acceptance criteria**:

  **Shape of the new sub-step** (identical prose in both
  skills, intentionally inlined — see Decisions in PLAN.md):

  1. Title: **Scope-shift detection (restrictive default)**.
  2. Trigger: runs only after the unit reaches `approved` in the
     panel. Flagged-and-iterating units do not check.
  3. **Signal sources** documented as a closed list:
     - **Evaluator finding** mentioning a missing or changed
       phase, deliverable, or load-bearing decision (look at the
       panel's findings text, both blocking and advisory).
     - **User comment during the unit** that mentions
       rethink, revise, scope-shift, "we should reconsider X",
       "Phase N isn't quite right", etc. (heuristic: the user
       described a change to plan structure, not just a tactical
       refinement of this unit's contract).
     - **Whiteboard contradiction** (round 2+ whiteboard surfaces
       a disagreement between engineers that the current PLAN
       doesn't resolve).
     - **Phase boundary** (this unit is the last in its phase
       OR the next phase about to start). Phase boundaries are
       natural inflection points; PLAN drifts are easier to
       spot here.
  4. **Two-signal-concurrence rule**: offer `/draft-revise` ONLY
     when 2+ signal sources fire simultaneously for the same shift.
     Single signals get a note in `Notes for the PR` with a
     `signal:` prefix; loop does NOT interrupt.
  5. **Offer flow**: surface the two signals to the user as a
     short paragraph:
     ```
     Scope-shift signal detected on <slug>:
     - <signal 1 type>: <one-line description>
     - <signal 2 type>: <one-line description>
     Proposed summary: <one-line rationale>
     Offer /draft-revise to update PLAN.md?
     ```
     Use `AskUserQuestion` (or natural-language confirm) for the
     accept/decline/defer. Default: decline (no interrupt unless
     user explicitly accepts).
  6. **Invocation on accept**: invoke
     `Skill(skill: draft-revise, args: "<slug> <one-line summary>")`.
     The summary is the proposed rationale shown to the user (or
     a refinement they typed).
  7. **Post-revise**: after `/draft-revise` returns, resume the
     loop's normal flow — proceed to Autosave (the just-committed
     unit's checkin event) and then Checkpoint. Do not re-execute
     the unit.
  8. **One-signal handling**: append a line to the unit's
     `## Notes for the PR` section in the checkin file:
     ```
     signal: <signal type>: <one-line description> (single signal; no revise offered)
     ```
  9. **Zero signals**: no action; loop continues normally.

  **Placement**:
  - `ev-loop-interactive/SKILL.md`: new sub-step inserted between
    Step 4 ("Iterate or commit") and Step 5 ("Autosave") in the
    Unit loop. Numbered as Step 4.5 OR re-numbered as Step 5 with
    subsequent steps shifted (pick the less-disruptive option;
    parent project's tone says be incrementalist).
  - `ev-loop-confidence/SKILL.md`: same sub-step inserted at the
    parallel location in the tier-level Unit loop. The
    confidence-loop framing is "after the tier's batch settles."

  **Cross-loop consistency**: the new sub-step's prose is
  identical in both files. If a third loop ever appears, extract
  to a shared substrate doc; until then, the duplication is
  bounded and acceptable.

  **Verification**:
  - `npm run lint` clean (markdown changes).
  - Both updated skills load by the harness.
  - Smoke test deferred to D8 (phase close) — synthetic
    two-signal scenario triggers the offer; user accepts;
    `/draft-revise` runs; PLAN.md updates.

- **Disqualifiers**:
  - Permissive default (offering on a single signal). The PLAN's
    Decisions explicitly chose restrictive.
  - Auto-running `/draft-revise` without user confirm. The offer
    is always opt-in.
  - Running the detection on a flagged unit. Only approved units
    trigger.
  - Different prose in the two loops. Identical inlined prose is
    a hard requirement.
  - Touching draft skills or CLI in this checkin. D6 already
    shipped both; this is pure loop wiring.

**Approach**:
- Authored the sub-step prose once and inserted into both files.
  Identical wording matters: a future reader comparing the two
  loops should see the same scope-shift step.
- Placement chosen carefully: AFTER "Iterate or commit" so the
  unit is fully settled (approved by panel) before the detection
  runs. BEFORE "Autosave" so the autosave records the just-
  committed unit's checkin event after detection completes (or
  the detection's no-op).
- Sub-step opens with the gating rule: runs only on approved
  units. Flagged-and-iterating units skip — no point checking
  scope shift while still resolving evaluator findings.
- Four signal sources documented as a closed list. Two-signal-
  concurrence is the offer rule; single signals get a `signal:`
  prefixed note in the unit's `## Notes for the PR` section.
- Offer flow uses `AskUserQuestion` with decline as the default.
  This honors PLAN's "permissive offers train the user to
  dismiss" concern from the round-1 whiteboard.
- On accept, the sub-step invokes `Skill(skill: draft-revise,
  args: "<slug> <summary>")` — the draft-revise skill from D6
  carries the actual revision composition + commit.
- No code, no automated tests. Skill-prose-only deliverable.

## Notes for the PR

- **Identical prose in both skills** is intentional. The
  duplication is bounded (two loops); the alternative (a
  shared substrate doc both skills reference) was deferred per
  PLAN's "extract when a third writer appears" pattern. If
  ev-loop-confidence and ev-loop-interactive ever diverge on the
  scope-shift step, that's the signal to extract.
- **Restrictive default**: two-signal concurrence to offer; single
  signal to note. The whiteboard's round-1 skeptic explicitly
  argued for restrictive (a missed shift costs one stale phase
  description; a spurious offer costs a context switch and may
  train the user to dismiss). PLAN's Decisions ratified
  restrictive; this PR implements that ratification.
- **Default decline**: the offer flow is opt-in. The user has to
  explicitly accept; declining doesn't even prompt for rationale.
  This further protects against prompt fatigue.
- **`/draft-revise` invocation**: composed via the `Skill` tool,
  not via Bash. The skill carries the surgical composition +
  user confirm + CLI shell-out internally. Loop's job is to
  detect the signal and route; revision composition is not
  loop-shaped work.
- **No automated tests** — pure prose change. The substrate's
  convention for skill changes is `/review-skill` audit + smoke
  tests at phase boundaries. D8 will exercise the full flow.
- Evaluator panel: `contract-fit`, `naming`. Both expected to
  approve (to be confirmed by panel run).
