# Checkin 01 — ev.draft-cli.revise-read-verbs

**Created**: 2026-05-15

**Unit**: D5 — Implement the `revise` and `read` verbs together

## Contract

- **Goal**: Add real handlers for `bin/draft revise` and
  `bin/draft read` to `.claude/cli/verbs/draft.ts`. Both use
  `resolveTroutProject` (D3) to locate existing trout projects.
  `revise` replaces PLAN.md with new content + appends to a
  `## Revision log` section + commits with a rationale-bearing
  message. `read` emits a JSON envelope structurally compatible
  with loom's read-verb shape. Test-first; existing 13 plan-verb
  tests + entry tests stay green.

- **Acceptance criteria**:

  **`reviseVerb` behavior**:
  1. Parse args: positional `<slug>`, required `--revision-file=<path>`,
     required `--rationale=<str>`, optional `--no-commit`,
     `--pretty`. Missing required → `LoomError('missing-args', ...)`.
  2. `resolveTroutProject(slug, projectsRoot)` to find the project
     directory. Throws `project-not-found` if absent.
  3. Read PLAN.md at `targetDir/PLAN.md`. If absent → throws
     `LoomError('plan-not-found', "no PLAN.md at ...; use 'draft plan' to create one")`.
  4. Compose new PLAN.md content:
     - Read revision content from `--revision-file`.
     - Append a `## Revision log` entry: one line of the shape
       `- <YYYY-MM-DD> — <rationale>` (no commit SHA — git history
       carries that; the log entry is human-readable in-tree).
     - If the revision content already contains a `## Revision log`
       section, append the new entry into it. If not, append a new
       section at the end of the document.
  5. Write the composed content back to PLAN.md.
  6. Unless `--no-commit`: `gitRunner.addAndCommit(repoRoot,
     [planMdPath], "[draft revise] <slug>: <rationale>")`.
  7. Emit JSON: `{slug, path, committed, rationale}`. `--pretty`
     produces indented JSON.

  **`readVerb` behavior**:
  1. Parse args: positional `<slug>`, optional `--pretty`. Missing
     positional → `LoomError('missing-args', ...)`.
  2. `resolveTroutProject(slug, projectsRoot)` → targetDir.
  3. Read `targetDir/PLAN.md`. Absent → `LoomError('plan-not-found',
     ...)`.
  4. Emit JSON envelope (per § Skill-to-CLI contract):
     ```json
     {
       "path": "projects/<date>-<slug>/PLAN.md",
       "content": "<full markdown>",
       "plan": {
         "slug": "<slug>",
         "interview_path": "projects/<date>-<slug>/INTERVIEW.md"
       }
     }
     ```
     `interview_path` is the path to INTERVIEW.md regardless of
     whether it exists on disk (the field tells callers where to
     look; existence is the caller's check).
  5. `--pretty` renders PLAN.md content directly to stdout (no
     envelope).

  **Wiring**:
  - `DRAFT_VERBS.revise = reviseVerb`; `DRAFT_VERBS.read = readVerb`.
  - Remove the `notImplemented('revise')` and `notImplemented('read')`
    stubs from D4 (the helper itself stays for forward-compat).
  - Update the `notImplemented`-related tests in
    `.claude/cli/draft.test.ts`: the "revise still stubs to
    not-implemented" and "read still stubs to not-implemented"
    tests become positive routing checks (call dispatch with
    revise/read + missing args → assert `missing-args`, not
    `not-implemented`).

  **`verbs/draft.test.ts` additions** (mirroring the D4 plan
  tests' shape):
  - `reviseVerb`:
    - happy path: existing project + revision file + rationale →
      PLAN.md replaced with content + Revision log appended +
      `addAndCommit` called with the right message.
    - `--no-commit` writes file but skips git.
    - `--pretty` produces indented JSON.
    - `project-not-found`: nonexistent slug → throws.
    - `plan-not-found`: project exists but no PLAN.md → throws.
    - existing `## Revision log` section: revision content with
      its own log section gets the new entry inserted INTO that
      section (not duplicated).
    - no existing `## Revision log`: revision content without
      log section gets a new section appended.
    - missing positional `<slug>` → `missing-args`.
    - missing `--revision-file` → `missing-args`.
    - missing `--rationale` → `missing-args`.

  - `readVerb`:
    - happy path: returns JSON envelope with `path`, `content`,
      `plan.slug`, `plan.interview_path`.
    - `--pretty` renders PLAN.md content directly (no envelope).
    - `project-not-found`: nonexistent slug → throws.
    - `plan-not-found`: project exists but no PLAN.md → throws.
    - missing positional → `missing-args`.

  **Verification**:
  - All existing tests still pass.
  - `npm run lint`, `npm run test`, `npm run build` clean.
  - Manual smoke: `bin/draft read <slug>` and `bin/draft revise
    <slug> --revision-file=... --rationale="..." --no-commit`
    against a throwaway project.

- **Disqualifiers**:
  - Implementing revise without `--rationale` required.
  - Skipping the "Revision log section already exists" insertion
    test — this is the load-bearing parser case (PR composition
    will produce revisions that already carry the section).
  - Hard-coding today's date in the Revision log entry — use
    `ctx.today` for test injection.
  - `read` emitting a different JSON envelope shape than loom's
    read verbs (incompatibility creates two introspection
    contracts).
  - Modifying INTERVIEW.md in either verb (revise touches only
    PLAN.md; read is purely emissive).

## Notes for the PR

- **Scope shift acknowledged**: the resolver filter broadened
  mid-PR (D3's "MANIFEST.md" rule → "PLAN.md" rule). This is the
  smallest change that closes the architectural gap between
  draft's output shape (PLAN.md only) and its own resolver
  expectation. Trout-managed projects (with both files) still
  qualify; loom projects (with manifest.json) still don't. The
  rename `TROUT_MARKER` → `PLAN_MARKER` in `draft-project.ts`
  reflects the actual semantics.
- **Combined delivery per user direction**: D5 (revise) and D6
  (read) ship in one PR as decided mid-project. Both share
  `resolveTroutProject` and parallel argument shapes; splitting
  them would have duplicated the test scaffolding and yielded two
  small reviews.
- **`appendRevisionLogEntry` is exported** so the future
  `/draft-revise` skill (D6 in the consolidated plan) can compose
  revision content programmatically before shelling out to the
  CLI. The skill's job is composing the revision file; the CLI's
  job is writing + committing. The helper is the seam.
- **Revision log ordering**: new entries land directly under the
  heading (newest-first within the section). Mirrors PR-changelog
  conventions and `git log` defaults.
- **`plan-not-found` defensive branch retained** in both
  reviseVerb and readVerb. Not reachable through the public test
  surface under the broadened filter (a project without PLAN.md
  isn't resolvable in the first place), but the check is cheap
  and survives any future filter changes.
- **`read --pretty`** short-circuits the JSON envelope and writes
  the raw PLAN.md content to stdout — the human-readable path.
  The envelope is the JSON contract for downstream introspection
  (loops, scripts).
- Evaluator panel: `contract-fit`, `react-api`, `naming` — all
  expected to approve. (To be confirmed by panel run.)
