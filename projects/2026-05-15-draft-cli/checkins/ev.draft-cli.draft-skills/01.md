# Checkin 01 — ev.draft-cli.draft-skills

**Created**: 2026-05-15
**Phase**: 1 — Ship CLI + two skills + loop wiring
**Unit**: D6 — Author the two draft skills together (consolidated per
mid-project revision). `/draft-plan` is the grill-me-shaped interview
that births a project; `/draft-revise` is the thin loop-invoked
composer that mid-flight rewrites a PLAN.md.

## Contract

- **Goal**: Ship the user-facing and loop-facing draft surfaces.
  Both skills shell out to `bin/draft` (D2/D4/D5); the skill bodies
  carry the LLM-shaped work (interview, synthesis, composition,
  confirmation) and rely on the CLI for the deterministic file IO
  and git seam. Both skills land in this PR per consolidated
  deliverable.

- **Acceptance criteria**:

  **`/draft-plan` skill** (`.claude/skills/draft-plan/SKILL.md`):

  1. Frontmatter:
     - `name: draft-plan`
     - `description:` two-line summary mirroring `/trout-plan`'s
       voice but framing the grill-me posture.
     - `argument-hint: "<topic or short description>"`
     - `allowed-tools: Read, Write, Bash(bin/draft *), Bash(date *), AskUserQuestion`
  2. Skill body covers, in order:
     - **Frame**: treat `$ARGUMENTS` as the topic; if empty/thin,
       ask the user; otherwise summarize back in one sentence and
       confirm.
     - **Grill-me interview**: walk the decision tree relentlessly,
       one question at a time, each with a recommended answer.
       Cover Scope, Phases, Dependencies, Verification, Risks,
       Decisions, and any other PLAN.md section that's load-bearing
       for the topic. Use `AskUserQuestion` where structured
       options sharpen the conversation; use natural-language
       follow-ups where the tree branches unpredictably.
     - **Slug derivation**: derive `kebab-case-slug` from the
       topic. Run `Bash("date '+%Y-%m-%d'")` to get today's date.
       Surface proposed `<YYYY-MM-DD>-<slug>` to the user and
       confirm. (The CLI's `createSlug` is the actual slug
       generator; the skill needs to show + confirm what the slug
       will be before shelling out, so the user catches mismatches.)
     - **Synthesis**: compose PLAN.md from the interview (clean
       structure: Context, Scope, Phases, Dependencies,
       Verification, Risks, Open questions, optionally Decisions).
       Compose INTERVIEW.md as the walked decision tree (questions,
       recommendations, user choices, branches taken). Both files
       written via `Write` to a temp location
       (`/tmp/draft-plan-<slug>.md`, `/tmp/draft-interview-<slug>.md`).
     - **Show + approve**: surface the composed PLAN.md content to
       the user for review. Iterate until they approve. Do not
       shell to the CLI until the user says go.
     - **Commit**: invoke
       `Bash("bin/draft plan <slug-or-topic> --plan-file=/tmp/draft-plan-<slug>.md --interview-file=/tmp/draft-interview-<slug>.md")`.
       Surface the CLI's JSON envelope output to the user.
     - **Report**: one short paragraph in the shape:
       ```
       Created draft plan: <title>
       Location: projects/<date>-<slug>/
       Files: PLAN.md, INTERVIEW.md
       Next: run /trout-plan <slug> to add the execution substrate, or /ev-run <slug> if it's already trout-scaffolded.
       ```
  3. The skill body documents the seam explicitly: `/draft-plan`
     (skill) and `bin/draft plan` (CLI) are distinct surfaces. The
     skill carries the grill-me conversation; the CLI carries the
     deterministic file IO.
  4. No emojis.
  5. Reuses the existing PLAN.md template shape from `/trout-plan`
     (cross-referenced, not re-described in full).

  **`/draft-revise` skill** (`.claude/skills/draft-revise/SKILL.md`):

  1. Frontmatter:
     - `name: draft-revise`
     - `description:` two-line summary noting this is invoked
       loop-side after a scope-shift signal, not user-facing under
       normal conditions.
     - `argument-hint: "<slug> <one-line scope-shift summary>"`
     - `allowed-tools: Read, Write, Bash(bin/draft *), AskUserQuestion`
     - `user-invocable: false` — this skill is composed by ev-loop
       on a scope-shift signal; users don't invoke directly.
  2. Skill body covers, in order:
     - **Parse**: take `$ARGUMENTS` as `<slug> <summary>`. If slug
       missing, refuse. If summary missing, ask the user for one.
     - **Read current**: invoke `Bash("bin/draft read <slug>")` to
       get the JSON envelope. Parse to extract `content` (the
       current PLAN.md) and `path`.
     - **Compose revision**: read the current PLAN.md, integrate
       the scope-shift signal/summary, write a full new PLAN.md
       reflecting the change. Preserve unrelated sections (the
       composer's job is surgical, not rewriting from scratch).
       Use `appendRevisionLogEntry` semantics: a new entry will
       land under `## Revision log`; the rationale is the
       summary.
     - **Surface**: show the user the proposed revision summary
       (1-3 sentences explaining what's changing and why) +
       optionally a diff-like preview. Ask user to confirm.
     - **Commit**: write the revision content to a temp file
       (`/tmp/draft-revision-<slug>.md`). Invoke
       `Bash("bin/draft revise <slug> --revision-file=<tmp> --rationale=<summary>")`.
       Surface the CLI's JSON envelope output.
     - **Report**: one short line in the shape:
       ```
       Revised <slug>: <rationale>
       PLAN.md updated; ## Revision log entry appended.
       ```
  3. No grill-me ceremony — revisions are surgical. The skill body
     explicitly says this and points future contributors at the
     "lift to grill-me later if quality slips" deferred item in
     PLAN.md.
  4. No emojis.
  5. `user-invocable: false` keeps the skill out of the user's
     slash-command surface; ev-loop composes it via the `Skill`
     tool.

  **Verification**:
  - `npm run lint` clean (no .ts changes; biome covers .md
    frontmatter check).
  - Smoke test for `/draft-plan`: invoke against a synthetic topic
    in a throwaway projects-root; confirm it produces PLAN.md +
    INTERVIEW.md and the CLI commits them. (Manual smoke; no
    automated test of LLM prose.)
  - Smoke test for `/draft-revise`: against a project created by
    the prior smoke, invoke with a synthetic summary; confirm
    revision lands.
  - `/review-skill` pass on each new SKILL.md before commit.

- **Disqualifiers**:
  - Writing PLAN.md or INTERVIEW.md directly via `Write` to the
    project directory rather than shelling to `bin/draft plan`
    (bypasses the CLI's collision check + git seam).
  - `/draft-revise` running a grill-me interview (D6 explicitly
    says no — surgical revisions only in v1).
  - `/draft-plan` skipping the slug-confirm step (the user needs
    to catch slug mismatches before the project directory is
    written).
  - Emojis.
  - Either skill omitting `--rationale` (revise) or skipping
    `--no-commit` exposure (both should default to commit per
    PLAN's decisions).
  - `/draft-revise` having `user-invocable: true` (it's
    loop-invoked, not user-facing).

- **Inputs**:
  - `.claude/skills/trout-plan/SKILL.md` — frontmatter convention,
    interview prose pattern
  - `.claude/skills/loom-pr/SKILL.md` — CLI-shelling skill pattern
    (`allowed-tools: Bash(bin/loom *)` syntax)
  - `.claude/cli/verbs/draft.ts` — verb behavior the skills shell
    to
  - `.claude/cli/lib/draft-git.ts` — git seam (skills don't touch
    git directly)
  - `projects/2026-05-15-draft-cli/PLAN.md` § Context, § Decisions
    (skill-to-CLI contract, no grill-me in revise, revision
    rationale shape)

## Execution

**Files**:
- New: `.claude/skills/draft-plan/SKILL.md` (~220 LoC) — the grill-me
  interview skill. Eight-step process: frame → grill-me → propose
  slug → synthesize PLAN.md → synthesize INTERVIEW.md → show +
  approve → write temp files → shell to `bin/draft plan` → report.
  `allowed-tools: Read, Write, Bash(bin/draft *), Bash(date *), AskUserQuestion`.
- New: `.claude/skills/draft-revise/SKILL.md` (~180 LoC) — the thin
  loop-invoked composer. Six-step process: parse → read current →
  compose → surface + confirm → shell to `bin/draft revise` →
  report. `allowed-tools: Read, Write, Bash(bin/draft *), AskUserQuestion`.
  `user-invocable: false` keeps it out of the slash-command surface;
  ev-loop composes it via the `Skill` tool.

**Approach**:
- Followed the `/trout-plan` SKILL.md frontmatter convention for
  `/draft-plan` (matches the existing planning-skill voice). The
  body explicitly documents the grill-me posture as the differentiator
  vs `/trout-plan`.
- Followed the `/loom-pr` SKILL.md pattern for the CLI-shelling
  shape (`allowed-tools: Bash(bin/draft *)`).
- `/draft-plan` documents the seam between skill and CLI explicitly
  (per PLAN.md's Decisions: matched naming, distinct surfaces).
- `/draft-revise` documents the no-grill-me posture explicitly and
  points future contributors at the deferred-item escape hatch
  (lift to grill-me if revision quality slips).
- Both skills carry the `## Failure modes` section with concrete
  stop-and-report paths.
- After authoring, both skills appeared in the available-skills
  list automatically — confirmed the harness picked them up
  without requiring a Claude Code restart (skills load dynamically,
  unlike agents).

## Notes for the PR

- **Combined per user direction**: D7 (`/draft-plan`) + D8
  (`/draft-revise`) ship as one PR. Both are draft-skill prose; the
  shape is parallel; reviewers can read them side-by-side.
- **No automated tests** for these skills — they're LLM prose, not
  code. The substrate's convention is `/review-skill` for skill
  audits (which the evaluator panel partially covers) plus smoke
  tests at phase close. D8 will run the full end-to-end smoke.
- **`user-invocable: false` on `/draft-revise`** keeps it out of
  the slash-command surface. ev-loop composes it via `Skill(skill:
  draft-revise, args: "<slug> <summary>")` once D7 (loop wiring)
  lands. Users land here indirectly through the loop's offer.
- **`/draft-plan` is opinionated about pacing**: stop at ~20
  questions or when the user says approved. The skill body bakes
  in the checkpoint instruction so a runaway interview doesn't
  drag.
- **`appendRevisionLogEntry` (D5 export) is unused in
  `/draft-revise`** because the composer writes raw content and
  the CLI handles the log-entry insertion. The export remains for
  future programmatic composers (e.g., a higher-fidelity revise
  flow that builds the entry inline).
- Evaluator panel: `contract-fit`, `react-api`, `naming` — all
  expected to approve. React-api is non-applicable for markdown
  skill files.
