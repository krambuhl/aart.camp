## Contract

- **Goal**: Author `.claude/agents/evaluator-tokens.md` as the fourth
  domain evaluator in the `evaluator-*` family, sibling to D1 (a11y),
  D2 (nextjs), D3 (react-api). Per PLAN.md § Phase 2, D4 is the second
  "smaller advisory-only catalog initially" deliverable — same shape
  as D3: pure agent-file change, no CLI signal, no scripts, no devDeps.
  Covers literal-value antipatterns in CSS Modules and inline styles
  in this project's design-token system (`token("namespace.path")`
  PostCSS function as the canonical abstraction, with
  `tokens/design-tokens.json` as the source of truth).

- **Acceptance criteria**:
  1. **Agent file**: `.claude/agents/evaluator-tokens.md` exists.
  2. **Frontmatter** matches the existing `evaluator-*` shape:
     `name: evaluator-tokens`, `role: evaluator`,
     `description:` (one-line — must explicitly call out
     **advisory by default**, mirroring `evaluator-react-api`'s
     framing), `tools:` (Read, Glob, Grep, plus read-only Bash for
     `npm run lint:*`, `npm run build:*`, `git status:*`,
     `git diff:*` — no `lint:tokens:*` since no script),
     `model: inherit`, `maxTurns: 5`.
  3. **Body — inherited base contract**: opens with an
     "Inherited base contract" section pointing at
     `evaluator-base.md`. Does NOT duplicate stance, verdict format,
     base flag taxonomy, or read-only-constraint content.
  4. **Body — Process section**: 3–5 numbered steps walking the
     evaluator through (a) detecting token scope (`.module.css`
     files; inline `style={...}` JSX props with literal-looking
     values; direct imports from `tokens/tokens.ts` for runtime
     value use), (b) running grep heuristics against the in-scope
     files for the catalog entries that support them
     (e.g. `#[0-9a-fA-F]{3,8}\b` for hex literals,
     `\b\d+(\.\d+)?(px|rem|em)\b` for hardcoded sizing),
     (c) manual inspection for entries that require it, (d)
     assembling verdict with all findings
     advisory by default unless the diff demonstrably regresses
     an already-tokenized path back to literals.
  5. **Body — Antipattern catalog**: 6–10 entries. Each entry:
     pattern, symptom, impact, detection method (`grep` |
     `manual`), severity (`advisory` for all entries in this
     initial pass), flag code (prefixed `tokens-`). Catalog must
     include at minimum:
     - Hex literal color in a `.module.css` rule (e.g.
       `color: #ff0000`) instead of `token("fg.*")` / `token("bg.*")`
     - Named CSS color (`red`, `blue`, `black`, `white`) used as a
       text/background fill in a `.module.css` rule when a
       semantic token would apply
     - Hardcoded `px` (or `rem`/`em`) spacing value where a
       `token("space.xN")` value exists for that magnitude
     - Hardcoded font-family / font-weight / line-height / font-
       size literal where a `token("fontFamily.*")` /
       `token("fontWeight.*")` / `token("lineHeight.*")` / token
       size value exists
     - Hardcoded breakpoint pixel value in a literal `@media
       (min-width: NNNpx)` query, bypassing
       `map-breakpoints()` / `@each` generation
     - Inline `style={{ color: '#abc', padding: 16 }}` literal
       in JSX where a CSS Module rule using `token()` would
       apply
     - Importing a token value from `tokens/tokens.ts` for use
       in a JSX `style={}` prop (runtime reads of a static
       token value, rather than declarative CSS via the module)
  6. **Body — Flag codes section**: table listing each
     tokens-specific code mapped to its catalog entry. Codes are
     `tokens-`-prefixed and do NOT duplicate base flag taxonomy.
  7. **Body — Inspection signals section**: explicitly documents
     that no dedicated CLI signal exists for this evaluator (in
     contrast with `evaluator-a11y`'s `npm run test:a11y` and
     `evaluator-nextjs`'s `npm run lint:nextjs`). Names the
     rationale, same shape as `evaluator-react-api`'s rationale:
     advisory-only initial scope; Stylelint /
     `stylelint-declaration-strict-value` deliberately not adopted
     (mirrors D1/D2/D3 precedent against linter-framework adoption
     for evaluator detection); composability — consumer projects
     with their own token systems can compose this evaluator
     without inheriting a project-specific lint config.
  8. **Body — Carve-outs section**: explicitly documents the
     contextual exclusions a grep-based detector would otherwise
     false-positive on:
     - p5.js sketch files under `sketches/` (canvas semantics,
       sometimes literal colors are intentional — the sketch IS
       the artistic statement)
     - Component-scoped CSS custom properties intentionally
       defined for runtime-tweakability (e.g.,
       `--sketch-aspect-ratio` in `Sketch.module.css`)
     - CSS keywords and viewport units (`100%`, `100dvh`, `auto`,
       `min-content`, `max-content`, `0`, `inherit`, `unset`,
       `1em` for relative sizing) — not tokens, not flags
     - `globals.css` and `tokens.css` themselves (the source-of-
       truth files; literals there are expected)
  9. **Boundary note**: brief section clarifying overlap with
     adjacent evaluators:
     - `evaluator-naming` (D5, future) owns token **naming**
       semantics — using `color.background.surface` over
       `color.gray.200` when both exist. D4 catches the literal-
       vs-token decision; D5 catches the right-name decision.
     - `evaluator-a11y` owns contrast outcomes (axe-core
       `color-contrast`). A color that passes contrast but
       bypasses the token system is a D4 flag, not a D1 flag.
     - Cross-evaluator dedup at panel level is D6's concern.
  10. **Verification**: `npm run lint` (Biome) clean. `npm run
      build` clean. `npm test` passes (no new tests since no new
      script). `git status` shows only
      `.claude/agents/evaluator-tokens.md` and
      `projects/2026-05-02-agent-guilds/...` paths modified —
      no edits elsewhere.
