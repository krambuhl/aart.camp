- Trade-off accepted: the `> [!NOTE]` callout is now template-driven (two slots), which means the skill needs to read the MANIFEST title at PR-author time. That read already happens implicitly (the skill knows the project path); making it explicit in the callout-author path is minor surface area. The win is real reviewer context vs. substrate jargon.
- correction: PR `> [!NOTE]` callouts shouldn't be generic substrate boilerplate when the skill already has project context to weave in. The previous wording ("This PR is tracked by the project substrate. ...") read like infrastructure-talking-to-itself; the new wording ("Part of the **<project title>** project — see [PLAN.md](...)") reads like reviewer-onboarding. Keeping the brevity-is-intentional framing as the second sentence preserves the load-bearing convention.
- correction: Loop sessions should always pull main first and surface loose files, because manifest staleness vs git reality is a recurring source of drift at session start. The git-sync pre-flight in /ev-run is the structural fix; downstream loops can trust working-tree state once Step 0.5 runs.
