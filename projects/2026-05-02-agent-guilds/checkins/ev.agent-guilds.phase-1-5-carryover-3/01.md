## Contract

- **Goal**: The `Item` type in `.claude/scripts/trout/pr-respond-plumbing.ts` packs four different meanings into one `location: string` field depending on `kind`: literal `'pr-conversation'` for issue-comments; the review state for reviews; a joined `path:line` string for review-comments; and the check name for ci-failures. This overload was a pragmatic shortcut when the script was first written (D9) but the typed shape ought to be a discriminated union — each `kind` carries its own kind-specific field(s). D15 refactors `Item` to a proper tagged union, removes the joined `path:line` string (split into typed `path` + `line` fields), updates `buildItems` to construct kind-specific shapes, and updates `/trout-pr-respond`'s SKILL.md prose so the LLM body knows what to expect from the new shape.

- **Acceptance criteria**:
  1. `Item` becomes a discriminated union over `kind`:
     - `{ kind: 'issue-comment'; source; body; url }` — no location field (the literal `'pr-conversation'` was redundant with `kind`)
     - `{ kind: 'review'; source; body; state; url }` — was `location: r.state`; renamed to `state` for clarity
     - `{ kind: 'review-comment'; source; body; path; line; url }` — was `location: \`${path}:${line}\``; split into typed fields
     - `{ kind: 'ci-failure'; source: 'ci'; body; checkName; url }` — was `location: name`; renamed to `checkName`
  2. `buildItems` constructs each item with its kind-specific shape; no joined strings in the output JSON.
  3. The JSON emitted by `fetch` (consumed by the `/trout-pr-respond` skill) reflects the new shape — each item has exactly the keys for its `kind`, no `location` field anywhere.
  4. `/trout-pr-respond` SKILL.md description of `items[]` updated: the existing prose `"(kind ∈ ...; source, body, location, url)"` becomes accurate to the new shape (per-kind fields enumerated, or generic description revised so the LLM understands what to expect at consume-time).
  5. Sibling `pr-respond-plumbing.test.ts` tests updated to match the new shape:
     - The existing happy-path test (`fetch happy path: items in documented order`) asserts on `location` for the ci-failure item — needs to assert on `checkName` instead.
     - At least one new test per kind verifying the typed-field shape: review-comment has `path` + `line` as separate fields with the right values (not a joined string); ci-failure has `checkName`; review has `state`; issue-comment has no extra location-replacement field.
  6. `npm run lint`, `npm run build`, `npm run test` all clean.
  7. No new runtime dependencies. No behavioral change to which items are emitted, their order, or any other downstream concern — only the **shape** of each item changes.
  8. The skill body of `/trout-pr-respond` is not separately tested by the substrate test suite (it's LLM-shaped), so the test surface for D15 lives entirely in `pr-respond-plumbing.test.ts` and the SKILL.md prose update is verified by reading.

- **Rules applied**:
  - `projects/CONVENTIONS.md` § Substrate scripts (TypeScript stdlib only, sibling `.test.ts`, vitest)
  - The "no joined-string overloads" implicit type-design principle that this whole deliverable is enforcing: if a field's meaning depends on a sibling field's value, that's two fields trying to be one — split them.
  - Existing test fixtures in `pr-respond-plumbing.test.ts:ghStub` for the mock-gh-via-PATH pattern (untouched; the test surface only updates what it asserts, not how it sets up).

- **Disqualifiers**:
  - **Keeping `location: string` for any kind** — every kind gets its kind-specific typed field. No backward-compat shim.
  - **Leaving the joined `path:line` string in review-comment** — split into typed `path: string` and `line: number`. The joined form was the specific overload the user flagged.
  - **Updating `pr-respond-plumbing.ts` but not the SKILL.md prose** — the SKILL describes what the LLM body should expect from the script's JSON. Out-of-date SKILL prose would mislead the LLM at consume-time.
  - **Adding a runtime location-fallback or aliased property** like `Object.defineProperty(item, 'location', { get: ... })` — adds a code path that does nothing the typed fields don't already do.
  - **Test cases that depend on `location` continuing to exist on any item** — every assertion against `location` migrates to the new kind-specific field name.

## Notes for the PR

- **Typed `path` + `line` is the most consequential shape change.** Consumers no longer have to parse `location.split(':')` (and worry about colons in filenames, malformed inputs, or missing line numbers) — the JSON has typed fields. The LLM body for `/trout-pr-respond` now sees real data, not a string it has to disambiguate by reading the sibling `kind`.
- **No backward-compat shim.** Items are internal between `pr-respond-plumbing.ts` and the `/trout-pr-respond` skill body; no external consumer depends on the prior `location: string` shape. The disqualifier "Adding a runtime location-fallback or aliased property" was specifically called out.
- **Issue-comment loses a property.** The `'pr-conversation'` literal was redundant with the `kind` tag, so the union variant for issue-comment has no location-replacement field at all. If a future consumer wants a uniform iteration shape, they can compute a synthetic location from `kind` on demand — but the type doesn't pretend uniformity exists where it doesn't.
- **Closes the D9-surfaced refinement.** `Item.location` was a pragmatic shortcut when `trout-pr-respond` was first migrated to a script (D9, PR #28). Session-2026-05-08-b flagged it for follow-up; D15 ships the proper shape. This deliverable closes the last item in the Phase 1.5 carryover backlog.
