## Contract

- **Goal**: Author `.claude/agents/evaluator-naming.md` as the fifth and
  final domain evaluator in the `evaluator-*` family, sibling to D1
  (a11y), D2 (nextjs), D3 (react-api), D4 (tokens). Per PLAN.md §
  Phase 2, D5 is the third "smaller advisory-only catalog initially"
  deliverable — same shape as D3 and D4: pure agent-file change, no
  CLI signal, no scripts, no devDeps. Covers naming antipatterns
  across `.tsx`/`.jsx` (component / hook / prop names),
  `.module.css` (class names, when the file's component name is the
  context), and TypeScript identifiers broadly. Grounded in the
  project's stated design-systems philosophy: **semantic over
  literal**, **naming is architecture**, **one concept = one name**,
  **clear over clever**. Owns the "right name" decision where D4
  owns the "any token vs a literal" decision.

- **Acceptance criteria**:
  1. **Agent file**: `.claude/agents/evaluator-naming.md` exists.
  2. **Frontmatter** matches the existing `evaluator-*` shape:
     `name: evaluator-naming`, `role: evaluator`,
     `description:` (one-line — must explicitly call out
     **advisory by default**, mirroring `evaluator-react-api` and
     `evaluator-tokens` framing), `tools:` (Read, Glob, Grep, plus
     read-only Bash for `npm run lint:*`, `npm run build:*`,
     `git status:*`, `git diff:*` — no `lint:naming:*` since no
     script), `model: inherit`, `maxTurns: 5`.
  3. **Body — inherited base contract**: opens with an
     "Inherited base contract" section pointing at
     `evaluator-base.md`. Does NOT duplicate stance, verdict
     format, base flag taxonomy, or read-only-constraint content.
  4. **Body — Project context section**: names the established
     shared component family
     (`Stack`, `Grid`, `Spacer`, `Area`, `Text`, `Card`,
     `PageHeader`, `AppLayout`, `SketchNav`, `TopBar` —
     `components/shared/`), the file conventions
     (`sketches/NN-slug.tsx`, `components/<Category>/<Component>/
     <Component>.{tsx,module.css}`, kebab-case scripts in
     `.claude/scripts/`), and the canonical token-naming pattern
     (semantic namespaces: `space.xN`, `fg.*`, `bg.*`,
     `breakpoint.*`). Grounds the catalog in real project
     vocabulary so a reader doesn't have to infer.
  5. **Body — Process section**: 3–5 numbered steps walking the
     evaluator through (a) detecting naming scope (any
     `.tsx`/`.jsx`/`.ts` file, plus `.module.css` files where the
     class names are in scope), (b) running grep heuristics
     against the in-scope files for the catalog entries that
     support them, (c) manual inspection for entries that
     require reading context, (d) assembling verdict with all
     findings advisory by default unless the diff demonstrably
     regresses an already-named-well artifact back to a worse
     name.
  6. **Body — Antipattern catalog**: 6–8 entries. Each entry:
     pattern, symptom, impact, detection method (`grep` |
     `manual`), severity (`advisory` for all entries in this
     initial pass), flag code (prefixed `naming-`). Catalog must
     include at minimum:
     - **Visual-literal name** — a component, token, prop, or
       CSS-class name that describes appearance rather than
       meaning (e.g. `BlueButton`, `RedAlert`, `LargeCard`,
       `color.gray.200` at a use-site where
       `color.background.surface` exists, `.darkBar` instead of
       `.appHeader`). Owns the boundary with D4 (D4 = literal
       vs token; D5 = wrong token name among valid options, and
       visual-name antipatterns generally).
     - **Inconsistent vocabulary across siblings** — same concept
       exposed under different identifiers in the codebase
       (`Container` / `Wrapper` / `Box` when `Area` or `Stack`
       already covers it; `onTap` / `onPress` / `onClick` for
       the same interaction; `getUser` / `fetchUser` /
       `loadUser` for the same operation in one module).
     - **Hungarian / type-in-identifier** — `colorString`,
       `userArray`, `itemCount` as `iCount`, `isOpenBool` —
       binding the type into the identifier when TS already
       provides it.
     - **Boolean prop without predicate form** — a boolean prop
       named as a noun (`<Section header>` to mean "show a
       header") rather than as a predicate (`<Section
       showHeader>` or `<Section hasHeader>`). Confusing
       because the noun reads like a content slot.
     - **Abbreviation at public API surface** — `<Btn>`,
       `<Hdr>`, `useMsg`, `props.cls`, exported helpers like
       `fmt()`, `calc()` without disambiguating context.
       Internal locals may abbreviate; the public surface
       should be clear.
     - **Implementation-leaking name** — names that bind callers
       to the implementation choice (`useFormikSubmit`,
       `withMobxObserver`, `memoizedCache` as a public field,
       `withReactRouterParams`). Refactoring the implementation
       leaks through to consumer code.
     - **File / directory naming inconsistency** — a new file
       whose path doesn't follow the established convention for
       its location (e.g., a sketch named `mySketch.tsx`
       instead of `NN-slug.tsx`, a component dir named
       `pageHeader/` instead of `PageHeader/`, a script named
       `MyScript.ts` instead of `my-script.ts`).
  7. **Body — Flag codes section**: table listing each
     naming-specific code mapped to its catalog entry. Codes are
     `naming-`-prefixed and do NOT duplicate base flag taxonomy.
  8. **Body — Inspection signals section**: explicitly documents
     that no dedicated CLI signal exists for this evaluator (in
     contrast with `evaluator-a11y`'s `npm run test:a11y` and
     `evaluator-nextjs`'s `npm run lint:nextjs`). Names the
     rationale, same shape as `evaluator-react-api` and
     `evaluator-tokens`: advisory-only initial scope; no
     purpose-built naming-linter framework adopted (mirrors D1/D2/
     D3/D4 precedent against linter-framework adoption for
     evaluator detection); composability — consumer projects with
     their own naming vocabularies can compose this evaluator
     without inheriting a project-specific lint config.
  9. **Body — Carve-outs section**: explicitly documents the
     contextual exclusions a grep-based detector would otherwise
     false-positive on:
     - p5.js sketch files under `sketches/` — single-letter math
       variables (`n`, `t`, `x`, `y`, `p`), library callbacks
       (`setup`, `draw`), and idiomatic p5 abbreviations are not
       naming flags
     - Test files (`*.test.ts`, `*.spec.ts`) — framework
       conventions (`it`, `describe`, `expect`, single-letter
       fixture names) are not flags
     - Library-imposed names — `setup` / `draw` from p5,
       `loader` / `action` from React Router, `getServerSideProps`
       from Next.js, etc. — names dictated by external contracts
       are not in scope
     - Type-imported domain vocabulary — names that originate from
       an external schema (GraphQL types, OpenAPI definitions,
       generated bindings) are not this evaluator's lane;
       upstream owns them
     - Diff-only scope — pre-existing naming in unchanged code is
       not the artifact's responsibility; the artifact is the
       diff. Note pre-existing inconsistencies once at the
       bottom of the verdict for reviewer awareness, not as
       per-rule findings.
  10. **Boundary note**: brief section clarifying overlap with
      adjacent evaluators:
      - `evaluator-tokens` (D4) owns the literal-vs-token
        decision. D5 owns the right-token-name decision among
        valid tokens AND broader visual-literal naming across
        components and CSS classes.
      - `evaluator-react-api` (D3) owns React's API surface
        (Hooks rules, state mutation, ref-in-render). A hook
        named `useReactHookFormSubmit` is a D5 flag
        (implementation-leak), not a D3 flag.
      - `evaluator-a11y` (D1) owns accessibility outcomes. An
        ARIA role named with a non-spec value is a D1 flag, not
        a D5 flag.
      - Cross-evaluator dedup at panel level is D6's concern
        (the panel-precedence + naming-vs-tokens boundary
        documentation flagged in PLAN.md § Phase 2).
  11. **Verification**: `npm run lint` (Biome) clean. `npm run
      build` clean. `npm test` passes (no new tests since no new
      script). `git status` shows only
      `.claude/agents/evaluator-naming.md` and
      `projects/2026-05-02-agent-guilds/...` paths modified —
      no edits elsewhere.
