## Contract

- **Goal**: Author the `/guild-whiteboard` substrate primitive ‚Äî the
  third member of the `guild-*` family (after `/guild-spawn` and
  `/guild-validate`) ‚Äî and wire it as an **opt-in** step into both
  `/ev-loop-interactive` and `/ev-loop-confidence` via a phase-config
  `whiteboard:` field documented in PLAN.md phase entries. The
  primitive composes `/guild-spawn` to invoke a panel of `whiteboard-*`
  engineer agents in parallel against a shared filesystem artifact (the
  "whiteboard"), then collects each engineer's contribution into the
  artifact as **attributed sections**. Multi-round invocations
  (round 2+) pass the prior whiteboard state into the next round's
  brief so engineers can address contradictions.

  D1 is the **substrate-only** scope: skill body + supporting script +
  test + ev-loop integration prose. D1 does **not** author the six
  `whiteboard-*` engineer agents (D2's job) and does **not** run a
  real design phase end-to-end (D3 closes Phase 3 with that
  exercise). Per **L-004** the engineers are session-boundary-gated
  anyway ‚Äî D2 introduces them, D3 invokes them.

     - `name: guild-whiteboard`
     - `description:` one paragraph describing the primitive's role

     - `argument-hint: "engineers=<comma-separated names>

     - `allowed-tools: Skill, Read, Bash` (Skill to compose

  2. **Skill body** documents:
     - **Inputs** (matching `argument-hint`):
       - `engineers`: comma-separated `subagent_type` names of
         whiteboard engineers. Order preserved through to the
         attributed-sections output.
       - `brief`: shared task description handed to every spawned
         engineer (the design question being explored).
       - `whiteboard`: path (repo-relative) where the attributed
         artifact lives. If the file does not exist, the skill
         creates it with a topical header. If it exists, the skill
         appends a new `## Round N` block (where N is auto-detected
         from existing `## Round` headers + 1, OR explicitly
         specified via the optional `round` arg).
       - `round` (optional): explicit round number override.
         Defaults to auto-detect.
     - **Output format** (locked, same idea as `/guild-validate`):
       ```json
       {
         "whiteboard_path": "<path>",
         "round": <N>,
         "sections": [
           { "engineer": "<name>", "section": "<verbatim engineer body>" }
         ],
         "contradictions": []
       }
       ```
       `contradictions` is empty in v1 (forward-compat infrastructure;
       cross-engineer contradiction detection is a future-work case,
       documented as such in the skill body).
     - **Process** (5 steps):
       1. Validate inputs (engineers non-empty, brief non-empty,
          whiteboard path non-empty). Emit
          `guild-whiteboard-error: <code>` and exit on validation
          failure.
       2. Resolve round number: if `whiteboard` exists, parse
          existing `## Round N` headers; new round is
          max(existing) + 1. If `round` arg supplied, use it
          verbatim (allows explicit re-runs). If file does not
          exist, create it with `# Whiteboard: <inferred topic
          from brief first line>` header, and round = 1.
       3. If round > 1: read the existing whiteboard content and
          construct `per_agent_context` that prepends the prior
          rounds to each engineer's brief, with a clear
          "## Prior whiteboard state" delimiter so engineers know
          they're addressing prior input (and where to find
          contradictions).
       4. Compose `/guild-spawn` via the `Skill` tool with
          `agents=<engineers>` and `brief=<input brief>`. Pass
          `per_agent_context` for the round-2+ case so each
          engineer sees the prior whiteboard state. (For round 1,
          omit `per_agent_context` ‚Äî every engineer sees only the
          shared brief.)
       5. Pipe `/guild-spawn`'s output to
          `node .claude/scripts/guild/whiteboard.ts append` via
          stdin (JSON array of `{engineer, section}`); the script
          writes the attributed `## Round N` block to the file and
          returns the structured `{whiteboard_path, round,
          sections, contradictions: []}` result.
     - **What this skill does NOT do**:
       - Engineer role validation (any named `subagent_type` can be
         passed; non-engineers will produce no usable section but
         won't error ‚Äî caller's responsibility).
       - Contradiction detection (v1 future-work; documented).
       - Iteration / retries across rounds (caller orchestrates
         round 2+ invocations).
       - Whiteboard cleanup or archival (orchestrator's concern).
       - Direct writes by engineers ‚Äî engineers are read-only via
         their `tools:` allowlist; **the orchestrator does all
         writes** to the whiteboard via the helper script.

  3. **New script**:
     `.claude/scripts/guild/whiteboard.ts` exists and is executable
     as `node whiteboard.ts <verb> [args]`. Implements:
     - **`init <path> --topic=<str>`** ‚Äî create the whiteboard
       file with `# Whiteboard: <topic>` header. No-op if file
       exists (idempotent). Exits 0 / writes
       `guild-whiteboard-error:` to stderr on failure.
     - **`detect-round <path>`** ‚Äî read file, return
       max-`## Round N`-header + 1 on stdout. Returns `1` if file
       does not exist OR has no round headers. Single integer
       output.
     - **`append <path>`** ‚Äî read JSON array from stdin (shape
       `[{ "engineer": "<name>", "section": "<text>" }]`), append
       a new `## Round N` block (N auto-detected) with each
       engineer's attributed `### From <engineer-name>` subsection.
       Emit the locked Result JSON on stdout.
     - **`read-state <path>`** ‚Äî read the whiteboard file, parse
       existing rounds, return JSON
       `{ rounds: [{ number, sections: [{engineer, section}] }] }`.
       Used by the skill for round 2+ context construction.
     - Hardcoded fallback / defensive error paths for unreadable
     file / unparseable input (mirroring `parse-and-aggregate.ts`'s
     error convention).

- **Architecture choice ‚Äî engineers are read-only**. Per the
  substrate convention already established by `evaluator-*`,
  whiteboard engineers READ the prior whiteboard state (via
  `per_agent_context` in round 2+) but never WRITE to the file
  directly. The orchestrator (the skill body) calls
  `whiteboard.ts append` after `/guild-spawn` returns; that
  script does all writes. This sidesteps race conditions on
  parallel writes and matches the read-only stance of
  `evaluator-base.md`. The skill body documents this design
