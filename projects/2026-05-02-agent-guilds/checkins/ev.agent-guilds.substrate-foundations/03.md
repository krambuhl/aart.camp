## Contract

- **Goal**: Author the `guild-validate` substrate skill — the antagonist panel coordinator that composes `guild-spawn` to run evaluators in parallel, parses each evaluator's verdict, and aggregates findings into a single panel verdict (`approved` | `flagged` | `flagged-conflict`) with blocking/advisory split and conflict surfacing. The output shape is locked to the design plan's full specification; v1 evaluators that don't emit the rich tags get parsed with a documented compatibility rule.

- **Acceptance criteria**:
  - `.claude/skills/guild-validate/SKILL.md` exists with valid frontmatter (`name`, `description`, `argument-hint`, `user-invocable: false`, `allowed-tools` including the `Skill` tool — required to invoke `guild-spawn`).
  - Inputs documented: `agents` (comma-separated evaluator `subagent_type` names), `packet` (the standard three-section evaluation packet: Contract / Artifact / Original ask), `precedence` (optional ordered evaluator list for non-conflicting overlap resolution).
  - **Output shape locked to the design plan's full specification**: `verdict` (`approved` | `flagged` | `flagged-conflict`), `blocking_findings`, `advisory_findings`, `cli_runs`, `conflicts`. Documented even when v1 will populate some fields as empty.
  - Process: (1) validate inputs → (2) spawn evaluators in parallel via `guild-spawn` (passing the same packet as the brief to each) → (3) parse each evaluator's returned verdict → (4) aggregate per the rules → (5) return the structured output.
  - **v1 compatibility rule documented**: if an evaluator emits the legacy `VERDICT: approved` / `VERDICT: flagged` shape without explicit BLOCKING/ADVISORY tags or a structured CLI runs section, treat all flagged reasons as blocking findings; populate `advisory_findings` and `cli_runs` as empty arrays. Phase 2 evaluators with antipattern catalogs and CLI validators will emit the richer tags natively.
  - Aggregation rules documented:
    - Any blocking finding from any evaluator → panel verdict = `flagged`, with consolidated `blocking_findings` list.
    - Only advisory findings (no blockers) → panel verdict = `approved`, with `advisory_findings` preserved separately.
    - Two or more evaluators emit blocking findings on the same code/scope with incompatible remedies → `flagged-conflict`, with the `conflicts` list populated.
    - No findings of any kind → `approved`, all lists empty.
  - **Conflict detection in v1**: documented as future-work. In the single-evaluator panels of v1 (`[evaluator-contract-fit]`), `flagged-conflict` cannot fire because there's only one evaluator. The mechanism for detecting conflict between multiple evaluators (e.g. same file:line flagged with incompatible remedies) is a Phase 2 concern; the skill body acknowledges this with a note that conflict detection is a no-op for single-evaluator panels.
  - Body explicitly does NOT do: iteration / retries (caller owns retry budget; loops decide whether to re-spawn flagged units), conflict auto-resolution (loops decide policy — surface to human, auto-pick by precedence, or whatever the style layer prefers).
  - **Composes `guild-spawn`** rather than spawning agents directly via the `Agent` tool. This is non-negotiable: the substrate's value is the layering, and `guild-validate` is the primary demonstration that one substrate skill can compose another.
  - Style-neutral framing — body refers to "evaluators" (their named role) and may mention `evaluator-*` family members as natural examples, but does not encode any specific evaluator's rubric or hardcode any agent name.
