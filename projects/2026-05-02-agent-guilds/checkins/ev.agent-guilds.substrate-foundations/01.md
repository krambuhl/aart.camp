## Contract

- **Goal**: Extract shared evaluator constraints into a reusable `evaluator-base.md` file, and introduce `evaluator-contract-fit.md` whose body references the base. Establishes the pattern future domain evaluators (a11y, tokens, naming, etc.) will follow. Preserves the existing `evaluator.md` unchanged for the duration of Phase 1; Deliverable 4 retires it once the ev-loop skills are migrated.

- **Acceptance criteria**:
  - `.claude/agents/evaluator-base.md` exists and contains: read-only constraint statement, no-praise / no-clarifications stance, packet-incomplete handling, verdict format (approved/flagged shapes), flag taxonomy starter set, isolation framing. Domain-agnostic (no contract-fit-specific rubric).
  - `.claude/agents/evaluator-contract-fit.md` exists with frontmatter `name: evaluator-contract-fit` and a body that (a) instructs the agent to read `.claude/agents/evaluator-base.md` and apply its constraints, (b) preserves the contract-fit-specific rubric (re-read contract → inspect artifact → check acceptance one-by-one → check disqualifiers → check ask-alignment → check rule adherence), (c) preserves the contract-fit-specific tools allowlist.
  - `.claude/agents/evaluator.md` is preserved verbatim. (Bridge file — deliberate temporary duplication keeps the in-flight loop functional. Deliverable 4 deletes it once the loop is migrated.)
  - No semantic behavior change: a contract-fit evaluation against `evaluator-contract-fit` should produce the same verdict it would have under `evaluator` for the same packet.
  - Agent files conform to substrate frontmatter conventions: `name:`, `description:`, `tools:`. The `role:` convention from PLAN.md is added to both new files (`role: evaluator` for contract-fit, `role: evaluator-base` for the base, used as a marker that the file is a base reference rather than a callable evaluator).

- **Rules applied**:
  - `npm run lint` (Biome) clean. (Agent files are Markdown; lint is a baseline check.)
  - Agent file frontmatter conventions: existing fields (`name:`, `description:`, `tools:`, `model:`, `maxTurns:`) preserved where present.
  - `~/.claude/CLAUDE.md` and `aart.camp/.claude/CLAUDE.md` conventions: no emojis, no speculative abstractions, no comments-for-comments-sake.

- **Disqualifiers**:
  - The existing `evaluator.md` is deleted or modified in this deliverable. (Must remain functional until Deliverable 4.)
  - `evaluator-contract-fit.md` body's reference to the base creates a spawn-time failure (e.g. wrong tools allowlist for a Read; circular instruction; missing file).
  - `evaluator-base.md` introduces logic specific to contract-fit. It must be domain-agnostic — every future evaluator-* file should be able to inherit it without conflict.
  - Behavior drift: any rubric, tool restriction, or flag code present in the original `evaluator.md` is silently dropped between the two new files.

## Execution

Decomposed `evaluator.md` into two pieces:

1. **`evaluator-base.md`** — domain-agnostic shared contract. Contains
   the stance (skeptical, terse, no praise, no scope creep,
   don't-second-guess), the evaluation packet shape (Contract /
   Artifact / Original ask), the verdict format (approved / flagged
   shapes), the read-only constraint, the shared flag taxonomy starter
   set, and the "what you never do" list. Frontmatter
   `name: evaluator-base`, `role: evaluator-base` (marker that this is
   a base reference, not a callable evaluator with a domain rubric),
   tools restricted to `Read, Glob, Grep`.

2. **`evaluator-contract-fit.md`** — contract-fit-specific rubric.
   Body opens with an instruction to read `evaluator-base.md` and
   apply its constraints, then defines the six-step contract-fit
   process (re-read contract → inspect artifact → check acceptance
   criteria one-by-one → check disqualifiers → check original-ask
   alignment → check rule adherence). Tools allowlist preserves the
   original verification commands from `evaluator.md`. Frontmatter
   `name: evaluator-contract-fit`, `role: evaluator`. The body notes
   that no new flag codes are introduced — the rubric maps cleanly to
   the shared codes from the base.

`evaluator.md` itself was not touched. It remains the agent the
in-flight loop spawns as `subagent_type: evaluator`. Deliverable 4 of
this phase will migrate the loop to `subagent_type: evaluator-contract-fit`
and then delete `evaluator.md`.

## Notes for PR

- This deliverable establishes the `evaluator-*` family pattern. Future
  domain evaluators (a11y, tokens, naming, react-api, nextjs) will
  follow the same shape: read `evaluator-base.md` first, then layer a
  domain-specific rubric.
- The `role:` frontmatter field is a substrate convention introduced
  here. Nothing reads it yet; `guild-validate` (unit 3) will use it to
  validate that spawned agents are role-appropriate.
- `evaluator-base.md` is callable as a subagent type (`name: evaluator-base`)
  but carries no domain rubric, so calling it directly produces no
  useful evaluation. It exists primarily as a reference file other
  evaluators read at spawn time.
- The original `evaluator.md` is preserved verbatim through Phase 1.
  Deliverable 4 will retire it once the ev-loop migration is verified.
