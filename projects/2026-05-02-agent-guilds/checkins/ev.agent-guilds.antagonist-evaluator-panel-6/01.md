- **Acceptance criteria**:
  1. **File**: `.claude/agents/PANEL-COMPOSITION.md` exists.
  2. **No agent frontmatter**: the file has no `---` block with a
     `name:` field, so Claude Code's agent loader does not try to
     register it as a callable `subagent_type`. May begin with a
     plain `# Panel composition` heading.
  3. **Intro section**: 2–4 sentences naming what this file is
     (substrate doc for the evaluator-* family), what it's not
     (not an agent), and who reads it (`/ev-loop-*` skills'
     auto-derivation step in D7; `/guild-validate` for conflict
     precedence; humans authoring panels manually).
  4. **File-type → evaluator mapping section**: a table mapping
     file patterns to the set of evaluators that should compose
     into the panel for an artifact touching that pattern.
     Must cover at minimum:
     - `evaluator-contract-fit` is ALWAYS included (baseline lens)
     - `.tsx` / `.jsx` → + `react-api`, `naming`, `a11y` (when
       JSX renders visible UI), `nextjs` (when Next-aware: route
       file, `'use client'` boundary, `getServerSideProps`-shaped
       export)
     - `.ts` → + `react-api` (when imports `react`), `naming`
     - `.module.css` → + `tokens`, `naming`
     - `.css` (non-module) → + `tokens`
     - Files under `sketches/` → carve-outs in each evaluator
       handle this; the composer still adds the appropriate
       lenses based on file extension, the in-evaluator carve-
       outs prevent firing on p5 idioms
     - `.md`, `.json`, `package.json`, lock files → contract-fit
       only (no domain evaluator adds value)
     - `.claude/agents/*.md`, `.claude/skills/*/SKILL.md`,
       `projects/**/checkins/**/*.md` → contract-fit only (this
       is substrate authoring; domain evaluators don't apply)
     - Scripts under `.claude/scripts/` → contract-fit + `naming`
       (script identifiers are public-API surface)
  5. **Union rule**: when an artifact touches multiple file types,
     the composed panel is the UNION of per-file evaluator sets,
     plus the always-on `contract-fit`. Duplicates dedup to one
     entry per evaluator. Documented explicitly so a reader doesn't
     have to infer.
  6. **Precedence list section**: an ordered list of evaluators
     from highest to lowest precedence, with a one-line rationale
     for each position. Used by `/guild-validate` for overlap
     resolution (when two evaluators flag the same `file:line`
     with compatible remedies, the higher-precedence one's
     finding is reported). Order:
     1. `evaluator-contract-fit` (baseline — contract violations
        trump everything else)
     2. `evaluator-a11y` (user-impact; accessibility failures harm
        real people)
     3. `evaluator-nextjs` (framework correctness; if Next breaks,
        nothing ships)
     4. `evaluator-react-api` (runtime correctness; non-crashing
        antipatterns still get heard)
     5. `evaluator-tokens` (design-system drift)
     6. `evaluator-naming` (rhetoric / readability)
  7. **Conflict-vs-overlap clarification**: a brief note
     distinguishing compatible overlap (dedup by precedence) from
     incompatible conflict (where applying both remedies would
     brick the artifact — surfaces as `flagged-conflict` per
     `/guild-validate`'s aggregation rules; v1 is documented
     no-op, real implementation deferred to when conflicts
     actually show up).
  8. **Tokens-vs-naming boundary section**: consolidates the
     boundary text that already appears in
     `evaluator-tokens.md` and `evaluator-naming.md`. Two
     paragraphs minimum:
     - **D4 owns the literal-vs-token decision.** Is the artifact
       using ANY token vs a hardcoded literal? Hex literals,
       named CSS colors, hardcoded `px`/`rem`/`em` spacing,
       hardcoded `@media` breakpoints, inline literal `style={}`.
       Examples with flag codes.
     - **D5 owns the right-name decision and broader naming.**
       Given two valid tokens, which is semantically correct
       (`color.background.surface` over `color.gray.200`)? Plus
       component naming (`<BlueButton>` → `naming-visual-
       literal`), file conventions, inconsistent vocabulary,
       implementation-leaking identifiers, etc.
     - **Concrete examples** showing which evaluator owns which
       artifact pattern.
  9. **How auto-derivation reads this spec section**: a brief
     description of the mechanical algorithm an /ev-loop-* skill
     (D7) would follow to derive a panel from an artifact's file
     list: (a) collect all changed/created files, (b) for each
     file, look up the applicable evaluator set from § 4, (c)
     union the sets, (d) always-include `contract-fit`, (e)
     order the resulting list by the precedence section. Not a
     runtime impl — just the spec D7 implements.
  10. **Cross-references**: short links back to each evaluator's
      own "Boundary with adjacent evaluators" section, so a
      reader can drill into the per-evaluator nuance.
  11. **Verification**: `npm run lint` clean. `npm run build`
      clean. `npm test` passes. `git status` shows only
      `.claude/agents/PANEL-COMPOSITION.md` and
      `projects/2026-05-02-agent-guilds/...` paths.

## Notes for the PR

- **Doc location**: `.claude/agents/` was chosen as a sibling to

  loader skips it (verified by reading the loader behavior in
  evaluator-base.md's "Not directly callable as a normal
  evaluator" disclaimer pattern — `evaluator-base.md` DOES

  and top-level `.claude/PANEL-RULES.md`) and dropped for the
  reasons captured in Execution § 2.

- **Boundary text is consolidated, not relocated**. Each
  evaluator's own "Boundary with adjacent evaluators" section
  remains the source of truth for per-evaluator nuance.
  PANEL-COMPOSITION.md adds the cross-cutting panel-level
  perspective without removing or rewriting the per-evaluator
  text. The cross-references section at the bottom points
  readers back to each evaluator file for depth.

- **D7-readiness note from the panel**: the `.tsx` mapping row
  has two conditional predicates (a11y "renders visible UI",
  nextjs "Next-aware") that don't have clean syntactic anchors.
  D7's implementation should either (a) make these always-on
  for `.tsx` and let each evaluator's own carve-outs suppress
  irrelevant findings, or (b) add explicit detection (grep for
  `<button|<a |<form|role=` for the a11y predicate; grep for
  `'use client'` / `^export (async )?(function|const)
  (getServerSideProps|getStaticProps|generateMetadata|loader|
  action)` for the nextjs predicate). Option (a) is simpler
  and lets the evaluators' existing scope-detection logic do
  its job; option (b) saves panel budget at the cost of
  brittle path-grep heuristics. D7's contract should pick.

  prior unit), D8 will exercise it for the FIRST time —
  technically the registry has cached it as of session start,
