## Contract

- **Goal**: Wire the **capture half** of the substrate feedback loop:
  evaluator findings flow from a `/guild-validate` verdict into
  `.guild-findings.jsonl` (frequency counter), and from there — once
  a signature crosses the recurring threshold — into the existing
  `learnings/session-notes/` capture pipeline as a classified note.
  Also extend `griot-compact` with **classification-aware promotion
  rules**, but defer the rollup section authoring (D2) and the
  `/griot-use` injection (D2) and the composability-proof loop (D3).
  D1 lands the data plumbing; D2 lands the surface it feeds; D3
  proves substrate is independently composable.

  **Scope per Round-1 whiteboard decisions**:
  - **Two classifications implemented** in v1: `recurring` +
    `generator-antipattern`. The other three (`catalog-gap`,
    `evaluator-conflict`, `sanctioned-exception`) are
    **named-but-unimplemented** — capture.ts MUST accept the
    classification arg-shape so the API surface is stable from day
    one, but invocations error with `not-yet-supported: <name>`. This
    keeps the contract honest about what works versus what's named.
  - **Decomposition is coarse**: D1 is "capture + classify + count";
    D2 is "rollup + injection + A/B verify"; D3 is "composability
    proof". This unit takes the largest chunk because the data plumbing
    is the load-bearing piece.
  - **Naming reframes from design-systems adopted**: classification
    names are `recurring` / `generator-antipattern` /
    `catalog-gap` / `evaluator-conflict` / `sanctioned-exception` —
    NOT the PLAN.md original spellings (`frequency` / `conflict` /
    `exception`). PLAN.md's prose is from before the Phase 5
    whiteboard; the round-1 reframes supersede.
  - **`AP-NNN` identifier scheme** for project antipatterns (distinct
    from learnings' `L-NNN`) is reserved for D2, but D1's capture
    output must include enough metadata that D2 can assign IDs
    deterministically.

- **Acceptance criteria**:

  1. **New helper script** at `.claude/scripts/guild/findings.ts`
     exists with two verbs:
     - `append --slug=<slug> --signature=<sig> --evaluator=<name> [--code=<code>] [--evidence=<text>]`
       — writes one JSONL row to
       `projects/<slug>/.guild-findings.jsonl`. Creates parent file
       on first call. Idempotent on identical-row second-call only if
       the row's timestamp differs (we ARE counting frequency, so
       duplicates are expected and desired).
     - `count --slug=<slug> --signature=<sig>` — returns integer
       count of rows matching `signature` for that slug on stdout.
       Used by ev-loop to detect threshold crossings.
     Sibling test file at `.claude/scripts/guild/findings.test.ts`
     covers both verbs + error paths, run via vitest.

  2. **JSONL row shape**:
     ```json
     {"ts":"<ISO-8601>","slug":"<project>","branch":"<branch>","unit":"<checkin-NN>","evaluator":"<agent-name>","code":"<flag-code>","signature":"<sha1-of-evaluator+code+normalized-evidence>","evidence":"<one-line>","severity":"blocking"|"advisory"}
     ```
     The `signature` field is what `count` queries against. Computing
     it locally inside `findings.ts` (sha1 of canonical concatenation)
     keeps the script self-contained.

  3. **Capture script extended**: `.claude/scripts/griot/capture.ts`
     accepts a new argument group:
     - `--evaluator-finding=<classification>` — required
       classification value; one of the five names. Mutually
       exclusive with `--correction-text=...` (existing arg).
     - `--evaluator-name=<name>` — required for evaluator findings.
     - `--code=<code>` — required for evaluator findings.
     - `--evidence=<text>` — required for evaluator findings.
     - `--file-line=<path:line>` — optional; included in the captured
       note's body when present.
     - `--frequency-count=<N>` — required when classification is
       `recurring`; reflects the count that triggered the capture.
     The script writes a session-note exactly like today's
     `--correction-text` flow, but with an additional frontmatter
     field `classification: <value>` in the captured note. The
     captured note's body documents the finding in the same plain-
     prose style as today's correction notes — the classification
     is metadata, not formatting.

  4. **Two classifications work end-to-end**:
     - `recurring` — capture.ts accepts the args, writes a note with
       `classification: recurring` frontmatter, includes
       `frequency-count` in the body. Test asserts the note file
       exists with the correct frontmatter and body shape.
     - `generator-antipattern` — capture.ts accepts the args, writes
       a note with `classification: generator-antipattern` frontmatter.
       Test asserts the note file exists with the correct frontmatter.

  5. **Three classifications named-but-unimplemented**: invoking
     capture.ts with `--evaluator-finding=catalog-gap` (or
     `evaluator-conflict`, or `sanctioned-exception`) exits non-zero
     with stderr `capture-error: not-yet-supported: <classification>`.
     Test covers all three.

  6. **`griot-compact/SKILL.md` extended** with classification-aware

     - `catalog-gap` / `evaluator-conflict` / `sanctioned-exception`

  7. **ev-loop-interactive SKILL.md extended** at process step 4
