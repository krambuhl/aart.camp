## Architectural findings (load-bearing for Phase 1.5+)

### Finding 1: `disable-model-invocation` blocks ALL model-initiated invocations, including transitive Skill tool calls

Empirically verified this session. Built a test fixture pair:
- `_test-disabled-child` with `disable-model-invocation: true`
- `_test-parent-invoker` (no flags) that tries to invoke the child via the `Skill` tool

Result: the parent's invocation failed with `Skill _test-disabled-child cannot be used with Skill tool due to disable-model-invocation`. Test fixtures were cleaned up after.

Implications:
- A skill cannot be both "composable from another skill" AND "blocked from ambient auto-discovery" simultaneously.
- The original directive ("don't disable model invocation on anything composed by other skills") is literally correct.
- For substrate that wants composition-only behavior, the only path is to use a different primitive (script or subagent) — there's no flag combination that achieves it.

This is captured as a project memory at `~/.claude/projects/-home-krambuhl-work-aart-camp/memory/project_disable_model_invocation_blocks_composition.md`.

### Finding 2: Skills as interfaces vs workers — the architectural principle

Skills are useful as stable, addressable named entry points. They are often the wrong primitive for the work itself. Per the user's strongly-held convention:

- **CRUD-shaped substrate** (read/edit/write files; deterministic; no LLM judgment) → write as a Node/bash script. No skill ambient surface. Callers shell out via `Bash`.
- **LLM-shaped substrate** (judgment, taste, narrative, classification) → stays as a skill.
- **Interactive substrate** (multi-turn user conversation) → must stay as a skill in the main thread.
- **Orchestration substrate** (composes Skill + Agent in same context) → stays as a skill. Guild substrate is the canonical example.

### Finding 3: Most "LLM-shaped" skills have a CRUD epilogue worth extracting

The model produces something (PR body, retro, plan) and then there's a deterministic tail (write file, invoke other substrate, update state). That tail is script-shaped even when the core is LLM. This is the "shell extraction" pattern — applies after the obvious pure-CRUD migrations.
