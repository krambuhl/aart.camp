The substrate-script convention got bootstrapped here because deliverable 2 is the first migration: a "Substrate scripts: layout and conventions" subsection in `projects/CONVENTIONS.md`, an `npm run test` script in `package.json`, the `Bash(node .claude/scripts/trout/*)` permission in project-wide `.claude/settings.json`, and a nested `.claude/scripts/package.json` with `{"type": "module"}` as a stopgap. The stopgap caveat is documented inline — adding `"type": "module"` to the root broke `next.config.js` and `postcss.config.js`, so we scoped ESM to substrate scripts via the nested package.json instead. The user explicitly framed the nested approach as transitional; the eventual target is root-level ESM with the config files migrated.

Two course corrections shaped the convention. First: I had proposed `npx tsx` for invoking the `.ts` scripts, anchoring on the project's existing `tsx` usage. The user prompted the right question ("can Claude execute ts scripts using tsx?") and verifying showed Node 24 strips type annotations natively — `node script.ts` works directly. Dropped tsx from the convention. Second: I had proposed a hybrid (shared TS const file for the event vocabulary plus a drift-detection test against CONVENTIONS.md). The user called it silly and suggested just parsing CONVENTIONS.md at runtime. Benchmarked at ~0.017ms per parse — single source of truth wins. Both corrections are captured as `correction:` lines in the checkin's Notes for the PR and now live in `learnings/session-notes/`.
