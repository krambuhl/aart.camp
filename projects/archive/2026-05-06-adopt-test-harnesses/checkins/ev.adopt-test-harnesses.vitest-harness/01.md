## Contract

- **Goal**: Install vitest and the React testing toolchain, configure them
  for this Next.js + tsx + CSS Modules setup, and add an `npm test` script.
  End state: `npm test` runs against an empty/trivial test surface
  successfully and the configuration is ready to host real tests added in
  D2.

- **Acceptance criteria**:
  - `vitest` installed as a devDependency
  - `@testing-library/react`, `@testing-library/jest-dom`, and `jsdom`
    installed as devDependencies (so D2 and Phase 2 can write component
    tests without re-tooling)
  - A `vitest.config.ts` at repo root with: `jsdom` environment, the `@/*`
    path alias resolved to match `tsconfig.json`, and CSS Modules handled
    sensibly (via vite's built-in handling — no custom mock plumbing
    unless needed)
  - A `vitest.setup.ts` (or equivalent) that imports `@testing-library/jest-dom`
    so `expect(...).toBeInTheDocument()` etc. work in D2
  - `package.json` has `"test": "vitest run"` and `"test:watch": "vitest"`
    scripts (run-once vs watch, standard split)
  - `npm test` exits 0 against zero tests (vitest's default behavior, or
    `--passWithNoTests` if needed)
  - `npm run build` still succeeds
  - `npm run lint` (Biome) is still clean — vitest config files are either
    formatted to pass Biome or ignored as appropriate

- **Rules applied**:
  - Project CLAUDE.md: simple over clever, no over-engineering, "good
    enough for now"
  - User CLAUDE.md: composition over configuration, don't add abstractions
    you don't need
  - Verification: `npm run lint`, `npm run build`, `npm test`
    (newly added)

- **Disqualifiers**:
  - `npm test` doesn't actually run vitest
  - Build or lint regress
  - Config pulls in plugins or libraries not needed for this Next.js + tsx
    + CSS Modules setup (no `@vitejs/plugin-react-swc`, no Storybook
    integrations, no coverage tooling — those are out of scope)
  - Config pre-bakes decisions for things we're explicitly deferring (no
    coverage thresholds, no test name patterns assuming a directory
    structure we haven't picked yet)
  - "Globals" mode enabled in vitest config — prefer explicit imports
    (`import { describe, it, expect } from 'vitest'`) for readability

- **Inputs**:
  - `package.json`
  - `tsconfig.json` (path alias: `@/*` → `./*`)
  - `next.config.js` (verify nothing affects test config)
  - `biome.json` (verify our new config files don't trip lint)

## Execution

Installed `vitest@4.1.5`, `@testing-library/react@16.3.2`,
`@testing-library/jest-dom@6.9.1`, and `jsdom@29.1.1` as devDependencies.
RTL + jest-dom front-loaded so D2 and Phase 2 component tests don't need
a second install round.

Created `vitest.config.ts` at repo root:
- `environment: 'jsdom'` for DOM-aware tests later
- `setupFiles: ['./vitest.setup.ts']` to register jest-dom matchers
- `resolve.alias: { '@': import.meta.dirname }` mirroring the
  `@/* → ./*` alias in `tsconfig.json`. Uses Node 24's
  `import.meta.dirname` (no `fileURLToPath` boilerplate)
- No `globals: true` — explicit `import { describe, it, expect } from 'vitest'`
  is the convention for this repo
- No coverage config, no plugins beyond what vitest brings — out of scope

Created `vitest.setup.ts`: single line importing
`@testing-library/jest-dom/vitest` (the vitest-specific entry that wires
matchers via vitest's `expect.extend`).

Added `package.json` scripts:
- `"test": "vitest run --passWithNoTests"` — `--passWithNoTests` is
  needed for D1 (no tests yet) and is harmless once D2 lands
- `"test:watch": "vitest"` — standard run-once vs. watch split

CSS Modules: vite's built-in CSS handling covers it transparently for
the unit-test surface we'll exercise. No custom transformer or mock
plumbing added — we'll deal with it if/when a test actually imports a
`.module.css` file and proves there's a real problem.
