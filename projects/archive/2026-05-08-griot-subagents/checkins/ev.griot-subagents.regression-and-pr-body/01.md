  2. **Regression suite section** (§3) follows the SDK shape but
     adapts to subagent dispatch:
     - Parse `learnings/rollup.md` to extract the list of `L-NNN`
       entries: each entry's `id`, `origin_slug` (from the
       `Origin: <slug>` line), and `rubric` text (from the
       `### Rubric` section).
     - Read the prior bench-history line's `passing_ids` set
       (the most recent line in `learnings/bench-history.jsonl`,
       parsed as JSON, `passing_ids` field, default empty if the
       file or field is missing). This set drives "new failure"
       detection — entries that were passing last run and are
       failing now.
     - For each entry, in order:
       - Read `learnings/session-notes/archived/<origin_slug>/prompt.md`.
         If missing, skip this entry (cannot regression-test
         without an origin prompt).
       - Spawn ONE `general-purpose` subagent at
         `test_subject.tier` for the treatment output. System
         prompt: `"You are Claude. Apply the validated learnings
         below where relevant: <full contents of rollup.md>"`.
         User prompt: the entry's origin prompt.
       - Spawn ONE `general-purpose` subagent at `opus` tier
         for the pass check (the SDK pinned this to opus —
         `top_tier`). System prompt:
         `"You evaluate whether a Claude output passes every
         assertion in a rubric. Binary judgement, no hedging."`
         User message contains the entry's rubric and the
         treatment output. The subagent's response ends with a
         fenced ` ```regression-check ` block containing JSON
         `{all_pass: boolean, failed_assertions: string[],
         reasoning: string}`. Extract and parse the block. Block
         extraction failure → skip this entry's regression
         result (don't count as either passing or failing) and
         log a JSONL intervention via `log-intervention` with
         category `regression_check_failed`.
       - If `all_pass: true` → increment a `passing` counter
         and add `id` to a `current_passing_ids` set.
       - If `all_pass: false`:
         - Increment a `failing` counter.
         - If `id` was in the prior `passing_ids` set → this
           is a NEW failure (a regression). Log a JSONL
           intervention via `log-intervention` with category
           `regression`, record shape `{learning_id: id,
           failed_assertions: <array>, reasoning: <text>,
           was_passing_in_prior_run: true}`. Increment a
           `new_regressions` counter.
         - If `id` was NOT in prior `passing_ids` → chronic
           failure. Do NOT log (avoids noise on every run).
           Just count toward `failing`.
     - At the end of §3, three counts and one set are
       captured for §4 / §5 use:
       `regressions_total = passing + failing` (entries
       attempted), `regressions_passing = passing`,
       `regressions_failing = failing`, `regressions_new =
       new_regressions`, `current_passing_ids =
       <set serialized as sorted array>`.
  3. **Compose nightly PR body section** (§4) instructs the
     skill to compose a markdown blob in this shape and emit it
     to the user via the skill response (no file write):

     ```markdown
     # Nightly /griot-compact run — <today's date YYYY-MM-DD>

     ## Run summary
     - Notes processed: <N>
     - Promoted: <N>
     - Did not reproduce: <N>
     - Escalated: <N> (<reason breakdown>)
     - No consensus: <N>
     - Rollup entries checked: <regressions_total>
       (passing: <P>, failing: <F>, new regressions: <NR>)

     ## Promoted learnings
     - L-NNN — "<short title>" (from <slug>) — attempt <M>
     [...for each note marked promoted; or "(none)" if N=0]

     ## Escalations needing review
     - <slug>: <reason>
     [...for each escalation; or "(none)"]

     ## New regressions
     - L-NNN — <count> failed assertions
     [...for each new regression; or "(none)"]

     ## Run metadata
     `<bench-history record JSON, single line>`
     ```

     The blob is the entire skill output to the user at the end
     of the run. Below the blob, the skill adds a one-line
     pointer: "Copy the section above into a GitHub PR. Diagnosis
     detail is in `learnings/operator-log.jsonl` and
     `learnings/regressions.jsonl`."

## Notes for PR

- This deliverable closes Phase 2. After this merges, the new
  `/griot-compact` is feature-complete relative to the SDK's
  `index.ts` flow (minus the explicitly-cut features:
  per-judge token telemetry and `learnings/runs/` transcripts).
- The regression suite is intentionally lighter than the
  per-note panel. SDK precedent: a single Opus pass-check per
  entry, sequential not parallel, treatment uses the full
  rollup (simulating what users actually see). We mirror this.
  Cost scales linearly with rollup size (2 subagent spawns
  per entry). For a 50-entry rollup, that's 100 spawns just
  for regression — meaningful but not prohibitive.
- Why no `griot-regression-check.md` subagent? Adding a 6th
  griot-* role in Phase 2 D3 would be scope creep on Phase 1
  D1's role-taxonomy decision. The pass-check semantics
  (binary "all assertions pass?" without control vs treatment
  comparison) are different enough from `griot-judge`'s shape
  that reusing it would require either an empty-control hack
  or post-hoc verdict interpretation. Inline `general-purpose`
  with an explicit system prompt is the simpler call.
- The PR body's `## Run metadata` section embeds the
  bench-history record JSON in single-line form. The same
  record appears in `bench-history.jsonl` via §5's append —
  the user sees it in the PR body and the file gets it on
  disk. Both reflect the same working-state record assembled
  before §4 emission.
- `passing_ids` is the cross-run hinge for "new failure"
  detection. Each run reads the prior run's `passing_ids` to
  know what was working last time, runs its own pass-check,
  and writes its own `passing_ids` for the next run to read.
  First-ever run has no prior, so the empty set means no
  regression flags fire (every entry is "new" — chronic, by
  the absence of a prior passing record). This matches the
  SDK's `previousPassingSet` semantics.
