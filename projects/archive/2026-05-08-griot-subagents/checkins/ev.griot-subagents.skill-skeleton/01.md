- **Goal**: Rewrite `.claude/skills/griot-compact/SKILL.md` from a
  thin `npm run learnings:compact` wrapper into the actual
  orchestrator skill that drives the new subagent-based pipeline.
  This deliverable lands the skeleton + happy path: pre-flight,

  resulting skill is invokable end-to-end on any session-note that
  reaches `IMPROVED` on the first or second panel round, AND on

  no `ANTHROPIC_API_KEY` reference anywhere.

  Composes the substrate landed in Phase 1: five `griot-*`
  subagents (D1), tier-based config schema (D2), `mediate-panel.ts`

- **Acceptance criteria**:
  1. `.claude/skills/griot-compact/SKILL.md` exists and is fully
     rewritten. Frontmatter preserves `name: griot-compact`,
     `user-invocable: true`, `disable-model-invocation: true`. The
     `description` is updated to remove "Expensive — spends real
     API tokens" and to mention pooled subscription tokens. The
     `allowed-tools` field is updated to include the tools the new
     skill actually needs: `Read`, `Write`, `Bash` (broad; the
     skill runs `node` for `mediate-panel.ts`, `ls`, file
     management), and `Agent` (for spawning subagents).
  2. Prerequisites section: removes the `ANTHROPIC_API_KEY`
     requirement and the `npm install` reference. Adds a one-line
     note that this pipeline runs on pooled Claude Code
     subscription tokens via subagents. The `learnings/config.yaml`
     reference is updated to call out the tier-based shape.
  3. **Pre-flight section**: instructs the running skill to
     - Read `learnings/config.yaml` and extract panel composition
       (`judges` array), consensus thresholds (`round_1_blind`,
       `round_2_debate`), tiebreak block (`rule`, `top_tier`),
       agent role tier assignments (`agents.rubric_author`,
       `agents.rewriter`, `agents.debate_summarizer`,
       `agents.operator`), and `test_subject.tier`.
     - List session-notes under `learnings/session-notes/`
       (excluding `archived/`).
     - If there are no unprocessed notes AND `rollup.md` is empty
       or absent, tell the user there's nothing to compact and
       stop.
  4. **Per-note pipeline section**: documents the steps for each
     session-note, in order:
     - **Step A — Rubric**: if the note's `rubric.md` is missing,
       spawn the `griot-rubric-author` subagent via the Agent
       tool with `model` matching `agents.rubric_author.tier`.
       Construct the brief from the note's `correction.md` and
       `wrong.md`. Extract the fenced ` ```rubric ` block from
       the response, write the assertions as the new
       `rubric.md`. If the rubric block fails to parse, log the
       failure and skip the note (this note's pipeline ends).
     - **Step B — Generate control + treatment**: spawn two
       Agent tool calls in parallel using the `general-purpose`
       subagent type with `model` matching
       `test_subject.tier`. The "control" call runs the note's
       prompt with no learning context; the "treatment" call
       runs the same prompt with the candidate learning injected
       as additional context. Capture both outputs as plain text
       — these become the inputs to the judge panel.
     - **Step C — Round 1 panel**: spawn 4 `griot-judge`
       subagents in parallel by issuing four Agent tool calls in
       a single tool-use message, one per entry in the
       `judges` array. Each spawn's `model` parameter matches
       the entry's `tier`. The brief includes: origin prompt,
       correction, candidate learning, rubric, control output,
       treatment output. Each judge's response should end with a
       fenced ` ```verdict ` block.
     - **Step D — Mediate round 1**: pipe the four raw outputs
       through `node .claude/scripts/griot/mediate-panel.ts` via
       Bash with stdin JSON shaped per the script's input
       contract: `{round_num: 1, verdicts: [...], config: {...}}`.
       Parse the JSON output. If `threshold_met` is true, jump
       to Step F (outcome). Otherwise continue to Step E.
     - **Step E — Round 2 panel** (only when round 1 didn't
       reach unanimity):
       - Spawn `griot-debate-summarizer` with the round 1
         per-judge verdicts and reasoning. Capture the fenced
         ` ```summary ` block.
       - Spawn 4 `griot-judge` subagents again, this time with
         the round 1 summary appended to each judge's brief.
       - Pipe through `mediate-panel.ts` with `round_num: 2`.
       - Parse the result. If `threshold_met`, the consensus is
         the verdict. If `tiebreak_applied`, the
         `tiebreak_verdict` is the verdict. Otherwise the panel
         had no consensus AND no tiebreak — record this in
         `learnings/operator-log.jsonl` with category
         `no_consensus` and skip the note.
     - **Step F — Outcome handling**: branch on the verdict
       arrived at in Step D or E.
       - `IMPROVED` → append a new entry to `learnings/rollup.md`
         in the existing format (see `scripts/learnings-compact/
         rollup.ts:appendToRollup` for the canonical entry
         shape). Move the session-note folder to
         `learnings/session-notes/archived/<original-name>/`.
         Mark this note as "promoted" in the run summary.
       - `DID_NOT_REPRODUCE` → append a JSONL line to
         `learnings/operator-log.jsonl` with category
         `did_not_reproduce`, archive the note. Mark as "skipped
         (did not reproduce)".
       - `UNCHANGED` or `REGRESSED` → log a JSONL line to
         `learnings/operator-log.jsonl` with category
         `deferred_pending_rewrite_loop` and a note that Phase 2
         D2 will handle the rewrite. Do NOT archive the note.
         Mark as "deferred (rewrite loop pending)".
  5. **End-of-run section**: documents the steps after all notes
     are processed:
     - Append a single line to `learnings/bench-history.jsonl`
       with shape `{ts, notes_processed, notes_promoted,
       notes_did_not_reproduce, notes_deferred,
       notes_no_consensus, run_kind: "compact-skeleton"}`. The
       `run_kind` field is forward-compat — D2 and D3 will
       update it.
     - Show the user a one-paragraph run summary listing each
       note's outcome.
     - Tell the user explicitly: "rewrite loop is not yet
       implemented; deferred notes will be picked up in Phase 2
       D2."
  6. **Do-not list**: preserved and updated to reflect the new
     architecture. Specifically: do not invoke the SKILL.md
     automatically; do not edit rubric.md files post-write; do
     not hand-promote learnings to rollup.md (only `IMPROVED` via
     the judge panel); do not modify `bench-history.jsonl` by
     hand. Removes the API-key reference.
  7. The skill body explicitly references the four substrate
     pieces it composes: `griot-*` subagents under
     `.claude/agents/`, `learnings/config.yaml`, and
     `.claude/scripts/griot/mediate-panel.ts`. Does NOT yet
     reference `.claude/scripts/griot/operator-checks.ts` (that's
     D2). Does NOT reference `scripts/learnings-compact/` (the
     SDK code is being abandoned; Phase 3 deletes it).
  8. The `learnings/pr-templates/` reference and the
     human-review-PR mention from the old skill are removed —

## Notes for PR

- This deliverable is a rewrite, not an addition. The `/griot-
  compact` skill that exists today (a thin wrapper around
  `npm run learnings:compact`) is replaced wholesale. Anyone who
  invokes `/griot-compact` after this PR merges gets the new
  subagent-driven pipeline — there is no backward-compat
  toggle.
- The new pipeline is end-to-end functional for any
  session-note that reaches `IMPROVED` (round-1 unanimity OR
  round-2 supermajority OR round-2 tiebreak) or
  `DID_NOT_REPRODUCE`. Notes that come back `UNCHANGED` or
  `REGRESSED` are surfaced and logged as deferred — D2 adds the
  rewrite loop that handles them.
- The skill calls Agent directly (not via guild-spawn) because
  the project explicitly chose to keep griot and guild as
  separate systems with no shared code. The parallel spawning
  pattern (4 judges in one tool-use message) is the same idea
  but implemented inline.
- `test_subject` uses `general-purpose` as the subagent_type
  with model override per `test_subject.tier` from config. We
  may want a dedicated `griot-test-subject` subagent in the
  future if the test subject's behavior diverges from
  general-purpose's defaults — flagging as future-work but not
  in scope for this deliverable.
- The bench-history `run_kind: "compact-skeleton"` field is
  forward-compat: it lets us distinguish runs made before the
  rewrite loop shipped from runs after, when reading
  bench-history later.
- Next deliverable (Phase 2 D2): rewrite loop + operator
  escalation. Replaces the `deferred_pending_rewrite_loop`
  branch with the full rewrite path: spawn `griot-rewriter`,
  use `operator-checks.ts verify-rubric` for tampering
  detection, spawn `griot-operator` after max_attempts for
  diagnosis, log all interventions through `operator-checks.ts
  log-intervention`.
