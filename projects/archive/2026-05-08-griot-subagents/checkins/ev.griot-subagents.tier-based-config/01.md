**Created**: 2026-05-08

**Unit**: Restructure learnings/config.yaml to tier-based schema

## Contract

- **Goal**: Rewrite `learnings/config.yaml` in place to a tier-based
  schema that the future Phase 2 orchestrator skill will consume. The
  judge panel shrinks from 7 version-pinned slots to 4 tier-based
  slots (2 Opus / 1 Sonnet / 1 Haiku). All other LLM roles
  (rubric_author, rewriter, debate_summarizer, operator,
  test_subject) likewise switch from `model: claude-X-Y` to
  `tier: <opus|sonnet|haiku>`. Consensus drops from three rounds
  (7/7 → 6/7 → 5/7) to two rounds (4/4 → 3/4) with an explicit
  tiebreak block. SDK-specific knobs that no longer apply to
  subagent dispatch (`weight`, `max_tokens`, `judge_timeout_ms`,
  `agent_timeout_ms`, `judge_parallelism`) are removed; subagent
  harness owns these concerns. The file is the only thing this
  deliverable changes.

  Acknowledged: this breaks `scripts/learnings-compact/config.ts`'s
  expected schema. The SDK script is non-runnable in the current
  environment (no `ANTHROPIC_API_KEY`) and is being deleted in
  Phase 3, so the schema break is intentional and accepted.

- **Acceptance criteria**:
  1. `judges:` is a 4-entry list. Two entries have `tier: opus`
     (`id: opus-A`, `id: opus-B`); one has `tier: sonnet`
     (`id: sonnet`); one has `tier: haiku` (`id: haiku`). No entry
     has a `model:`, `weight:`, or compound tier name
     (`top-current`, `top-previous`, etc).
  2. `agents:` has exactly four entries: `rubric_author`,
     `rewriter`, `debate_summarizer`, `operator`. Each is shaped
     `{ tier: <opus|sonnet|haiku> }`. No `model:` or `max_tokens:`
     fields. The pre-existing `mediator` key is gone — its LLM
     slice is now `debate_summarizer` (matching the subagent
     defined in D1).
  3. `test_subject:` is `{ tier: opus }`. No `model:` or
     `max_tokens:` fields. Comment preserved (or rephrased) noting
     this represents "the Claude your users actually use."
  4. `consensus:` has exactly two thresholds: `round_1_blind: 4`
     and `round_2_debate: 3`. The `round_3_final:` key is removed.
     A short comment explains the two-round shape and what happens
     on no-consensus (tiebreak or operator).
  5. `tiebreak:` block is added with shape
     `{ rule: top_tier_consensus, top_tier: opus }`. A short
     comment explains the rule.
  6. `execution:` block is removed entirely. The subagent harness
     owns timeouts and parallelism; config no longer carries them.
  7. `rewrite:`, `alerts:`, and `paths:` blocks are unchanged.
  8. No field anywhere in the file references a specific Claude
     version string (`claude-opus-4-7`, `claude-sonnet-4-6`,
     `claude-haiku-4-5-20251001`, etc.). All TODO comments about
     previous-gen IDs are gone.
  9. The file parses as valid YAML (verified via
     `node -e 'require("js-yaml").load(require("fs").readFileSync("learnings/config.yaml","utf8"))'`).

## Notes for PR

- This deliverable is one file. Small surface, but it's the contract
  the future Phase 2 orchestrator skill reads. Worth careful review
  of the new shape — adding fields later is easy; removing them
  after Phase 2 ships is harder.
- The tiebreak rule is named (`top_tier_consensus`) and the named
  top tier is in config (`top_tier: opus`). Phase 1 D3
  (`mediate-panel.ts`) implements this rule. If we add a second
  tiebreak rule later, the rule name dispatches.
- Consensus is two rounds, not three. Round 3 was explicitly cut as
  a yagni — if debate doesn't move 7-judge panels meaningfully, it
  wasn't going to move 4-judge panels meaningfully either, and the
  third round just delays escalation. Easy to add back if real runs
  show Round 2 → Round 3 conversion is meaningful.
- `agents.debate_summarizer` is the rename of what used to be
  `agents.mediator`. The LLM slice of the old mediator role is now
  this agent; the rest of mediator's procedural work moves to D3
  (`mediate-panel.ts`).
- `execution:` block removed entirely. If subagent runs hit
  rate-limits or harness timeouts in Phase 2, we'll surface that
  signal there and decide whether to add concurrency / timeout
  knobs back. None of those knobs apply to subagent dispatch
  semantically, so keeping them as dead documentation seemed
  worse.
- D3 (`mediate-panel.ts`) is the next deliverable; it will read
  this config (or accept it as input) and implement the consensus
  / tiebreak logic.
