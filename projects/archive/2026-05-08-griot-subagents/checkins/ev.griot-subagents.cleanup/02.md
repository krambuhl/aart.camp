# Checkin 02 — ev.griot-subagents.cleanup

**Created**: 2026-05-11
**Phase**: 3 — Cleanup
**Unit**: Delete `scripts/learnings-compact/`, drop the `learnings:compact` npm script, remove the `@anthropic-ai/sdk` dependency, and clean up obsolete prose references in skill and learnings docs.

## Contract

- **Goal**: Close the loop on the migration. With the new
  `/griot-compact` pipeline proven end-to-end by checkin 01's
  wet run, the SDK-era orchestrator and its API-key path are
  safe to delete. The deletion is small and satisfying — drop
  the directory, drop the npm script, drop the dep, update the
  prose that references the old path. After this checkin the
  repo has exactly one path to running the learnings panel
  (the skill), exactly one set of types feeding it
  (`learnings/config.yaml`), and zero `@anthropic-ai/sdk`
  imports anywhere in the tree.

  4. `.claude/skills/griot-compact/SKILL.md` no longer

     learnings:compact`" — that prohibition is obsolete once
     the script no longer exists. The companion line about
     not importing `@anthropic-ai/sdk` is preserved as a
     regression-prevention guard.
  5. `learnings/README.md` and `learnings/pr-templates/
     nightly.md` no longer reference `/learnings-compact` as
     the slash command — replaced with `/griot-compact` to
     match the actual skill name.
  6. **Pre-existing `npm run learnings:report` does not

## Execution

The deletion was small but raised two scope forks that needed
explicit resolution before completing.

**Fork 1: `scripts/learnings-report.ts` imports from the
deletion target.** The first grep for cross-file references
flagged that `scripts/learnings-report.ts:15` reads
`import { loadConfig, resolvePath } from './learnings-compact/
config';`. Deleting the directory would break
`npm run learnings:report` — a script the PLAN explicitly
preserves. Surfaced as a fork: (A) lift `config.ts` out before
delete, (B) inline the helpers into the report, (C) also delete
the report as legacy, (D) pause and revise PLAN. User picked
(A). The lift target is `scripts/learnings-config.ts` — sibling
to `learnings-report.ts`, single file, minimal surface. The
lifted version drops the SDK-era 7-judge validation
(`judges.length !== 7` throw and `round_1_blind !== 7` warn) —
those validations are wrong against the new tier-based config
and `learnings-report.ts` only consumes `config.paths.*`
anyway. The interface narrows to a `LearningsConfig` shape
that declares the five `paths.*` fields the report actually
uses, with a `[key: string]: unknown` escape hatch for any
yaml fields the report doesn't read. After the lift,
`learnings-report.ts`'s import updates from
`./learnings-compact/config` to `./learnings-config`. One-line
change in the consumer.

**Fork 2: `npm run learnings:report` runs but throws at
runtime against the new bench-history shape.** Discovered
post-deletion when running `npm run learnings:report` to
verify the lift didn't break the consumer. The lift worked
(the import resolves and `loadConfig()` returns the parsed
config), but the script reads `bench.verdict_counts.IMPROVED`
on every line — and the bench-history records written by the
new pipeline (the L-001 and L-002 entries from checkin 01's
wet run) don't have a `verdict_counts` field at all. They
have `verdict`, `attempt`, `promoted_as`, `round_1_tally`,
`round_2_tally`. The schema diverged in Phase 2 (PLAN.md
**Out** section: "Bench-history schema adjusted (sparser
rows, no per-call token granularity from subagents)") but
`learnings-report.ts` was never updated to read the new
shape. This is a pre-existing Phase 2 regression surfaced by
this checkin, not something this checkin caused. Surfaced as
a deferred fork: leaving `learnings-report.ts` as-is for now
(known runtime breakage when bench-history has records);
the substantive remediation — either rewrite the report to
read the new shape, or retire it alongside the SDK family —
deserves its own scoping. Not blocking Phase 3's stated
acceptance (lint + build clean).

The deletion itself was mechanical after the lift landed.
`rm -rf scripts/learnings-compact/` removed all 11 files;
the package.json edits removed the `learnings:compact`
script line and the `@anthropic-ai/sdk` dependency line;
`npm install` regenerated `package-lock.json` (reported "3
packages removed" — the SDK plus its two transitive
dependencies, presumably `agentkeepalive` and `humanize-ms`
or similar).

The SKILL.md edit was a four-line→three-line collapse in
the Do-not section: dropped the bullet about not calling
`npm run learnings:compact` (the script no longer exists),
preserved the bullet about not importing `@anthropic-ai/sdk`
as a regression guard. The new wording on the preserved
bullet was lightly updated to note "The dependency was
removed in Phase 3" — future Claude reading the skill knows
why the directive exists.

The README and pr-template edits were pure renames —
`s|/learnings-compact|/griot-compact|` in three README
locations (line 32 table cell, line 68 fresh-clone
instruction, line 183 placeholder-warning) and one
nightly.md location (line 43 footer). The README has other
stale `/learnings-*` slash command references that map to
the renamed `griot-*` family or to functionality that moved
into the substrate — those are deliberately left alone in
this checkin. They're a follow-up rename pass, not a
deletion concern.

## Notes for PR

- **What "closing the loop" means here.** The /griot-compact
  pipeline is now the only path from session-notes →
  rollup.md. The SDK orchestrator (`scripts/learnings-
  compact/`) is gone; the `@anthropic-ai/sdk` dep is gone;
  the API-key prereq prose is gone. Phase 3 is complete in
  the sense the PLAN intended: one path, one set of types,
  zero orphaned vestigial state in the deletion target.

- **Why the config-lift was necessary and the right shape.**
  The PLAN didn't anticipate that `learnings-report.ts`
  reaches into `learnings-compact/` for its config loader.
  The minimum-surface lift (single file, only the
  `loadConfig` / `resolvePath` helpers, only the
  `paths.*` interface fields the report consumes) preserves
  `npm run learnings:report` without introducing new
  abstractions. The dropped validations (7-judge assertion)
  were SDK-era and already broken against the new
  config.yaml — so the lift is also a small de-facto fix.

- **Pre-existing breakage `learnings:report` against the
  new bench-history shape.** This checkin surfaced it; this
  checkin does not fix it. The shape divergence was already
  on disk after Phase 2's intentional schema change; we just
  happened to put the first records on disk during checkin
  01's wet run, which is what tripped the runtime failure
  when we ran the consumer. Two reasonable remediation
  paths exist: (1) rewrite the report to read the new
  shape and surface tier-based panel metrics instead of
  judge calibration, or (2) retire the report alongside the
  SDK family since the metrics it surfaces (per-judge token
  counts, per-judge calibration) were explicitly cut by
  Phase 2 and don't have direct replacements in the new
  pipeline. Worth scoping as a separate small project or
  PR.

- **Why the README's other stale slash-command references
  weren't touched.** The README references `/learnings-
  capture`, `/learnings-use`, `/learnings-compact`,
  `/learnings-report` — the whole family was renamed
  `griot-*` during Phase 1 D1. Three of those are pure
  renames (`-capture`, `-compact`, `-report` map cleanly).
  But `/learnings-use` does *not* have a direct user-
  invocable equivalent — its job (loading rollup, installing
  citation contract) moved into `/ev-run`'s autoload step.
  Updating the README to reflect that correctly is bigger
  than a rename pass; it needs to restate what each command
  does and where the responsibilities now live. Scope-fork
  this to a follow-up doc pass.

- **The L-002 "halt-and-fork on scope expansion" learning
  was promoted from the wet run in checkin 01, and the
  cleanup work in this checkin gave it two real applications
  in the same session.** Both forks (config-lift, report
  runtime breakage) were surfaced as multi-option questions
  with explicit options, not silently absorbed. The
  validation-and-application loop is working as intended.
