# Checkin 04 — loom-cli/phase-3-lifecycle

**Created**: 2026-05-15 10:42
**Phase**: 3 — Lifecycle write API + lifecycle skills
**Unit**: Lifecycle skills (loom-session + loom-archive — closes Phase 3)

## Contract

- **Goal**: Author two `loom-*` skills that compose loom's CLI for
  the narrative/orchestration-shaped lifecycle work. With these
  shipped, Phase 3's deliverable is complete: loom owns the CLI
  layer (lifecycle bookends + state-update verbs) AND the skill
  layer (the LLM-shaped wrappers that consume it).

- **Acceptance criteria**:
  1. `.claude/skills/loom-session/SKILL.md` exists with YAML
     frontmatter (`name: loom-session`, `description`,
     `argument-hint`, `allowed-tools` set to the minimal surface
     the skill needs). Body documents the compose-then-write flow
     for a session handoff: read recent events, scan corrections,
     compose the structured Session JSON, write via
     `bin/loom session write`. Pairs with `bin/loom session write`
     the way trout-save-session pairs with autosave.
  2. `.claude/skills/loom-archive/SKILL.md` exists with the same
     frontmatter shape. Body documents the four-step flow:
     interview the user for color, synthesize a project Retro JSON
     from corpus + interview, write via
     `bin/loom retro write --type=project`, then
     `bin/loom project archive`. Skill stops at archive — composing
     the archive PR is out of scope for loom.
  3. Both skills compose `bin/loom` invocations at the seams,
     never `node .claude/cli/loom.ts` directly. The shim is the
     entry point of record.
  4. Both skills emit `correction:` lines in their guidance for
     when the user redirects mid-flight (matching the existing
     trout-* convention).
  5. Smoke verification: walk through each skill's process manually
     against a temp project. Verify the underlying `bin/loom`
     verbs produce the expected outputs at each step. Skills can't
     be invoked in this session per L-004 (newly-authored skills
     aren't in the session-cached registry), but the underlying
     CLI surface is exercised.
  6. `npm run test` passes (no new tests; existing 463 unchanged
     since skills are prose-only).
  7. biome lint clean.

- **Rules applied**:
  - Skills are markdown prose, no code.
  - Compose loom verbs via `bin/loom` shim.
  - Match trout-save-session / trout-archive structural style.
  - `correction:` convention preserved.
  - No emojis.

- **Disqualifiers**:
  - Authors PR composition skill (separate concern — Phase 4 or
    standalone).
  - Adds TypeScript code (skills are pure prose).
  - Skills that don't compose `bin/loom` (e.g. directly editing
    files).
  - `allowed-tools` granted beyond what each skill actually needs.

- **Inputs**:
  - `projects/LOOM-CONVENTIONS.md` — verb surface, file shapes
  - `.claude/skills/trout-save-session/SKILL.md` — structural reference
  - `.claude/skills/trout-archive/SKILL.md` — structural reference

no code, no tests beyond format validation. Single commit (TDD
ordering doesn't apply since there's no executable contract to
fail).

loom-session structures: frontmatter → Arguments → Process (4
steps: read state, compose JSON, write, report) → Rules → Failure
modes. Composes `bin/loom events read`, `session corrections`,

prose blobs.

`project archive`. Explicitly stops at archive — PR composition
is downstream.

(visible in available-skills after the Write calls completed).

just prose composing it.
