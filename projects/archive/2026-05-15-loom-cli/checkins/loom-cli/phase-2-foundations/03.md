# Checkin 03 — loom-cli/phase-2-foundations

**Created**: 2026-05-15 09:58
**Phase**: 2 — Foundations + read API
**Unit**: Read verbs (project + phase + events)

## Contract

- **Goal**: Wire the storage lib from unit 02 into the CLI dispatch
  from unit 01 by implementing seven read verbs across three
  namespaces: `project read/list/status`, `phase read/list`,
  `events read/latest`. Each verb takes a slug (or no slug for
  `project status` / `project list`), parses its flags, calls the
  appropriate lib function, formats the result as JSON to stdout
  (or pretty-printed with `--pretty`), and forwards `LoomError`
  thrown by lib functions to stderr with non-zero exit.

- **Acceptance criteria**:
  1. `.claude/cli/verbs/project.ts` exports handlers `projectRead`,
     `projectList`, `projectStatus` plus a `PROJECT_VERBS` map that
     dispatch reads from. `list` has `ls` as an alias.
  2. `.claude/cli/verbs/phase.ts` exports `phaseRead`, `phaseList`,
     and `PHASE_VERBS`. `phase read <slug> <N>` parses the
     two-positional form.
  3. `.claude/cli/verbs/events.ts` exports `eventsRead`,
     `eventsLatest`, and `EVENTS_VERBS`. `eventsRead` accepts
     `--since=<iso>`, `--event=<name>`, `--limit=<N>` flags
     mirroring the lib function. `eventsLatest` accepts
     `--event=<name>` filter.
  4. Read verbs emit JSON to stdout by default. `--pretty` flag
     pretty-prints (using `JSON.stringify(..., null, 2)`) for
     human view. Write verbs are not in scope here — no JSON
     result-wrapping yet.
  5. Verb handlers catch `LoomError` and return a `DispatchResult`
     with stderr set to `JSON.stringify(loomErr.toPayload())` and
     `exitCode: 1`. Non-LoomError exceptions bubble up.
  6. A `CliContext` type carries the `projectsRoot` path. `main()`
     in `loom.ts` derives this from `process.env.LOOM_PROJECTS_ROOT
     ?? join(process.cwd(), 'projects')`. Tests inject a fixture
     `projectsRoot` directly.
  7. `dispatch()` in `loom.ts` is updated to look up the namespace
     in a `VERBS_BY_NAMESPACE` map; unknown verbs within a known
     namespace return a structured `unknown-verb` error with
     `candidates` set to the namespace's verb names. The
     `not-implemented` placeholder is removed for the three
     wired-up namespaces but remains for the five not yet wired
     (checkin, session, pr, retro, doctor) until unit 04.
  8. Each verb file has a sibling `.test.ts` exercising it with
     `mkdtempSync`-based fixture project trees. Tests cover happy
     paths plus at least one error case per verb (project-not-found,
     phase-not-found, missing-args, etc.).
  9. `loom.test.ts` is updated to cover the new dispatch behavior
     (verb routing, unknown-verb-within-namespace error).
  10. `npm run test` passes; biome lint clean. Manual smoke:
      running `./bin/loom project list` and `./bin/loom project
      read <slug>` against the actual current project's manifest
      produces valid JSON.

- **Rules applied**:
  - TDD: tests-first commit, then impls commit (same pattern as
    unit 02).
  - Stdlib only.
  - Erasable TS.
  - Vitest, sibling `.test.ts`.
  - No emojis.

- **Disqualifiers**:
  - Implements write verbs (`scaffold`, `update`, `write`, etc. —
    Phase 3).
  - Implements checkin/session/retro/pr/doctor verbs (unit 04).
  - Adds new lib modules (unit 02 owns storage; unit 03 only adds
    verb handlers).
  - Hardcodes a projects root path that ignores `LOOM_PROJECTS_ROOT`
    or `process.cwd()`.
  - Uses non-erasable TS.

- **Inputs**:
  - `projects/LOOM-CONVENTIONS.md` — verb signatures and output
    conventions
  - `.claude/cli/lib/project.ts`, `manifest.ts`, `events.ts`,
    `config.ts`, `errors.ts` — storage lib from unit 02
  - `.claude/cli/loom.ts` — existing dispatch surface from unit 01

## Execution

Wired three namespaces (project, phase, events) into the CLI dispatch
across two TDD-ordered commits:

- **TDD step 1** (commit `07bb9a8`): 3 verb test files, true red
  state (value-import failure).
- **TDD step 2** (commit `393061f`): 3 verb impl files + dispatch
  routing in loom.ts + a small adjacent fix to lib/project.ts
  (filter listProjects by manifest.json marker).

Verb handlers all follow the same shape: parseArgs for flags, call
into the lib, emit() to JSON-or-pretty, errToResult() to catch
LoomError. `CliContext` lives in verbs/project.ts as the shared
CLI-runtime type hub; verbs/phase and verbs/events type-import it
from there. loom.ts re-exports CliContext + DispatchResult so they
have a sensible public surface.

`projectStatus` uses a `cwdOverride` option in CliContext for
testability — production code reads `process.cwd()` from main(),
tests inject a fake cwd.

Mid-flight scope expansion (documented as a `correction:` in Notes):
smoke test against the real `projects/` revealed listProjects mixing
trout-substrate projects with loom ones (both match the slug regex).
Fixed lib/project.ts with a 3-line `manifest.json`-existence filter
per LOOM-CONVENTIONS.md's coexistence policy. Updated lib tests to
write manifest.json markers in setup; added a coverage test for the
filter (trout-only dir excluded from listing). Per L-002, this is a
true scope expansion — adjacent work, not the contract's literal
"no new lib modules." Took the do-here branch (vs defer) because
the bug was a real correctness issue and the fix was tiny.

Evaluator's observation worth noting (not a flag): the
`not-implemented` placeholder branch in dispatch is load-bearing
for unit 04 — checkin/session/retro/doctor namespaces will route
through it until they're wired. That's by design.
