## Contract

- **Goal**: Wire the two PR-feedback verbs. `pr comments` fetches
  comments from a PR via gh and returns them structured. `pr respond`
  takes a responses JSON file and writes one response file per
  entry under `checkins/<branch>/responses/`. The two verbs are
  deliberately uncoupled — comments goes through gh, respond is
  pure filesystem.

- **Acceptance criteria**:
  1. `prComments(rest, ctx)`: Args `<slug> --pr=<N>`. Runs `gh pr
     view <N> --json comments,headRefName` and returns
     `{pr, branch, comments: [...]}` JSON. The branch field is the
     PR's head ref so callers know where to write response files.
  2. Comments structure from gh: each entry has `{id, author,
     body, createdAt}` at minimum. The verb passes them through
     verbatim (no filtering, no transformation — classification
     belongs to the consuming skill).
  3. `prRespond(rest, ctx)`: Args `<slug> --responses-file=<path>`.
     Reads the file. Expects `{pr, branch, responses:
     [{comment_id, body}, ...]}` shape. For each response, writes
     `checkins/<branch>/responses/response-NN.json` with `{comment_id,
     body, created}` (NN is the next unused number per branch).
     Returns `{paths: [...]}` listing written files.
  4. `pr respond` does NOT post to GitHub. It only writes local
     files. The consuming skill (loom-pr-respond, unit 04) or the
     user is responsible for posting via `gh pr comment` if desired.
  5. `prRespond` refuses to overwrite (immutability per existing
     substrate convention): if a response with the same number
     already exists, advances to the next number rather than
     overwriting.
  6. PR_VERBS map grows `comments` and `respond` keys.
  7. Sibling tests in `verbs/pr.test.ts` cover happy paths +
     missing-args + invalid responses-file shape.
  8. TDD: tests-first commit, then impls.
  9. `npm run test` passes; biome lint clean.

- **Rules applied**:
  - TDD.
  - Stdlib only.
  - Erasable TS.
  - Compose `ghRunner` for gh calls; pr respond is gh-free.
  - LoomError shape.
  - No emojis.

- **Disqualifiers**:
  - Implements skills (unit 04).
  - `pr respond` posts to GitHub (it's filesystem-only by design).
  - Classifies comments inside the CLI verb (classification is
    LLM-shaped, lives in loom-pr-respond).
  - Hardcodes gh path.

prComments is the gh-side fetch — wraps `gh pr view <N> --json
comments,headRefName`, returns `{pr, branch, comments}`. Comments
are passed through verbatim — classification belongs to the
consuming skill.

prRespond is the filesystem-side writer — reads a responses JSON
file with shape `{pr, branch, responses: [{comment_id, body}]}`,
writes one `checkins/<branch>/responses/response-NN.json` per
entry. `nextResponseNumber` scans existing files via a regex to
find the next free number; the loop advances per write so
multi-response invocations don't collide with themselves.

Departure from PLAN.md note: the `--pr=` flag is elided from
`pr respond` since the responses-file already embeds the PR number.
Single source of truth per invocation.
