# Checkin 01 — loom-cli/phase-4-pr

**Created**: 2026-05-15 10:48
**Phase**: 4 — PR write API + PR skills
**Unit**: pr discover + marker state machine

## Contract

- **Goal**: Implement the PR state machine — `pr discover` reads
  the on-disk checkin set for a branch + the marker comment from
  the live PR body (via gh), and returns the comparison verdict.
  This is the foundation of the PR workflow: `loom-pr` (unit 04)
  will read it to decide whether to author fresh, refresh stale,
  or refuse on drift.

- **Acceptance criteria**:
  1. New lib `cli/lib/pr-marker.ts` exports
     `parseCheckinMarker(body: string): number[] | null` and
     `computeMarkerState(disk: number[], marker: number[] | null):
     MarkerState` where MarkerState is `'fresh' | 'stale' | 'drift' | 'new'`.
  2. Marker format: `<!-- loom-pr-checkins: NN[,NN,...] -->`
     (kebab + plural; loom-prefixed to coexist with trout's
     `<!-- project-pr-checkins: -->` marker).
  3. Marker-state rules (per LOOM-CONVENTIONS.md):
     - `marker == disk` → `fresh`
     - `marker ⊂ disk` (proper subset, including null/empty) →
       `stale`
     - `marker ⊃ disk` OR sets diverge in both directions → `drift`
     - No PR yet (marker `null`, disk non-empty) → `new`
  4. New lib `cli/lib/gh.ts` exports `runGh(args: string[]):
     string` (default impl wraps `execFileSync('gh', args)`)
     and a typed `GhRunner` for injection. CliContext gains an
     optional `ghRunner?: GhRunner` field so tests can stub gh
     without an actual GitHub round-trip.
  5. New verb file `cli/verbs/pr.ts` with `prDiscover(rest, ctx)`
     handler. Args: `<slug> --branch=<b>`. Returns
     `{checkins: number[], marker_state: MarkerState, pr: {number,
     url} | null}` as JSON.
  6. Verbs map: PR_VERBS exports the new handler.
  7. `loom.ts` dispatch's VERBS_BY_NAMESPACE adds `pr` entry.
     With this unit's merge, the `pr` namespace is no longer
     unwired (the `not-implemented` placeholder retires).
  8. Sibling tests: `lib/pr-marker.test.ts` exercises all four
     marker-state transitions + format edge cases. `verbs/pr.test.ts`
     uses a stubbed `ghRunner` to exercise the three scenarios:
     no PR (new), marker matches disk (fresh), marker is stale.
  9. TDD discipline: tests-first commit, then impls.
  10. `npm run test` passes; biome lint clean.

- **Rules applied**:
  - TDD.
  - Stdlib only (`node:child_process` for execFileSync).
  - Erasable TS.
  - Vitest sibling tests.
  - LoomError shape.
  - No emojis.

- **Disqualifiers**:
  - Implements pr open/update/comments/respond (unit 02/03's job).
  - Implements loom-pr or loom-pr-respond skills (unit 04).
  - Hardcodes gh path or repo.
  - Allows the disk-marker comparison to depend on order
    (sets are unordered).

- **Inputs**:
  - `projects/LOOM-CONVENTIONS.md` — marker-state semantics
  - `.claude/cli/lib/checkin.ts` — listCheckins for disk side
  - `.claude/scripts/trout/pr-plumbing.ts` — reference impl
    (`MarkerScan`, `analyzeMarkerState`)
