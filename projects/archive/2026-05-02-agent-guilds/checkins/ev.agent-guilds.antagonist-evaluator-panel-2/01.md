## Contract

- **Goal**: Author `.claude/agents/evaluator-nextjs.md` as the second
  domain evaluator in the `evaluator-*` family, backed by a static
  CLI signal that does real semantic analysis (not just regex). The
  signal is a new `./scripts/check-nextjs.ts` that uses TypeScript's
  compiler API to detect `'use client'` directive correctness
  (files using client features without the directive, and files
  with the directive but no client features), plus a small set of
  regex-based framework antipatterns (`<img>` vs `<Image>`, raw
  `<a href>` for internal links, App-Router-incompatible Pages-API
  hooks). Blocking by default. Bundle-analysis checks deferred to
  a future deliverable per user decision. No new devDeps —
  `typescript` is already in devDeps and provides the AST.

- **Acceptance criteria**:
  1. **AST analyzer script**: `./scripts/check-nextjs.ts` exists.
     Uses `typescript`'s compiler API (`ts.createSourceFile`,
     `ts.forEachChild`) to walk `.tsx` and `.jsx` files under
     `app/`, `components/`, and `sketches/`. Detects:
     - `'use client'` directive presence (top-of-file string
       literal).
     - React hook usage (`useState`, `useEffect`, `useMemo`,
       `useCallback`, `useRef`, `useContext`, `useReducer`,
       `useLayoutEffect`, `useId`, `useTransition`, `useDeferredValue`,
       `useImperativeHandle`, `useSyncExternalStore`, plus any
       identifier matching `/^use[A-Z]/` to catch user-defined hooks).
     - JSX event-handler props (`onClick`, `onChange`, `onSubmit`,
       `onKeyDown`, `onKeyUp`, `onMouseDown`, etc. — any prop
       matching `/^on[A-Z]/` on a JSX element).
     - Browser-only API references (`window.`, `document.`,
       `localStorage`, `sessionStorage`, `navigator`).
     - Cross-checks: emit a finding when a file uses any of the
       above without `'use client'`, and a separate finding when a
       file has `'use client'` but uses none of the above (vacuous
       directive). Tolerate p5.js sketches as a documented
       exception — sketches are dynamic-rendered via `<Sketch>`
       and need `'use client'` even if they don't reference
       browser APIs directly (the p5 hook runs them in the
       browser).
  2. **Regex-based framework checks** (in the same script, simpler
     path): detect `<img ` JSX tags in `app/` or `components/`
     (Next has `<Image>`); detect `<a href="/` JSX patterns in
     `app/` or `components/` (Next has `<Link>`); detect
     `getServerSideProps` / `getStaticProps` named declarations
     in `app/` files (Pages Router API, deprecated for App
     Router); detect raw `<head>` JSX in `app/` files (metadata
     API antipattern). Each emits a finding with file:line.
  3. **Script output shape**: when invoked without arguments,
     walks the default scope. Exits 0 with no output if no
     findings. Exits 1 with one finding per line on stderr if any
     findings, format
     `<file>:<line>: <code>: <one-line description>`. Optional
     `--json` flag emits findings as a JSON array on stdout
     (forward-compat for evaluator consumption; the agent can
     grep stderr just as easily).
  4. **Script tests**: `./scripts/check-nextjs.test.ts` exists,
     uses vitest, covers the headline detection branches:
     (a) file with hook + no `'use client'` → flagged;
     (b) file with `'use client'` + no client features →
     flagged (vacuous); (c) p5 sketch with `'use client'` +
     no direct browser API → NOT flagged (sketch exception);
     (d) clean file → 0 findings; (e) `<img>` in component →
     flagged; (f) `getServerSideProps` in app → flagged. Tests
     use temp fixture files (write/read/delete pattern, similar
     to `pr-plumbing.test.ts` if it does that, or in-memory via
     `ts.createSourceFile`). At least 6 test cases.
  5. **npm scripts**: `package.json` gains
     `"lint:nextjs": "node ./scripts/check-nextjs.ts"`. Uses
     `node` directly (Node 24 strips type annotations natively
     per **L-001**) rather than `tsx`. Source must stay in
     erasable TypeScript — no enums, no `namespace`, no parameter
     properties, no `const enum`. Existing project scripts use
     `tsx` (`generate-tokens.ts`, `learnings-report.ts`); this
     script deviates as the forward-looking pattern. The script
     also gets picked up by `npm test` (vitest defaults pick up
     all `*.test.ts` files, but verify by running `npm test` and
     confirming the new tests appear).
  6. **Permissions**: `.claude/settings.json` gains
     `Bash(npm run lint:nextjs *)`. No other permission changes.
  7. **The evaluator agent**: `.claude/agents/evaluator-nextjs.md`
     exists with frontmatter matching the existing `evaluator-*`

     `description:` (one-line), `tools:` (Read, Glob, Grep, plus

  8. **Agent body — inherited base contract**: opens with an

  9. **Agent body — Process section**: 3–6 numbered steps walking

## Execution

Two non-trivial decision points came up during execution.

**Implicit addition: `./scripts/package.json`.** L-001 says to use
`node ./scripts/check-nextjs.ts` rather than `tsx`. But `./scripts/`
inherits CommonJS from the root `package.json`, and the analyzer
uses ESM imports (`import ts from 'typescript'`). The cleanest fix
is a 3-line `./scripts/package.json` declaring `{"type": "module"}`
— the same pattern `.claude/scripts/` already uses. The existing
tsx-invoked project scripts (`generate-tokens.ts`,
`learnings-report.ts`, `learnings-config.ts`) continue to work
because tsx handles both module modes. Noted here rather than
revising the AC mid-execution because it's an implementation
detail of "use node-direct invocation per L-001."

**Halt-and-fork at script first run.** `npm run lint:nextjs`
against the existing codebase produced 2 findings:
`components/app/FileListing/index.tsx` and
`components/app/FormulaSketch/index.tsx` both use `useMemo` but
lack `'use client'`. Both files work today because Next's import
flow makes them transitive client components (their parents in
the route tree have `'use client'`). The strict analyzer rule
treats this as missing-directive; the pragmatic reading treats it
as fine. Per L-002, the four-way fork was surfaced to the user
(fix in place / loosen analyzer / known-allowed list / defer with
`--allow-existing` flag). User picked **fix in place** — added
`'use client';` to both files. Behavior unchanged; files now
self-documenting.

## Notes for the PR

- **AST analyzer over jsx-a11y-style flat-config**. The PLAN's
  parenthetical called for ESLint-style tooling; D1 already
  rolled back ESLint after user pushback, and D2 follows the
  precedent — TypeScript's compiler API is already in devDeps,
  and a single ~210-line analyzer covers the same headline
  patterns with less surface area. No new devDeps in D2.

- **correction: Implicit `./scripts/package.json` addition.**
  Adding the 3-line ESM declaration to `./scripts/` was not in
  the AC but was necessary for L-001 compliance (node-direct
  invocation of `.ts` files in a CJS-default subtree). Existing
  tsx-invoked scripts in the same directory continue to work
  unchanged.

- **correction: User redirected the 2 existing-file findings to
  fix-in-place.** The analyzer flagged
  `components/app/FileListing/index.tsx` and
  `components/app/FormulaSketch/index.tsx` for using hooks
  without `'use client'`. Both files work today as transitive
  client components. Per L-002, four options were surfaced
  (fix-in-place / loosen analyzer / known-allowed list /
  `--allow-existing` flag); user picked fix-in-place. Both files
  now have explicit `'use client';` directives — same runtime
  behavior, better self-documentation.

- **Sketch exception is intentionally generous.** Three
  independent signals trigger it: importing `@p5-wrapper/react`,
  rendering `<Sketch>`, or being under `sketches/`. This
  prevents the vacuous-`'use client'` check from firing on ~55
  sketches that legitimately need the directive without
  referencing browser globals directly. Each signal alone would
  miss some legitimate sketch files; together they're robust.

- **Bundle-analysis was explicitly deferred** per user decision
  at contract negotiation. The PLAN names it parenthetically; a
  future deliverable can add it if the catalog growth justifies
  the complexity. Current 15-entry catalog covers the
  high-impact static patterns without it.

- **Future entry 11** (client boundary placed too high) and
  **entry 14** (`<Image>` without `priority`) are advisory-only
  and marked **manual inspection**. They live in the catalog so
  reviewers know to consider them but don't trigger automated
  failures.
