## Contract

- **Goal**: Rework `.claude/skills/trout-pull-request/SKILL.md` (and the matching `projects/CONVENTIONS.md` § PR marker spec) so the skill (1) supports PRs that span more than one unit by enumerating all checkins on the branch and authoring the body across them, (2) leads PR descriptions with a "Motivation" section sourced from PLAN.md's Context or per-checkin Goal rationale rather than diving straight into requirements, and (3) when motivation content is thin in the source material, stops and asks the user for the why instead of papering over it. The replan PR (this branch) shipped because the original "always author from the latest checkin" rule narrowed a multi-commit PR to a single unit's framing — this rework closes that gap and also addresses the long-running pattern of PR descriptions skipping the architectural why.

- **Acceptance criteria**:
  - **Marker format change.** New marker is `<!-- project-pr-checkins: NN[,NN,...] -->` (plural; comma-separated list of checkin numbers). The old singular `<!-- project-pr-checkin: NN -->` continues to parse as the set `{NN}` for backward-compat. Both forms are accepted on read; only the plural form is written.
  - **Multi-checkin enumeration.** Process § 1 changes from "find the highest-numbered file" to "enumerate every numbered checkin file in `checkins/<branch>/`, parse each." Resulting set `D` is the disk truth.
  - **Staleness as set comparison.** Let `M` = checkin numbers in marker, `D` = disk truth. Outcomes: `M == D` → fresh (no-op); `M ⊂ D` (proper subset) → stale (rewrite, marker now lists all of `D`); `M ⊃ D` (marker references checkins missing on disk) → drift (refuse, report). Old single-checkin behavior is the special case `|D| = 1`.
  - **Motivation gate.** A new process step (call it § 3a "Why check") runs before authoring. Read PLAN.md's `## Context` section (project-level) and any `## Context` paragraph attached to the relevant phase (the phase the checkin set belongs to); also read each checkin's Contract Goal. Heuristic for "thin": no project-level Context section OR Context section is shorter than ~80 characters AND no checkin Goal contains rationale words (`because`, `so that`, `motivated by`, `to ensure`, `to avoid`, etc.). If thin → stop with a structured prompt asking the user for the motivation; the prompt offers to also append the answer back to PLAN.md's Context for the relevant phase so future runs don't re-ask. Skill resumes when re-invoked with the motivation supplied.
  - **Body template, single-checkin.** Adds a `## Motivation` section before Summary. Otherwise current shape (Summary → Contract with Goal + Acceptance criteria → Verification → Notes → footer line). Marker is the plural form with one entry: `<!-- project-pr-checkins: NN -->`.
  - **Body template, multi-checkin.** Adds a `## Motivation` section before Summary. `## Summary` distills 2-5 bullets across all checkin Goal+Execution sections. The Contract section becomes a `## Units` table for 3+ checkins (columns: `#`, `Unit`, `Goal`) or a per-checkin list with Goal + Acceptance criteria for 2 checkins. `## Verification` merges the rules-applied commands across checkins, deduplicated. `## Notes` merges Notes-for-the-PR sections from each checkin with `_(checkin NN)_` attribution. Footer lists all checkin numbers: `Tracked by project substrate: <path> — checkins NN, NN, NN`.
  - **Title authoring.** Single-checkin retains `[Phase N] <unit name>`. Multi-checkin: if all checkins are within the same phase, use `[Phase N] <phase name>` from PLAN.md; if checkins span multiple phases or the phase has no name, stop and ask the user for a title. Always under 70 characters.
  - **Invariants section** updated. Invariant 2 changes from "always authored from the current latest checkin" to "always authored from the current set of checkins on the branch." Invariant 3 changes from "marker equals latest checkin NN, no-op" to "marker set equals disk set, no-op." A new invariant added: "when motivation source material is thin, refuse to author and ask the user; do not synthesize a motivation."
  - **Failure modes section** gains entries for: `motivation thin → stop and prompt`; `multi-checkin spans multiple phases without explicit phase context → stop and ask for title`. Existing entries preserved.
  - **Report section** gains a new terminal state: `pr: paused, awaiting motivation input from user`. Existing four states preserved.
  - **`projects/CONVENTIONS.md` § PR marker** updated to document the plural marker, the backward-compat parser rule, and the staleness-as-set-comparison semantics. Also document the why-focus expectation (PRs authored by this skill have a Motivation section; thin source material triggers an interactive ask).
  - **No regressions for the single-checkin path.** A single-checkin PR authored by the new skill should produce a body very close to the existing one for that PR (modulo the new Motivation section and the marker form). The skill's typical use case stays uncomplicated.
  - **No emojis, terse prose, format-spec flavored** — consistent with the rest of the skill body and CONVENTIONS.md.
  - `npm run lint` clean.
  - `git status` after the unit shows only `M .claude/skills/trout-pull-request/SKILL.md`, `M projects/CONVENTIONS.md`, `A projects/2026-05-02-agent-guilds/checkins/ev.agent-guilds.replan-phase-1-5/02.md` (plus the carved-out `.claude/settings.local.json` drift).

- **Disqualifiers**:
  - **Idempotence regression.** Re-running the skill on a PR with marker matching disk triggers a rewrite. The set-comparison logic must produce a no-op for `M == D`.
  - **Backward-compat break.** Old singular marker form (`project-pr-checkin: NN`) causes a drift report or refusal instead of being parsed as `{NN}`.
  - **Why gate fires too eagerly.** A PLAN.md with a substantive `## Context` section and a checkin Goal containing rationale should NOT trigger the motivation prompt. The thin heuristic must accept "good enough" why content.
  - **Why gate fires too quietly.** A PLAN.md with no Context AND a checkin Goal that's a pure "what" statement (no rationale words) MUST trigger the prompt. Skipping it silently and authoring a thin Motivation section is a worse outcome than asking.
  - **Acceptance criteria dropped from single-checkin body.** The criteria list is reviewer-load-bearing for single-checkin PRs; it stays in the body for that case. (For multi-checkin PRs, the table form drops criteria — they live in checkin files.)
  - **CONVENTIONS.md and SKILL.md disagree on the marker format or the staleness semantics.** Three-way agreement: SKILL.md, CONVENTIONS.md, and the actual marker as written.
  - **The skill loses the `/trout-autosave` invocation step** for `pr-opened` / `pr-updated` events. (Easy regression to introduce while restructuring; preserve the autosave call.)
  - **Title rule mishandles "all checkins in same phase" detection.** If checkins span phases the skill must ask, not silently pick the first phase or synthesize a name.
  - **Skill body grows past readability.** If the rewrite balloons the SKILL.md file substantially (say, >2× current length), it's a sign the rework is doing too much and should be split or simplified.
