**Unit**: Tighten `/trout-pull-request`: concise PR description template

## Contract

**Goal**: Update `.claude/skills/trout-pull-request/SKILL.md` so PR descriptions authored by the skill are concise and reviewer-oriented rather than exhaustive duplications of the checkin file. PR #10 (the immediately prior author from this same skill) shipped with a ~3000-word body that reproduced the full contract verbatim — practically nobody reads that. This unit caps every body section, drops the "acceptance criteria verbatim in single-checkin body" rule, renames the contract section to "Reference" with a path link to the checkin, simplifies the multi-checkin Units to a single table form with a checkin-link column, and adds an invariant that makes conciseness load-bearing.

**Acceptance criteria**:

- **§ 4.2 Body — common shell** caps every section explicitly: Motivation 2-4 sentences, Summary 3-5 one-line bullets, Verification one line per command, Notes 3-5 reviewer-relevant items. Each cap stated in the template comments and reinforced as a hard rule below the template. Total body cap: 500-600 words. Past that, the unit is doing too much — split into multi-checkin or compress harder.
- **§ 4.3 Single-checkin section renamed** Contract → Reference. Body shows: `**Goal**` as a 1-2 sentence distillation (paraphrased if Goal is long); `**Checkin**` as a markdown link to the checkin file path with a short caption naming what's there. Acceptance criteria are NOT pasted into the PR body.
- **§ 4.4 Multi-checkin section simplified** to one table form with columns `# | Unit | Goal | Checkin`. Drop the "for 2 checkins use a per-checkin list with full criteria" sub-form — same conciseness rule applies regardless of count. Reviewers click the checkin-link column for depth.
- **Invariant 7 added**: "Concise over exhaustive" makes the section caps and the 500-600 word total cap contractually binding. A PR body that wants to be longer than 600 words signals the unit is doing too much or the author is over-explaining. Acceptance criteria, scope, execution, evaluator verdict are explicitly NOT pasted into the body — they live in the checkin file.
- **PR #10 re-authored** with the new template via the skill's stale-rewrite path (M={01} ⊂ D={01, 02} after this checkin lands). The new body comes out under ~500 words for the multi-checkin set; the prior 3000-word body is fully replaced. Self-documenting smoke test of the convention.
- **No regressions to the why-check, marker semantics, or staleness logic.** Sections § 1, § 2, § 3, § 5, § 6 untouched. The conciseness change is scoped to the body-authoring section (§ 4) and the new invariant.
- **No emojis, terse prose, format-spec flavored** — consistent with the rest of the skill.
- `npm run lint` clean, `npm run build` clean, `npm run test` clean.

**Rules applied**:
- Project conventions per `~/.claude/CLAUDE.md` and `aart.camp/.claude/CLAUDE.md`: terse, no speculative abstractions.
- The user's PR-conventions: under 500 additions for most PRs; description leads with motivation; reviewer-oriented summaries over exhaustive specs; "if a PR feels too big to hold in your head during review, it should be split."
- Pre-evaluation `git status` (lesson from replan branch checkin 01).

**Disqualifiers**:
- **Conciseness rule lost in the body but absent from invariants**, or vice versa — the rule must be load-bearing in both places so future skill edits can't quietly drop it.
- **Acceptance criteria still pasted into single-checkin body** — that's the biggest source of length in PR #10 and the change must be visible in the template.
- **Multi-checkin loses the per-checkin path link** — the link is the whole point of the conciseness trade. Reviewers need a one-click path to the checkin file for depth.
- **Why-check or staleness logic accidentally edited** while restructuring § 4.
- **Skill body grows past readability** — if the rewrite balloons the SKILL.md file substantially, it's a sign the rework is doing too much.
- **PR #10 not re-authored**. The whole point of self-documenting is that the skill authors the very PR that delivered it under the new shape. If the rewrite doesn't happen, the convention isn't proven.
