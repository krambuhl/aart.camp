  decisions captured in
  `projects/2026-05-02-agent-guilds/whiteboards/4-domain-pairs.md`
  round 1 + the AskUserQuestion outcome):

  - **Specialist sequencing flipped**: PLAN.md Phase 4
    originally said "specialist runs solo first, panel only on

    not a control-flow change in ev-loop.

  - **Three stub pairs dropped**: PLAN.md originally said ship
    three stub pairs (storybook-stories, unit-tests,
    playwright-e2e) with activation criteria. The whiteboard

    unit-tests, or Playwright e2e in its stack anyway.

  1. **`generator-base.md`** at `.claude/agents/generator-base.md`
     — shared base contract for the `generator-*` family,
     parallel to `evaluator-base.md` and `whiteboard-base.md`.

       not a new I/O contract. Same shape as the human/main-
       thread generator already produces.
     - **Input contract**: the unit Contract block + any
       phase-level Whiteboard output cited as an Input.
     - **Output contract**: proposed file changes + Scope +
       Execution narrative. NO commits. NO PR authoring. The
       orchestrator (the loop) does writes-to-git.
     - **Frontmatter `activation` field**: every
       `generator-*` agent declares `activation: active` (the
       v1 default). The field exists for forward-compat —
       future stubs could declare `activation: stub` with an
       `activates-when:` list (per **react-architect's
       frontmatter-structured-criteria** framing) — but D1
       ships zero stubs per the user's adopted decision.
     - **Cross-perspective courtesy**: same lane-deferral
       rules as `evaluator-base` / `whiteboard-base`.

  2. **`generator-css-codemod.md`** at
     `.claude/agents/generator-css-codemod.md` — the first
     active `generator-*` agent. Frontmatter: `name`, `role:
     generator`, `description`, `tools` (as inherited per
     base, write-capable), `model: inherit`, `activation:
     active`. Body covers:
     - "Read `generator-base.md` and apply its constraints."
     - **Domain scope** (per **performance's strong push for
       narrow scope**): plan a CSS codemod on a SPECIFIC
       `.module.css` file or a small set, transforming
       literal values (hex, px, named colors, raw spacings)
       into `token(...)` calls. The generator's deliverable is
       a planned migration + an explained codemod artifact (a
       diff or a sequence of Edit operations). It does NOT
       run the token-regeneration pipeline (`npm run
       generate:tokens`) — that's a separate step assumed.
     - **Scope discipline**: per-file or small-set; explicit
       file list in the unit contract; the generator does not
       sweep entire directories autonomously.
     - **Output shape**: file modifications + a Scope section
       (list of files touched + line-level changes) + an
       Execution section (the migration narrative).
     - **Sketch carve-outs** (per **sketch-ideation's drive-
       by**): `components/app/Sketch/Sketch.module.css` has
       **load-bearing `!important` on canvas width/height**
       (overrides p5.js inline styles) and a **descendant
       `& canvas` selector** (p5 owns the canvas DOM node,
       can't carry a CSS-Module class). The generator MUST
       NOT normalize these away. Body documents this as a
       known exception. D2 adds an inline comment in
       Sketch.module.css for in-place reinforcement.
     - **Resolved-value diff requirement** (per **a11y's
       cheap-version recommendation**): for any codemod
       touching rendered visual properties (color,
       background, border, font-size, line-height, focus
       styles), the generator's output includes a
       resolved-value table (before-literal → after-token-
       resolved-value). This shifts the contrast-impact
       burden to the generator's deliverable, keeps the
       evaluator's job tractable.

  3. **`evaluator-css-architecture.md`** at
     `.claude/agents/evaluator-css-architecture.md` — the
