## Contract

- **Goal**: Author `.claude/agents/evaluator-react-api.md` as the third
  domain evaluator in the `evaluator-*` family, sibling to
  `evaluator-a11y` (D1, blocking, axe-core-backed) and
  `evaluator-nextjs` (D2, blocking, AST-backed). Per PLAN.md § Phase 2,
  D3 is intentionally lighter: **smaller advisory-only catalog**, **no
  dedicated CLI signal**, detection by grep heuristics + manual
  inspection only. Covers framework-agnostic React API antipatterns
  not already handled by `evaluator-nextjs` (which owns `'use client'`
  correctness, the App-Router boundary, and Next-specific concerns).
  Pure agent-file change — no scripts, no npm scripts, no devDeps,
  no permission changes.

- **Acceptance criteria**:
  1. **Agent file**: `.claude/agents/evaluator-react-api.md` exists.
  2. **Frontmatter** matches the existing `evaluator-*` shape:
     `name: evaluator-react-api`, `role: evaluator`,
     `description:` (one-line — must explicitly call out
     **advisory by default**, contrasting with the blocking
     defaults of `evaluator-a11y` and `evaluator-nextjs`),
     `tools:` (Read, Glob, Grep, plus read-only Bash for
     `npm run lint:*`, `npm run build:*`, `git status:*`,
     `git diff:*` — no `lint:react:*` since no script exists),
     `model: inherit`, `maxTurns: 5`.
  3. **Body — inherited base contract**: opens with an
     "Inherited base contract" section pointing at
     `evaluator-base.md`. Does NOT duplicate stance, verdict
     format, flag taxonomy, or read-only-constraint content.
  4. **Body — Process section**: 3–5 numbered steps walking the
     evaluator through (a) detecting React scope (`.tsx` / `.jsx`
     files with JSX or React imports — skip pure-data `.ts` files),
     (b) running grep heuristics for catalog entries that support
     them (e.g. `\.push\(|state\.\w+\s*=` for direct mutation,
     `key=\{` audits against `.map(`), (c) manual inspection for
     entries that require it, (d) assembling verdict with all
     findings advisory by default — escalate to blocking only with
     explicit evidence (e.g. demonstrated runtime breakage in
     diffs or commits).
  5. **Body — Antipattern catalog**: 6–10 entries. Each entry:
     pattern, symptom, impact, detection method (`grep` heuristic |
     `manual` inspection), severity (`advisory` for all entries in
     this initial pass), flag code (prefixed `react-`, e.g.
     `react-hooks-conditional-call`). Catalog must include at
     minimum:
     - Hooks called conditionally / inside loops / after early
       returns
     - `useEffect` with missing or stale dependency array
     - Direct state mutation (`state.push(...)`,
       `state.foo = bar`, etc.)
     - Missing `key` prop on rendered list items (or non-stable
       key like array index when items can reorder)
     - `useEffect` for derived state that should be `useMemo`,
       `useCallback`, or render-time compute
     - Ref accessed/written during render (read/write of
       `ref.current` outside an event handler, effect, or
       imperative-handle method)
     - Context `value=` passed an inline object/array literal
       (forces every consumer to re-render every parent render)
     - Anonymous function as prop to a `React.memo`-wrapped child
       (advisory perf smell — defeats memo on every render)
  6. **Body — Flag codes section**: table listing each react-api
     code mapped to its catalog entry. Codes are `react-`-prefixed
     and do NOT duplicate base flag taxonomy (`parse-failure:*`,
     etc., come from `evaluator-base.md`).
  7. **Body — Inspection signals section**: explicitly documents
     that no dedicated CLI signal exists for this evaluator (in
     contrast with `evaluator-a11y`'s `npm run test:a11y` and
     `evaluator-nextjs`'s `npm run lint:nextjs`). Names the
     rationale: advisory-only initial scope; ESLint /
     `eslint-plugin-react-hooks` deliberately not adopted (D1 and
     D2 both rejected ESLint precedent); composability goal — a
     consumer project that wants a CLI gate can add one without
     this agent depending on it.
  8. **Boundary note**: brief section clarifying overlap with
     `evaluator-nextjs`. `'use client'` correctness, the App-Router
     boundary, `<Image>` / `<Link>` / metadata API — all owned by
     `evaluator-nextjs`. `evaluator-react-api` covers
     framework-agnostic React patterns that would apply equally in
     a CRA / Vite / standalone-React project. When both evaluators
     are present in a panel, precedence and dedup are panel-level
     concerns (handled in D6).
  9. **Verification**: `npm run lint` (Biome) clean. `npm run
     build` clean. `npm test` passes (no new tests since no new
     script). `git status` shows only `.claude/agents/evaluator-
     react-api.md` and `projects/2026-05-02-agent-guilds/...` paths
     modified — no edits elsewhere.

## Notes for the PR

- **Advisory-only is the deliberate initial framing.** PLAN.md
  § Phase 2 explicitly scopes D3–D5 as "smaller advisory-only
  catalogs initially." Four of the eight entries include an
  "escalate to blocking if..." conditional pointing at concrete
  in-diff evidence; the conditional preserves rubric honesty
  while the default severity stays advisory. D6 (precedence +
  panel-level boundary work) is the right place to revisit
  severity escalations if real-world panel usage surfaces
  consistent false-negatives.

- **No CLI signal is the deliberate cheaper shape.** D1 and D2
  each shipped with a dedicated CLI signal (axe-core / Playwright
  for a11y; TypeScript AST analyzer for nextjs). D3 ships
  without one. Three reasons documented in the agent file's
  Inspection signals section: (1) advisory-only initial scope
  doesn't yet warrant the script-authoring cost, (2) ESLint /
  `eslint-plugin-react-hooks` already rejected as a precedent
  by D1 and D2, (3) composability — an evaluator depending only
  on `Grep`/`Read` ports cleanly to downstream consumers.

- **L-004 split honored**. This checkin authors the new agent
  file but does NOT spawn `evaluator-react-api` as a real
  subagent on the panel. The agent registry reloads at session
  boundary; same-session invocation would silently fail. A
  follow-up unit (likely D6 or a dedicated smoke-test deliverable)
  picks up runtime verification once the registry has cycled.

- **Boundary with `evaluator-nextjs` is documented but not
  enforced yet.** Both evaluators may legitimately inspect a
  React/JSX artifact. The new agent file's "Boundary with
  `evaluator-nextjs`" section divides ownership in prose, but
  panel-level dedup/precedence is a D6 concern. For now, if both
  evaluators are present in a panel and both flag the same
  pattern, the aggregator surfaces both findings unaltered —
  matches the current `guild-validate` v1 conflict-resolution
  no-op.

- **correction: Branch cut went through stash, not direct
  checkout.** The first `git checkout main` was rejected
  because the MANIFEST.md reconcile diff conflicted with main's
  state (main's MANIFEST ends at `pr-opened | #45`, the local
  diff added `pr-merged | #45` plus the row flip). Worked around

  checkout → pull → branch-cut → stash pop. The pop applied
