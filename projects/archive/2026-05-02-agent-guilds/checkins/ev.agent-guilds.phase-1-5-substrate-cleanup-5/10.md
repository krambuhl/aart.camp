## Contract

**Goal**: Author `.claude/scripts/trout/save-session-finalize.ts` (TypeScript, stdlib only) plus sibling `save-session-finalize.test.ts`. The script handles the deterministic finalize tail of the session-handoff ritual: pick the next available `<YYYY-MM-DD>-<letter>.md` filename under `<project>/sessions/`, write the handoff content there, and (when applicable) invoke `griot/capture.ts` for any `correction:` lines found in the session's checkins. The skill body's steps 1–6 (resolve project, read manifest, read checkins, draft narrative content) stay as LLM-shaped prose — those are the heart of the handoff. Step 7 (write file), step 5 (next-letter filename determination), and the correction-capture loop in step 4 collapse into a single `Bash("node .claude/scripts/trout/save-session-finalize.ts ...")` invocation. The autosave call (step 8) stays as a separate skill-level invocation, mirroring D7's precedent — autosave is its own primitive with its own permission, test surface, and event vocabulary; folding it in violates the family-of-verbs convention.

**Acceptance criteria**:

1. **Script: `.claude/scripts/trout/save-session-finalize.ts`.** ESM, stdlib-only. Argument surface (parseArgs):
   - First positional: project slug or path (resolved like `autosave.ts::resolveProject` — exact slug → suffix → full path → archive-aware error).
   - `--content-file=<path>`: required. Path to a temp file containing the LLM-authored handoff body (frontmatter through "Notes" section). The script reads the file verbatim and writes it to the chosen sessions/ filename. The skill is responsible for cleaning up the temp file after the script returns; the script does not delete its input.
   - `--date=<YYYY-MM-DD>`: optional. Defaults to today (UTC, mirroring `griot/capture.ts::timestamp` precedent). Test injection point.
   - Exit 0 on success: stdout line `session-saved: <relative-path-from-cwd>`. Stderr on failure: `save-session-finalize-error: <reason>` and exit 1.

2. **Filename determination.** Iterate `a` through `z`. The first letter for which `<project>/sessions/<date>-<letter>.md` does not exist wins. If `z` is taken, fail with `save-session-finalize-error: all letters a-z taken for <date>; manual intervention required` (matches the SKILL.md current failure-mode wording).

3. **Correction capture.** The script reads the project's `MANIFEST.md` events table, walks **backwards** from the most recent event to find the cutoff: the most recent `session-saved` event before this run (if any), or the start of the table. Within that window, every `checkin-created` event identifies a checkin file path: `<project>/checkins/<branch>/<NN>.md`, where `<branch>` and `<NN>` are parsed from the event's `detail` column (current convention: `<NN> on <branch>`). For each such checkin, the script reads the file, extracts its `## Notes for the PR` section, and finds every line matching `^[-*]?\s*correction:\s*(.+)` (multi-line corrections continue until the next `^[-*]\s+` bullet or blank line, mirroring `capture.ts::extractCorrections`). For each correction, the script invokes `Bash("node .claude/scripts/griot/capture.ts --from-checkin=<checkin-path> --slug=<derived> --correction-text=<text>")` via `spawnSync`. The slug is derived from the correction text (kebab-case, max 5 tokens, mirroring `capture.ts::kebabize`); collisions across multiple corrections in the same checkin get suffixed `-a`, `-b`, etc.

4. **Tests: `.claude/scripts/trout/save-session-finalize.test.ts`.** Vitest with per-file `// @vitest-environment node` directive. Cases (10–12 tests):
   - Happy path, no prior session for the date → writes `<date>-a.md`, no captures.
   - Happy path, `<date>-a.md` exists → writes `<date>-b.md`.
   - Happy path, `<date>-a.md` through `<date>-y.md` exist → writes `<date>-z.md`.
   - Error: all letters a-z taken for the date.
   - Error: project not found.
   - Error: `--content-file` missing.
   - Error: `--content-file` points to a non-existent path.
   - Correction capture: a checkin in the session window has one `correction:` line → captures it (assert via stdout/dirent count under `learnings/session-notes/`). Use the real `capture.ts` script via the actual fixture; no stub.
   - Correction capture: a checkin has multiple correction lines → captures each with distinct slugs.
   - No session-saved cutoff: every `checkin-created` since the start of the table is in scope.
   - Cutoff respected: a `checkin-created` event before the most recent `session-saved` event is NOT captured.
   - Slug suffix collision: two corrections in the same checkin that kebabize to the same slug get `-a`, `-b`.

- **Correction extraction reimplemented in the finalize script.** The script invokes `capture.ts` (which owns correction parsing); it does not re-implement `extractCorrections`. The finalize script's only correction-related logic is: locate session-window checkin paths, read their `## Notes for the PR` sections, find `correction:` lines (regex-only — same shape as capture.ts but call into capture.ts for the per-correction work), derive slugs, invoke capture per line.

- **Tests skip the cutoff-respected case (correction in a checkin before the previous session-saved event).** That test is what proves the script does NOT re-capture corrections from prior sessions.
- **Script body grows past ~220 lines** or test file past ~340 lines.
- **Skill body grows past ~120 lines** (the rewrite should be smaller than the current 140; if it grows, the deletion of the "Capturing corrections" detail section didn't fully land or new content slipped in).

## Notes for the PR

- **PLAN-vs-contract divergence on "appends session-saved event."** PLAN.md D8 wording reads as if the finalize script should append the `session-saved` event to MANIFEST. The contract (and implementation) deliberately does NOT — autosave stays a separate skill-level invocation per D7's family-of-verbs precedent (D7 disqualifier-list rationale: "Autosave is a separate primitive with its own permission, its own test surface, and its own event vocabulary; folding it in violates the family-of-verbs convention"). The PR description leads with this reconciliation. PLAN's wording is descriptive of the overall flow (skill + script + autosave together), not prescriptive of which script owns the append.
- **Cap-miss-and-recovery story.** Initial 277-line draft → first compression pass (`tryOrFail` helper, inlined assignments) → 231 → panel run 1 flagged → second compression pass (single-line `if/throw`s in `resolveProject`, inlined `path` in `sessionWindowCheckins`) → 211 → panel run 2 approved. Two takeaways for future contracts: (a) line-cap estimates for scripts that combine multiple substrate concerns (parser + walker + delegation loop) should err generous; (b) when an evaluator flags a soft-prefix "~N" cap as binary, the prefix's hedge is not load-bearing — soft-pedal is what the antagonist lens is built to resist. Both data points worth carrying into future migration contracts.
- **Local correction parsing in finalize.** The script reimplements `extractNotesForPR` and `extractCorrections` regex-only — same shape as `capture.ts` but called directly. The contract scoped this explicitly ("regex-only — same shape as capture.ts but call into capture.ts for the per-correction work"). This is the boundary case: finalize needs to know which correction lines exist *before* it can spawn capture per-slug. A cleaner future version would have `capture.ts` grow a `--list-only` mode that returns the corrections it would capture, and finalize would call it once. Out of D8 scope; tracked as a substrate-refinement candidate.
- **Test fixture copies real capture.ts via cpSync.** No stubs. `capture.ts`'s arg surface and folder-write behavior are exercised end-to-end in the suffix-collision and cutoff-respected tests. If `capture.ts`'s contract drifts, the finalize test catches it before the skill body's caller does.

- **Refinement backlog unchanged.** None of session 2026-05-08-b's five open-thread items addressed in D8 (autoload gh check, commit allowlist tighten, submit pr-number sanity check, autosave header propagation, submit `<N>` placeholder). Same deferral reasoning as D7.
