## Contract

**Goal**: Move the correction-capture primitive from a markdown skill to a Node script in the new `.claude/scripts/griot/` family. This is deliverable 4 of Phase 1.5 — the third migration in the series (autosave → autoload → capture). Per PLAN, the script handles `--from-checkin=<path>` and derives all five session-note files; the old skill directory is deleted. Two structural calls explicitly recorded:

(a) `learning.md` is generated as a thin draft (correction text + provenance footer) rather than the LLM-synthesized 2-paragraph lessons the prior skill produced — the skill spec already calls drafts "rewritable" and the compaction pipeline (`/griot-compact`) is the LLM-driven value gate. The script-generated drafts are seeds the judges can accept or have rewritten.

(b) Interactive mode (the user invoking `/griot-capture` in-session after a correction, with the LLM-driven gate + transcript-reading) is discontinued by this migration. The skill spec's interactive workflow required live transcript access and judgment — neither belongs in a script. From-checkin mode (called by `/trout-save-session` for each `correction:` line) is the only path post-migration. Workflow loss flagged for review.

**Disqualifiers**:

- **Script tries to do LLM work**: drafting `learning.md` as a substantive 2-paragraph lesson requires LLM judgment. The script must produce a thin template (correction text + provenance footer) and let `/griot-compact` refine. If the script attempts to "synthesize" beyond template substitution, it's overreaching.
- **Interactive-mode loss not flagged**: the workflow disappears with the skill deletion. This is a real consequence; review attention is required. If it's not surfaced in Notes for PR, reviewers might miss it.
- **Multi-correction granularity broken**: the existing pattern is one folder per correction line. The script's `--correction-index=N` must preserve this granularity — both for trout-save-session iteration and for the judge panel's per-correction evaluation during compaction.
- **Folder collision silently overwrites**: timestamp resolution is per-second; rapid invocations could collide. Script must `fail` on collision rather than overwrite (loses prior capture data).
- **Call sites left dangling**: any `Skill(skill: "griot-capture", ...)` invocation left after the skill is deleted breaks the substrate.
- **Skill deletion sequenced before call-site updates**: would break in-flight invocations. Order: write script + tests, update callers, then delete skill.
- **Permission entry forgotten**: `Bash(node .claude/scripts/griot/*)` must be added to `.claude/settings.json` since this is the first griot-family script. Without it, invocations from other skills will hit permission prompts.
- **Insufficient test coverage**: tests must exercise the multi-correction case (the new mode-switching surface), the fallback path for empty Execution sections, and the `correction.md` format that downstream tooling depends on.

## Notes for PR

- Mirror migration of deliverables 2 and 3's autosave/autoload shape; this is the third script in the substrate, exercising the convention across two families (trout + griot) for the first time. The pattern is holding — same `parseArgs` shape, same `fail()` helper, same custom error class, same `<verb>-error: <reason>` stderr format, same hardcoded-fallback-on-CONVENTIONS-read-failure framing where applicable.
- correction: Interactive `/griot-capture` (the in-session "user just got corrected, capture it" workflow) is gone with this migration. Per PLAN deliverable 4 wording ("Delete `.claude/skills/griot-capture/`"), but this is a real workflow loss worth flagging. The from-checkin path via `correction:` lines in checkin Notes for PR remains the primary capture mechanism — every `/trout-save-session` at session end iterates and captures. If the in-conversation capture is ever needed back, it can be re-added as a separate skill (or as a thin LLM-driven wrapper that calls the script for the deterministic file derivations).
- correction: `learning.md` is now a thin draft rather than a substantive LLM-synthesized 2-paragraph lesson. The script can't synthesize narrative drafts; the compaction pipeline (`/griot-compact`) is where LLM judgment lives now. Per the original skill spec, drafts are explicitly rewritable by judges — so the thin-template approach is a tactical retreat that aligns with the design intent. If `/griot-compact`'s rewriter consistently has more work to do because drafts are thinner, that's a signal to either bring the LLM-drafting step back into the substrate (as a separate skill) or to enrich the template with more deterministic context.
- Multi-correction handling via `--correction-index=N` matches the existing iteration pattern in `/trout-save-session` (one folder per correction line, distinct slugs). The default `index=0` keeps single-correction invocations zero-config, which is the common case. Out-of-range indices fail with the valid range printed for caller debugging.
- The `wrong.md` fallback (Execution empty → reconstruct from Changes/Verdict with a one-line note about the reconstruction) is the rare edge case but real — early or fast-failed checkins might not have Execution. Handles gracefully without crashing.
- Trade-off accepted: the script's correction extraction is regex-based and assumes `correction:` lines start with optional `-` or `*` bullet markers. If a project ever uses `+` bullets or no marker, the regex needs updating. Real-world checkins so far use `- correction:` consistently.
- Reviewers should focus on: (1) is the interactive-mode loss acceptable, or should we keep a thin `/griot-capture` skill for in-conversation captures (Bash-script-backed for file writes, LLM-driven for the gate + draft); (2) is the `learning.md` thin-template approach acceptable, or should we restore LLM-driven drafting (which would make this a "shell extraction" pattern instead of a full migration); (3) does the multi-correction `--correction-index` design feel right, or would `--correction-text=<exact match>` be safer for `/trout-save-session`'s iteration.
