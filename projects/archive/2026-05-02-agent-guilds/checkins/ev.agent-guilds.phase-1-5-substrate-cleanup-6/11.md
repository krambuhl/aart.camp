**Acceptance criteria**:

1. **Script: `.claude/scripts/trout/pr-respond-plumbing.ts`.** ESM, stdlib-only. Two verbs invoked via `node pr-respond-plumbing.ts <verb> <args...>`:
   - `fetch <slug> <pr-number>`: validates the PR belongs to the project (PR's branch must have a corresponding `projects/<slug>/checkins/<branch>/` directory), shells out to `gh` for PR metadata + comments + reviews + review-comments + check-runs, emits one JSON document on stdout. JSON shape:
     ```json
     {
       "pr": { "number", "url", "state", "branch", "title" },
       "items": [
         { "kind": "issue-comment" | "review" | "review-comment" | "ci-failure",
           "source": "<author handle>",
           "body": "<comment text>",
           "location": "<file:line | review-state | check-name>",
           "url": "<permalink or null>" }
       ],
       "next_response_number": "NN",
       "response_path": "projects/<slug>/checkins/<branch>/response-<NN>.md"
     }
     ```
     - `kind: "review"` items use `location: <state>` where state ∈ `APPROVED | CHANGES_REQUESTED | COMMENTED | DISMISSED`; `body` is the review's top-level body (may be empty).
     - `kind: "ci-failure"` items: one entry per failing check-run; `location` is the check name; `body` is the conclusion + a link description; `url` is the check's `details_url` if available.
     - Items are emitted in the order: issue-comments (chronological), reviews (chronological), review-comments (chronological), ci-failures (alphabetical by check name).
   - `write-plan <slug> <pr-number> --content-file=<path>`: reads the LLM-authored response-plan content, writes to `projects/<slug>/checkins/<branch>/response-<NN>.md` where `<NN>` matches `next_response_number` from the most recent `fetch` (re-computed independently here — single source of truth), exits 0 with stdout `response-plan-written: <relative-path>`.
   - **Errors** (both verbs): stderr `pr-respond-plumbing-error: <reason>` and exit 1. Specific failure modes:
     - PR not found in repo: `pr <N> not found in <owner>/<repo>` (gh exits non-zero; surface stderr verbatim).
     - PR's branch has no checkin directory: `pr <N> belongs to branch <branch>; no checkins/<branch>/ directory found in project <slug>` (the cross-project check the current SKILL describes).
     - `--content-file` missing/non-existent (write-plan only).
     - Project resolution failure (mirrors `archive-relocate.ts::resolveProject` error vocabulary).
     - `gh` not on PATH: `gh CLI not found; install GitHub CLI`.

2. **Next-response-number computation.** Iterate `projects/<slug>/checkins/<branch>/`, collect every numeric `<NN>.md` and `response-<NN>.md` filename, take the max, return `(max + 1).padStart(2, '0')`. If the directory contains no numeric files, start at `01`. Both `fetch`'s emitted `next_response_number` and `write-plan`'s computed write target use the same algorithm — the contract is "next NN matches the latest checkin number + 1, or starts at 01 when empty." Note that `fetch` and `write-plan` independently compute this; if a checkin lands between them, `write-plan` will pick a higher number than `fetch` reported. This is a known race window the SKILL.md acknowledges; no mitigation in scope for D9.

3. **gh CLI invocation surface.** All gh calls use `spawnSync('gh', ['<sub>', '<args...>'], { encoding: 'utf-8' })`. The script does NOT shell out to `bash` or interpolate user input into a shell string. Specific calls:
   - PR metadata: `gh pr view <N> --json number,url,state,headRefName,title,statusCheckRollup`.
   - Issue comments: `gh api repos/<owner>/<repo>/issues/<N>/comments`.
   - Reviews: `gh api repos/<owner>/<repo>/pulls/<N>/reviews`.
   - Review comments: `gh api repos/<owner>/<repo>/pulls/<N>/comments`.
   - `<owner>/<repo>` derived from `gh repo view --json owner,name`. Cached for the duration of one fetch invocation.
   - All gh JSON output is parsed via `JSON.parse`; on parse failure, fail with `gh returned unparseable JSON: <stderr or first 200 chars of stdout>`.

4. **Tests: `.claude/scripts/trout/pr-respond-plumbing.test.ts`.** Vitest with per-file `// @vitest-environment node` directive. **gh is NOT available in CI** by default in fixture form, so the test strategy uses a **gh stub**: the fixture creates a `gh` shell script that responds to known argv patterns with canned JSON, prepends the fixture's `bin/` to `PATH` for the `execFileSync` invocation, and asserts the script consumes the stubbed responses correctly. Cases (10–12 tests):
   - `fetch` happy path with 2 issue-comments + 1 review + 1 review-comment + 1 failing check → JSON output matches the documented shape; items emitted in the documented order; `next_response_number` = `02` when one prior `01.md` exists.
   - `fetch` empty PR (no comments/reviews/check failures) → `items: []`, valid JSON.
   - `fetch` with PR's branch having no checkin directory → exit 1, stderr matches "no checkins/... directory found".
   - `fetch` PR not found in repo (gh stub returns non-zero) → exit 1, stderr surfaces gh's stderr.
   - `fetch` with no prior checkins on branch → `next_response_number` = `01`.
   - `fetch` with multiple `<NN>.md` files (max=05) and one `response-03.md` → `next_response_number` = `06` (max across both filename patterns).
   - `write-plan` happy path → file written to expected path; stdout matches `response-plan-written:`; content matches `--content-file` byte-for-byte.
   - `write-plan` missing `--content-file` → exit 1.
   - `write-plan` `--content-file` points to non-existent path → exit 1.
   - `write-plan` writes `response-NN.md` not `NN.md` (the response-plan file is intentionally distinct from the eventual checkin at the same NN; SKILL.md's "intentionally share the number" semantics).
   - gh not on PATH (clear PATH for the run) → exit 1, stderr contains `gh CLI not found`.
   - All ci-check items use `kind: "ci-failure"` only when the check-run's conclusion is `failure` (or `cancelled`/`timed_out`); `success` and `neutral` are filtered out.

**Disqualifiers**:
- **Skill body Step 2 (classification taxonomy) or Step 3a (response-plan authoring template) substantively rewritten or shortened.** Those are LLM-shaped and out of scope.
- **Script absorbs classification logic.** A `--classify` flag, hard-coded taxonomy in the script, or the script deciding which items are blockers vs nits is over-scoped. Classification is the LLM's call.
- **`write-plan` writes a checkin file (`<NN>.md`) instead of `response-<NN>.md`.** The two filename patterns are intentionally distinct — `response-NN.md` is the plan; the matching `<NN>.md` is the eventual checkin authored after the plan executes.
- **`fetch` shells through `bash -c` or interpolates user input into a shell command.** All gh invocations go through `spawnSync('gh', [args])` with array args.
- **Tests use a real gh CLI or hit the network.** Stub via fixture-local script + PATH override; otherwise tests aren't reproducible.
- **Tests skip the cross-project guard** (PR branch with no checkins/&lt;branch&gt;/ directory). That guard is what keeps the script from leaking PR data across projects.
- **`fetch` emits items in arbitrary order.** The documented order (issue-comments → reviews → review-comments → ci-failures, each chronological/alphabetical) is part of the LLM's input contract; non-determinism breaks classification reproducibility.
- **`write-plan` and `fetch` use different next-NN algorithms.** Both must scan `<NN>.md` AND `response-<NN>.md` patterns; both must take `max + 1` or default to `01`.
- **MCP github tools or `Bash(ls:*)`/`Bash(git branch:*)` retained in allowed-tools.** The whole point is the skill no longer talks to GitHub directly; leaving them in is dead surface area.
- **Script body grows past ~280 lines** or test file past ~400 lines.

## Notes for the PR

- **Test PATH gotcha worth carrying forward.** The fixture-local gh stub strategy (shell script + PATH override) is clean, but PATH override drops the node binary's actual location too. Symptom: 12 silent test failures with empty stderr. Fix: invoke node via `process.execPath` (absolute path) so name-lookup is bypassed. This is the second time a substrate-script test has hit a PATH-related gotcha during fixture authoring; the first was D8's `cpSync` import-from-wrong-module mistake. Both surface the same lesson: fixture builders that override the test environment need careful audit of what the shell needs from PATH to actually run.
- **SKILL.md compression took two passes.** Initial draft 149, after deleting prose JSON-shape explanations 112, after condensing Quality bar bullets 110. Carrying forward: when an LLM/CRUD split simplifies a SKILL's mechanical content, the prose tends to grow elsewhere ("here's how the script works") to fill the deletion. Future contracts should explicitly cap that compensatory prose, not just the overall word count.
- **`Item.location` overloads three concepts.** For `kind: 'issue-comment'`, location is the literal string `'pr-conversation'`. For `kind: 'review'`, it's the review state (`APPROVED | CHANGES_REQUESTED | COMMENTED | DISMISSED`). For `kind: 'review-comment'`, it's `<file>:<line>`. For `kind: 'ci-failure'`, it's the check name. The LLM consumer can disambiguate via `kind`, but the overload is a near-miss on "homogeneous shape per AC" — works in practice, deserves a one-line comment near `buildItems`. Tracked, not addressed in D9.
- **gh-stub fixture strategy is reusable.** The `ghStub(responses, opts)` helper at test:41-66 takes a map of argv-prefix → JSON-string and emits a bash dispatcher. Future substrate scripts that shell out to gh can copy this pattern verbatim. Worth promoting to a shared test helper if a third script needs it (D11/D12 might).
- **Cross-project guard works as designed.** The test at `test:158-178` exercises the case where `gh pr view` returns a `headRefName` for a branch that has no `checkins/<branch>/` directory in the resolved project — script refuses with the documented error. This is what keeps `/trout-pr-respond` from leaking PR data across projects when invoked with the wrong slug.
