  1. **CONVENTIONS.md gains a new top-level section**:
     `## Generator-evaluator pair authoring`. Placement: after the
     existing `## PLAN.md phase-config extensions` section, before
     `## Event vocabulary`. Body covers (concise):
     - The substrate has three agent families: `evaluator-*`
       (read-only verdicts), `whiteboard-*` (read-only prose),
       and `generator-*` (write-capable artifacts).
     - Per the Phase 4 whiteboard, a "domain pair" is a *pattern*
       — a `generator-*` paired with a specialist `evaluator-*`
       that participates in the panel with elevated precedence —
       not a separate agent family.
     - The shape for adding a new pair:
       - Author `.claude/agents/generator-<domain>.md`
         (frontmatter: `name`, `role: generator`, `description`,
         `tools` per `generator-base.md`'s allowlist or a narrower
         subset, `model: inherit`, `activation: active | stub`).
       - Author `.claude/agents/evaluator-<domain>.md` (matches
         existing `evaluator-*` shape per `evaluator-base.md`).
       - Register both in `PANEL-COMPOSITION.md` — the specialist
         evaluator gets a slot in the precedence list (placement
         depends on the domain's blast radius); document the
         boundary with adjacent evaluators in the appropriate
         section.
       - Invoke via phase-config `Panel: +<evaluator-name>` on a
         unit's contract; the panel runs the specialist with
         elevated precedence per the gate-then-review pattern
         (also documented in `/ev-loop-*` SKILLs).
     - **Stubs vs active**: per the Phase 4 whiteboard's
       skeptic-flagged risk, stubs ship only when concrete
       activation criteria can be named AND the toolchain
       actually exists in the project. Speculative stubs invite
       drift; they're worst-of-both-worlds (enough disk presence
       to bit-rot, not enough callability to provide value). The
       active pair (`generator-css-codemod` /
       `evaluator-css-architecture`) is the only v1 pair.
     - **Activation criteria documentation shape** (forward-
       compat): when a stub IS authored, its frontmatter
       includes an `activates-when:` list (string array) per
       `whiteboard-react-architect`'s round-1 recommendation;
       the loop reads frontmatter (not body prose) for the
       loud-error path.
     - Cross-reference: see `generator-base.md` for the
       family's shared contract; see `evaluator-css-architecture`
       and `generator-css-codemod` for the v1 active-pair
       reference implementation.
     - Section length: ≤ 70 lines. This is a pattern doc, not
       an exhaustive spec.
