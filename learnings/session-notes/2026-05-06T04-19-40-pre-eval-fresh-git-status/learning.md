When a unit's acceptance criterion specifies tree state at exact granularity — "`git status` shows only X and Y" or any binary state-shape claim — re-run the underlying command immediately before invoking the evaluator and record its output verbatim into the checkin. Do not record the verification claim from a recalled summary earlier in the session, even if the unit's own work hasn't touched anything outside the named files. Framework auto-generation (Next.js writing to `next-env.d.ts`, build-output references, codegen on dev/build hooks) can silently mutate tracked files between the time you "knew the state" and the time the evaluator reads it.

Concrete pattern: every binary state-shape acceptance criterion gets a paired pre-evaluation step. `git status` → record output → check against criterion → invoke evaluator. Failing the check shifts the unit into a fix-or-amend-contract decision before the evaluator round, not after. Recalled summaries are evidence about *intent*, not state.
