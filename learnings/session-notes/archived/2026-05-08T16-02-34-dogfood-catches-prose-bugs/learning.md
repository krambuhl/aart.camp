Before writing prose that describes what a script does at runtime — what it resolves, substitutes, parses, or passes along — you must ground the description in execution evidence, not in code-reading inference or the shape of the surrounding contract. Run the script against a real input, capture its actual output, and cite that output (a command invocation and the relevant lines of its result) as the basis for the claim. Reading the script's source, its types, or its CLI surface and then narrating what it "should" do is the failure mode; the runtime behavior frequently diverges from what the code structure suggests, and prose written from inference will confidently assert substitutions and resolutions that never happen. If you cannot run the script in the current session, do not write a runtime claim as fact — either ask for a dogfood run, or explicitly mark the description as unverified inference from source so a reviewer knows to confirm it.

Concretely: when a CLI like `pr-plumbing.ts submit --phase-update='<N>:in-progress:pr=#<N> (open)'` is involved, do not assume the script resolves `<N>` from the `gh` response and substitutes it before invoking autosave — it does not, and the literal placeholder text ends up in the manifest. The actual pattern, confirmed by running it, is two steps: invoke `submit` without `--phase-update`, parse the new PR number from `submit`'s terminal-state output line, then run a follow-up `autosave --phase-update` with the resolved number to set the phase row's PR field. Treat any prose claim about cross-step value resolution, placeholder substitution, or implicit data flow between subcommands as requiring an execution receipt before it ships.
